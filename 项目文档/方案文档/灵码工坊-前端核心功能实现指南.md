## 一、前端架构全景——五面板三模式的 SSE 驱动体系

### 1.1 灵码工坊前端不是什么

在深入技术细节之前，先澄清一个容易产生的误解：灵码工坊的前端**不是一个普通的 Vue 3 SPA**。普通的 SPA 是"用户发请求 → 后端返回数据 → 前端渲染"的请求-响应模式。灵码工坊的前端完全不同——它是一个**事件驱动的实时协作系统**。

用户输入需求后，后端的六阶段流水线持续运行数十秒甚至数分钟。在这段时间里，后端不是在"等待前端请求"——后端在主动推送事件，前端的职责是**监听这些事件并实时更新五个面板**。这不是请求-响应模式，这是**发布-订阅模式**。SSE（Server-Sent Events）就是这个发布-订阅模式的数据通道。

理解这一点至关重要——如果你按传统的"先设计 API 接口、再设计前端页面"的思路来设计灵码工坊的前端，你会设计出一堆 GET/POST 接口，然后前端轮询这些接口来获取进度更新。这会彻底破坏用户对"实时生成"的感知——当用户看到进度清单每隔 2 秒跳一次、文件树每隔 2 秒多一个节点，他会觉得 AI 很慢、系统很卡。**实时性不是性能优化，是产品体验的基础。**

### 1.2 五面板 + 三模式的架构全景

灵码工坊工作台的界面由五个核心面板和一个顶部工具栏组成：

```
┌─────────────────────────────────────────────────────────────────────────┐

├────────┬───────────────────┬───────────────────┬─────────────────────────┤
│ 列1    │ 列2               │ 列3               │ 列4                     │
│ 项目列表│ ChatPanel         │ FileTreePanel     │ PreviewPanel            │
│        │ （对话区）          │ + EditorPanel     │ （预览区）                │
│        │                   │ （文件树+编辑器）    │                         │
│        │ ┌───────────────┐ │ ┌───────────────┐ │ ┌─────────────────────┐ │
│        │ │ 对话气泡       │ │ │ 文件Tab筛选    │ │ │ 预览/代码切换        │ │
│        │ │ 进度清单       │ │ │ 文件树        │ │ │ 设备切换            │ │
│        │ │ 文件变更列表   │ │ │ Monaco编辑器   │ │ │ iframe 应用画布     │ │

└─────────────────────────────────────────────────────────────────────────┘
```

这五个面板不是独立的孤岛，它们共享三条数据通道：

| 数据通道 | 技术 | 方向 | 携带什么数据 |
|---------|------|------|------------|
| **SSE 流** | EventSource `/api/stream/generation/{taskId}` | 后端 → 前端 | 六阶段进度、AI 回复文本、文件生成事件、构建日志、预览 URL |
| **WebSocket** | WebSocket `/ws/sandbox/{projectId}/logs` | 后端 → 前端 | 沙箱容器实时日志（npm install 输出、构建过程） |
| **HTTP REST** | Axios `/api/**` | 前端 → 后端 | 创建任务、停止生成、获取文件内容、启动/停止沙箱 |

**SSE 流是主通道**——它驱动了 ChatPanel 的 AI 回复、GenerationChecklist 的进度更新、FileTreePanel 的文件节点新增、EditorPanel 的代码注入、PreviewPanel 的 URL 加载。WebSocket 是辅通道——它只负责沙箱日志的实时推送。HTTP REST 是控制通道——前端用它来发起操作（创建任务、停止生成、切换文件等）。

### 1.3 三模式切换机制

工作台有三种视图模式，由 `useWorkbenchStore` 的 `mode` 状态驱动：

```
简洁模式 (simple)
  → 仅对话区居中展开，其余面板折叠
  → 适合"从零开始"的新对话
  → M1 阶段首次实现

生成模式 (generation)
  → 四列面板同时展开
  → AI 正在生成，进度可见
  → 文件树与预览实时变化
  → M2 阶段首次实现

完整模式 (complete)
  → 四列全展开 + 底部诊断区
  → 用户可手动编辑代码
  → 可继续对话迭代
  → M3 阶段首次实现
```

模式切换的触发条件：

```
简洁模式 ──用户提交需求──→ 生成模式
生成模式 ──生成全部完成──→ 完整模式
完整模式 ──用户继续对话──→ 生成模式
简洁模式 ──恢复已有项目──→ 完整模式
完整模式 ──新建对话────→ 简洁模式
```

**注意：** 在 M1-M4 开发阶段（核心组件优先策略），左侧列1（项目列表）和模式切换逻辑先做简化版——不做真实登录、不做项目列表持久化。但 `useWorkbenchStore` 的 `mode` 状态和模式切换的骨架要在一开始就搭好，后续只需要替换数据源，不需要改组件结构。

### 1.4 前端新增依赖

相比当前原型阶段（纯静态 HTML+CSS），接入真实后端需要新增以下依赖：

| 包名 | 做什么 | 必要性 | 说明 |
|------|--------|--------|------|
| `axios` | HTTP 客户端 | **必须** | POST 创建任务、GET 项目列表等非 SSE 请求；拦截器中注入 Token、处理 401 刷新 |
| `monaco-editor` | 代码编辑器 | **必须** | VS Code 同款引擎，支持语法高亮、Diff 模式、多 Tab |
| `markdown-it` | Markdown 渲染 | **必须** | AI 回复内容含 Markdown 格式（代码块、列表、链接） |
| `pinia-plugin-persistedstate` | Pinia 持久化 | **推荐** | 自动将 token/用户信息同步到 localStorage，页面刷新不丢失 |
| `splitpanes` | 面板拖拽调整 | **推荐** | 四列布局需要可拖拽的分割线，用户可自由调整面板宽度 |

安装命令：

```bash
npm install axios monaco-editor markdown-it pinia-plugin-persistedstate splitpanes
npm install -D @types/markdown-it
```

**为什么不用 `EventSource` 的 polyfill？** 因为 `EventSource` 是浏览器原生 API，所有现代浏览器（Chrome 6+、Firefox 6+、Safari 5+、Edge 79+）都原生支持。不需要额外安装任何 SSE 客户端库——浏览器的 `EventSource` 已经提供了连接、自动重连、事件监听等所有我们需要的功能。

## 二、SSE 客户端——前端的数据大动脉

### 2.1 SSE 不是"一个请求"

很多人把 SSE 理解为"一个长连接的 GET 请求"。技术上这是对的，但理解上不够。SSE 是前端和后端之间的一条**持久化事件流**——它不是"前端请求一次、后端响应一次"，而是"前端建立连接、后端持续推送事件、前端持续处理事件"。

这条事件流是前端所有面板的数据来源。没有它，ChatPanel 不知道 AI 回复了什么、GenerationChecklist 不知道进行到哪个阶段了、FileTreePanel 不知道文件生成完了没有、PreviewPanel 不知道沙箱 URL 是什么。**SSE 是前端的脊椎，五个面板是脊椎上长出的器官。**

### 2.2 后端推送的 SSE 事件规范

后端（`GenerationService`）通过 `Sinks.Many<ServerSentEvent<GraphNodeResponse>>` 推送事件。前端通过 `EventSource` 接收。事件的格式是：

```
event: message
data: {"threadId":"task-001","nodeName":"requirement_analysis","text":"...","textType":"JSON","error":false}

event: message
data: {"threadId":"task-001","nodeName":"code_generation","text":"...","textType":"MARK_DOWN","error":false}

event: complete
data: {"threadId":"task-001","url":"https://sandbox-xxx.lingmaforge.dev","port":5173,"buildTime":1.23}

event: error
data: {"threadId":"task-001","nodeName":"build_verification","text":"构建失败: TS2307...","error":true}
```

**`event` 字段决定前端的行为类型：**
- `message` → 更新对应面板的内容（AI 回复文本、文件树节点、日志条目）
- `complete` → 全部生成完成，关闭 SSE 连接，加载预览
- `error` → 展示错误信息，提供重试入口

**`data.nodeName` 字段决定前端**哪个面板**响应这个事件：**

| nodeName | 路由到哪个面板 | 面板做什么 |
|----------|-------------|-----------|
| `requirement_analysis` | ChatPanel + Checklist | 展示分析结果 JSON + 清单项"需求分析"亮起 |
| `execution_planning` | ChatPanel + FileTreePanel + Checklist | 初始化文件树骨架 + 清单项"执行规划"亮起 |
| `code_generation` | ChatPanel + FileTreePanel + EditorPanel + Checklist | 文件树新增节点 + 编辑器注入代码 + 清单项"代码生成"亮起 |
| `style_optimization` | ChatPanel + EditorPanel + Checklist | CSS diff 展示 + 清单项"样式优化"亮起 |
| `build_verification` | LogPanel + Checklist | 构建日志展示 + 清单项"构建验证"亮起 |
| `preview_deploy` | PreviewPanel + Checklist | 加载沙箱 URL + 清单项"预览部署"亮起 |

**`data.textType` 字段决定前端怎么渲染 `text` 内容：**

| textType | 渲染方式 | 用在哪个阶段 |
|----------|---------|------------|
| `TEXT` | 普通文本，直接展示 | AI 的自然语言回复 |
| `JSON` | 语法高亮的 JSON 展示 | 需求分析结果、执行规划结果 |
| `MARK_DOWN` | `markdown-it` 渲染为 HTML | AI 回复中的 Markdown 内容 |
| `SQL` | 暂不使用（预留） | — |
| `HTML` | 暂不使用（预留） | — |

### 2.3 SSE 客户端封装——不是简单的 EventSource 包装

`EventSource` 是一个简单的 API——`new EventSource(url)` 就建立连接了。但生产级的 SSE 客户端需要处理很多边界情况：

1. **连接中断后自动重连**：`EventSource` 原生支持自动重连，但重连间隔是固定的（浏览器默认 3 秒）。对于灵码工坊来说，如果重连成功，后端需要从断点恢复（LangGraph4j Checkpoint），前端需要从最后一次收到的 `nodeName` 继续更新 UI。

2. **用户主动停止生成**：用户点击"停止生成"按钮，前端应该关闭 SSE 连接并调用 `DELETE /api/generation/{taskId}/stop`。不能只关闭前端连接——后端还在执行，资源没有释放。

3. **网络切换导致的重连**：用户从 WiFi 切换到移动热点，SSE 连接会断开。`EventSource` 会自动重连，但重连的 URL 可能不同（如果网络切换导致 IP 变化）。前端应该监听 `error` 事件，在连续重连失败时提示用户。

4. **多个 Tab 共享连接**：用户可能在同一个浏览器中打开多个工作台 Tab。如果它们都建立 SSE 连接到同一个 taskId，后端的 SSE Sink 是 unicast（一对一），第二个 Tab 的连接会失败。解决方案是在连接失败时提示"该任务已在另一个窗口运行"。

```typescript
// src/utils/sseClient.ts

export interface SSEMessage {
  threadId: string
  nodeName: string
  text: string
  textType: 'TEXT' | 'JSON' | 'MARK_DOWN' | 'SQL' | 'HTML'
  error: boolean
}

export interface SSECompleteData {
  threadId: string
  url: string
  port: number
  buildTime: number
}

export interface SSEHandlers {
  onMessage: (msg: SSEMessage) => void
  onComplete: (data: SSECompleteData) => void
  onError: (error: string) => void
  onReconnecting?: (attempt: number) => void
}

export class GenerationSSEClient {
  private eventSource: EventSource | null = null
  private taskId: string
  private handlers: SSEHandlers
  private reconnectAttempt = 0
  private maxReconnectAttempts = 5
  private reconnectDelay = 3000  // 3秒
  private manuallyClosed = false
  private lastReceivedNodeName: string | null = null

  constructor(taskId: string, handlers: SSEHandlers) {
    this.taskId = taskId
    this.handlers = handlers
  }

  /** 建立 SSE 连接 */
  connect(): void {
    this.manuallyClosed = false
    const url = `/api/stream/generation/${this.taskId}`

    this.eventSource = new EventSource(url)

    // 监听 message 事件 —— 生成进度推送
    this.eventSource.addEventListener('message', (event: MessageEvent) => {
      const data: SSEMessage = JSON.parse(event.data)
      this.lastReceivedNodeName = data.nodeName

      if (data.error) {
        this.handlers.onError(data.text)
      } else {
        this.handlers.onMessage(data)
        this.reconnectAttempt = 0  // 成功收到消息，重置重连计数
      }
    })

    // 监听 complete 事件 —— 生成全部完成
    this.eventSource.addEventListener('complete', (event: MessageEvent) => {
      const data: SSECompleteData = JSON.parse(event.data)
      this.handlers.onComplete(data)
      this.close()
    })

    // 监听 error 事件 —— 连接异常
    this.eventSource.addEventListener('error', () => {
      if (this.manuallyClosed) return

      this.reconnectAttempt++
      if (this.reconnectAttempt <= this.maxReconnectAttempts) {
        this.handlers.onReconnecting?.(this.reconnectAttempt)
        // EventSource 会自动重连，这里只通知 UI 展示"重连中"
      } else {
        this.handlers.onError('SSE 连接已中断，请刷新页面重试')
        this.close()
      }
    })

    // 监听 open 事件 —— 连接建立成功
    this.eventSource.addEventListener('open', () => {
      this.reconnectAttempt = 0
    })
  }

  /** 主动关闭连接（用户点击停止或生成完成） */
  close(): void {
    this.manuallyClosed = true
    this.eventSource?.close()
    this.eventSource = null
  }

  /** 获取最后一次收到的节点名称（用于断点恢复） */
  getLastNodeName(): string | null {
    return this.lastReceivedNodeName
  }
}
```

### 2.4 SSE 事件的路由——从 nodeName 到 UI 更新

当 `onMessage` 被调用时，前端需要根据 `nodeName` 决定更新哪个面板。这个路由逻辑集中在一个地方——避免散落在各个组件中：

```typescript
// src/composables/useGenerationStream.ts

import { useChatStore } from '@/stores/chat'
import { useProjectStore } from '@/stores/project'
import { useSandboxStore } from '@/stores/sandbox'

export function useGenerationStream() {
  const chatStore = useChatStore()
  const projectStore = useProjectStore()
  const sandboxStore = useSandboxStore()

  function handleMessage(msg: SSEMessage): void {
    // 1. 更新进度清单（所有阶段都会触发）
    chatStore.updateChecklist(msg.nodeName, msg.error ? 'error' : 'done')

    // 2. 根据 nodeName 路由到具体面板
    switch (msg.nodeName) {
      case 'requirement_analysis':
        // ChatPanel 展示分析结果
        chatStore.appendAssistantMessage(msg.text, msg.textType)
        break

      case 'execution_planning':
        // ChatPanel 展示规划结果
        chatStore.appendAssistantMessage(msg.text, msg.textType)
        // FileTreePanel 初始化文件树骨架
        if (msg.textType === 'JSON') {
          const plan = JSON.parse(msg.text)
          projectStore.initFileTree(plan.generationOrder || [])
        }
        break

      case 'code_generation':
        // ChatPanel 展示 "正在生成 xxx.tsx..."
        chatStore.appendAssistantMessage(msg.text, msg.textType)
        // FileTreePanel 新增/更新文件节点
        // EditorPanel 注入代码内容
        // 具体的文件路径和内容从 text 中解析
        break

      case 'style_optimization':
        chatStore.appendAssistantMessage(msg.text, msg.textType)
        break

      case 'build_verification':
        // LogPanel 展示构建日志（通过 SandboxStore）
        sandboxStore.appendLog('build', msg.text)
        break

      case 'preview_deploy':
        // PreviewPanel 加载沙箱 URL
        break
    }
  }

  function handleComplete(data: SSECompleteData): void {
    chatStore.completeChecklist()
    sandboxStore.setPreviewUrl(data.url)
    sandboxStore.setStatus('running')
    sandboxStore.appendLog('system', `构建耗时: ${data.buildTime}s`)
  }

  function handleError(error: string): void {
    chatStore.appendSystemMessage(error, 'error')
    chatStore.setGenerating(false)
  }

  return { handleMessage, handleComplete, handleError }
}
```

### 2.5 SSE 与前端生命周期的配合

SSE 连接的生命周期应该和生成任务的生命周期一致，而不是和页面组件的生命周期一致：

- **不需要把 SSE 连接放在 `onMounted` / `onUnmounted` 中**——因为用户可能在生成过程中切换 Tab（从代码编辑器切换到预览区），组件可能会卸载再挂载。如果 SSE 连接绑在组件上，切换 Tab 就会断连。
- **SSE 连接应该绑定到 Pinia Store**——Store 的生命周期和页面一致。用户在同一个工作台页面上不管怎么切换面板，Store 都不会销毁，SSE 连接也不会断开。
- **页面离开时（`onBeforeUnmount`）主动关闭 SSE**——用户在生成过程中离开工作台页面（比如导航到首页），应该关闭 SSE 连接并调用停止 API，释放后端资源。

```typescript
// 在 WorkbenchView.vue 中
import { onBeforeUnmount } from 'vue'
import { useChatStore } from '@/stores/chat'

const chatStore = useChatStore()

onBeforeUnmount(() => {
  if (chatStore.isGenerating) {
    chatStore.stopGeneration()
  }
})
```

## 三、Pinia Stores——前端的脊椎神经

### 3.1 为什么需要三个核心 Store

灵码工坊的工作台有五个面板，每个面板有各自的状态。但有些状态是跨面板共享的——文件树的状态影响编辑器的内容、编辑器的内容影响预览区的渲染、对话区的状态影响工具栏的显示。如果把所有状态都散落在各个组件中，面板之间的通信会变成一团乱麻——A 组件通过 emit 告诉 B 组件、B 组件通过 props 传给 C 组件、C 组件又通过 emit 告诉 A 组件。

**Pinia Store 是跨组件共享状态的唯一来源。** 它扮演的角色类似于后端 StateGraph 中的 `AgentState`——所有节点读写同一个 State，每个节点看到的是同一个数据视图。

与后端 `AgentState` 的对应关系：

| 后端 AgentState 字段 | 前端 Pinia Store | 说明 |
|---------------------|-----------------|------|
| `analysisResult` | `chatStore.checklistState.requirement_analysis` | 需求分析完成状态 |
| `planResult.generationOrder` | `projectStore.fileTree` | 文件树骨架 |
| `generatedFiles` | `projectStore.fileTree`（带 content） | 文件节点内容 |
| `previewUrl` | `sandboxStore.url` | 预览 URL |
| `buildStatus` | `sandboxStore.status` | 沙箱运行状态 |
| `buildLog` | `sandboxStore.logs` | 构建日志 |

### 3.2 useChatStore——对话状态管理

这是三个 Store 中最核心的一个。它管理对话历史、SSE 连接、生成状态、六阶段进度：

```typescript
// src/stores/chat.ts
import { defineStore } from 'pinia'
import { ref, computed } from 'vue'
import type { SSEMessage } from '@/types'
import { GenerationSSEClient } from '@/utils/sseClient'
import { generationApi } from '@/api/generation'

export type ChecklistStage = 'idle' | 'running' | 'done' | 'error'

export interface ChecklistState {
  requirement_analysis: ChecklistStage
  execution_planning: ChecklistStage
  code_generation: ChecklistStage
  style_optimization: ChecklistStage
  build_verification: ChecklistStage
  preview_deploy: ChecklistStage
}

export interface ChatMessage {
  id: string
  role: 'user' | 'assistant' | 'system'
  content: string
  contentType: 'TEXT' | 'JSON' | 'MARK_DOWN' | 'HTML'
  timestamp: number
  nodeName?: string        // 消息来自哪个阶段（可选）
  fileChanges?: FileChange[]  // 文件变更列表（可选，code_generation 阶段附带）
}

export const useChatStore = defineStore('chat', () => {
  // ===== State =====
  const messages = ref<ChatMessage[]>([])
  const isGenerating = ref(false)
  const checklistState = ref<ChecklistState>({
    requirement_analysis: 'idle',
    execution_planning: 'idle',
    code_generation: 'idle',
    style_optimization: 'idle',
    build_verification: 'idle',
    preview_deploy: 'idle'
  })

  // SSE 客户端实例（存储在 Store 中，不随组件销毁）
  let sseClient: GenerationSSEClient | null = null

  // ===== Getters =====
  const currentStage = computed<keyof ChecklistState | null>(() => {
    const entries = Object.entries(checklistState.value) as [keyof ChecklistState, ChecklistStage][]
    const running = entries.find(([, stage]) => stage === 'running')
    return running ? running[0] : null
  })

  const isComplete = computed(() =>
    Object.values(checklistState.value).every(s => s === 'done')
  )

  const stages = computed(() => [
    { key: 'requirement_analysis' as const, label: '需求分析', icon: 'search' },
    { key: 'execution_planning' as const, label: '执行规划', icon: 'map' },
    { key: 'code_generation' as const, label: '代码生成', icon: 'code' },
    { key: 'style_optimization' as const, label: '样式优化', icon: 'palette' },
    { key: 'build_verification' as const, label: '构建验证', icon: 'check' },
    { key: 'preview_deploy' as const, label: '预览部署', icon: 'rocket' }
  ])

  // ===== Actions =====

  /** 发送用户输入并启动 SSE 连接 */
  async function sendMessage(prompt: string): Promise<void> {
    // 1. 添加用户消息
    messages.value.push({
      id: crypto.randomUUID(),
      role: 'user',
      content: prompt,
      contentType: 'TEXT',
      timestamp: Date.now()
    })

    // 2. 创建生成任务
    const { taskId } = await generationApi.create({ prompt })

    // 3. 开始生成
    isGenerating.value = true
    resetChecklist()

    // 4. 建立 SSE 连接
    sseClient = new GenerationSSEClient(taskId, {
      onMessage: (msg: SSEMessage) => {
        handleSSEMessage(msg)
      },
      onComplete: (data) => {
        handleSSEComplete(data)
      },
      onError: (error) => {
        handleSSEError(error)
      }
    })
    sseClient.connect()
  }

  /** 处理 SSE message 事件 */
  function handleSSEMessage(msg: SSEMessage): void {
    // 更新进度清单
    const nodeName = msg.nodeName as keyof ChecklistState
    if (checklistState.value[nodeName] !== undefined) {
      checklistState.value[nodeName] = 'done'
    }

    // 找当前阶段的前一个阶段，标记为 done
    const stageOrder = Object.keys(checklistState.value)
    const currentIndex = stageOrder.indexOf(nodeName)
    if (currentIndex > 0) {
      for (let i = 0; i < currentIndex; i++) {
        checklistState.value[stageOrder[i] as keyof ChecklistState] = 'done'
      }
    }
    // 当前阶段标记为 running（下一个 message 来时才变 done）
    checklistState.value[nodeName] = 'running'

    // 追加 assistant 消息
    messages.value.push({
      id: crypto.randomUUID(),
      role: 'assistant',
      content: msg.text,
      contentType: msg.textType,
      timestamp: Date.now(),
      nodeName: msg.nodeName
    })
  }

  /** 处理 SSE complete 事件 */
  function handleSSEComplete(data: any): void {
    // 所有阶段标记为完成
    Object.keys(checklistState.value).forEach(key => {
      checklistState.value[key as keyof ChecklistState] = 'done'
    })
    isGenerating.value = false
    sseClient = null
  }

  /** 处理 SSE error 事件 */
  function handleSSEError(error: string): void {
    // 当前阶段标记为 error
    const current = currentStage.value
    if (current) {
      checklistState.value[current] = 'error'
    }
    messages.value.push({
      id: crypto.randomUUID(),
      role: 'system',
      content: error,
      contentType: 'TEXT',
      timestamp: Date.now()
    })
    isGenerating.value = false
  }

  /** 停止生成 */
  async function stopGeneration(): Promise<void> {
    sseClient?.close()
    sseClient = null
    isGenerating.value = false
    // 调用后端停止 API（如果 taskId 已知）
  }

  /** 重置清单 */
  function resetChecklist(): void {
    Object.keys(checklistState.value).forEach(key => {
      checklistState.value[key as keyof ChecklistState] = 'idle'
    })
  }

  /** 更新清单（外部调用） */
  function updateChecklist(stage: keyof ChecklistState, status: ChecklistStage): void {
    checklistState.value[stage] = status
  }

  /** 全部完成 */
  function completeChecklist(): void {
    Object.keys(checklistState.value).forEach(key => {
      checklistState.value[key as keyof ChecklistState] = 'done'
    })
  }

  /** 追加系统消息 */
  function appendSystemMessage(content: string, level: 'info' | 'error' = 'info'): void {
    messages.value.push({
      id: crypto.randomUUID(),
      role: 'system',
      content,
      contentType: 'TEXT',
      timestamp: Date.now()
    })
  }

  /** 追加 Assistant 消息 */
  function appendAssistantMessage(content: string, contentType: SSEMessage['textType']): void {
    messages.value.push({
      id: crypto.randomUUID(),
      role: 'assistant',
      content,
      contentType,
      timestamp: Date.now()
    })
  }

  return {
    messages,
    isGenerating,
    checklistState,
    currentStage,
    isComplete,
    stages,
    sendMessage,
    stopGeneration,
    resetChecklist,
    updateChecklist,
    completeChecklist,
    appendSystemMessage,
    appendAssistantMessage,
    setGenerating: (val: boolean) => { isGenerating.value = val }
  }
})
```

### 3.3 useProjectStore——项目与文件管理

`useProjectStore` 管理当前项目的文件树、当前打开的文件、编辑器的 Tab 状态：

```typescript
// src/stores/project.ts
import { defineStore } from 'pinia'
import { ref, computed } from 'vue'

export interface FileNode {
  id: string
  name: string
  path: string
  type: 'file' | 'folder'
  language?: string        // TypeScript / CSS / JSON 等
  status?: 'new' | 'modified' | 'unchanged'
  content?: string         // 文件内容（从数据库加载或 SSE 注入）
  children?: FileNode[]    // 文件夹的子节点
}

export interface EditorTab {
  id: string
  path: string
  name: string
  language: string
  content: string
  isDirty: boolean         // 是否有未保存的修改
}

export const useProjectStore = defineStore('project', () => {
  // ===== State =====
  const projectId = ref<string | null>(null)
  const projectName = ref('')
  const fileTree = ref<FileNode[]>([])
  const activeFile = ref<EditorTab | null>(null)
  const openTabs = ref<EditorTab[]>([])
  const expandedFolders = ref<Set<string>>(new Set())

  // ===== Getters =====
  const fileCount = computed(() => countFiles(fileTree.value))
  const pageCount = computed(() =>
    findNodesByPath(fileTree.value, 'src/pages').length
  )
  const apiCount = computed(() =>
    findNodesByPath(fileTree.value, 'src/api').length
  )

  // ===== Actions =====

  /** 根据执行规划的 generationOrder 初始化文件树骨架 */
  function initFileTree(generationOrder: Array<{
    path: string
    type: string
    description: string
    dependencies: string[]
  }>): void {
    const root: FileNode[] = []

    for (const item of generationOrder) {
      const parts = item.path.split('/')
      const fileName = parts.pop()!
      const folderPath = parts.join('/')

      // 找到或创建父文件夹
      let parent = findOrCreateFolder(root, folderPath)

      // 创建文件节点（状态为 'new'，等待 SSE 注入内容）
      const fileNode: FileNode = {
        id: item.path,
        name: fileName,
        path: item.path,
        type: 'file',
        language: inferLanguage(fileName),
        status: 'new',
        content: ''  // 初始为空，SSE code_generation 事件会注入内容
      }

      if (parent) {
        parent.children = parent.children || []
        parent.children.push(fileNode)
      }
    }

    fileTree.value = root
    // 默认展开所有文件夹
    expandAll(root)
  }

  /** SSE 事件触发的文件内容更新 */
  function updateFileContent(path: string, content: string): void {
    const node = findNodeByPath(fileTree.value, path)
    if (node) {
      node.content = content
      node.status = 'modified'

      // 如果该文件正在编辑器中打开，同步更新
      const tab = openTabs.value.find(t => t.path === path)
      if (tab) {
        tab.content = content
        tab.isDirty = false
      }
    }
  }

  /** 打开文件（点击文件树节点） */
  function openFile(path: string): void {
    const node = findNodeByPath(fileTree.value, path)
    if (!node || node.type !== 'file') return

    // 检查是否已经打开
    const existingTab = openTabs.value.find(t => t.path === path)
    if (existingTab) {
      activeFile.value = existingTab
      return
    }

    // 创建新 Tab
    const tab: EditorTab = {
      id: crypto.randomUUID(),
      path: node.path,
      name: node.name,
      language: node.language || 'text',
      content: node.content || '',
      isDirty: false
    }

    openTabs.value.push(tab)
    activeFile.value = tab
  }

  /** 关闭文件 Tab */
  function closeTab(tabId: string): void {
    const index = openTabs.value.findIndex(t => t.id === tabId)
    if (index === -1) return

    openTabs.value.splice(index, 1)

    if (activeFile.value?.id === tabId) {
      // 激活相邻的 Tab
      activeFile.value = openTabs.value[Math.min(index, openTabs.value.length - 1)] || null
    }
  }

  /** 手动编辑保存 */
  function saveFile(tabId: string, content: string): void {
    const tab = openTabs.value.find(t => t.id === tabId)
    if (!tab) return
    tab.content = content
    tab.isDirty = false

    // 同步更新文件树中的节点
    const node = findNodeByPath(fileTree.value, tab.path)
    if (node) {
      node.content = content
      node.status = 'modified'
    }

    // 调用后端 API 保存
    // projectApi.saveFile(projectId.value, tab.path, content)
  }

  // ===== 内部辅助方法 =====
  function findOrCreateFolder(nodes: FileNode[], folderPath: string): FileNode[] | null {
    if (!folderPath) return nodes

    const parts = folderPath.split('/')
    let current: FileNode[] = nodes

    for (const part of parts) {
      let folder = current.find(n => n.type === 'folder' && n.name === part)
      if (!folder) {
        folder = {
          id: part,
          name: part,
          path: part,
          type: 'folder',
          children: []
        }
        current.push(folder)
      }
      folder.children = folder.children || []
      current = folder.children
    }

    return current
  }

  function findNodeByPath(nodes: FileNode[], path: string): FileNode | null {
    for (const node of nodes) {
      if (node.path === path) return node
      if (node.children) {
        const found = findNodeByPath(node.children, path)
        if (found) return found
      }
    }
    return null
  }

  function countFiles(nodes: FileNode[]): number {
    let count = 0
    for (const node of nodes) {
      if (node.type === 'file') count++
      if (node.children) count += countFiles(node.children)
    }
    return count
  }

  function findNodesByPath(nodes: FileNode[], prefix: string): FileNode[] {
    return nodes.filter(n => n.path.startsWith(prefix))
  }

  function expandAll(nodes: FileNode[]): void {
    for (const node of nodes) {
      if (node.type === 'folder') {
        expandedFolders.value.add(node.path)
        if (node.children) expandAll(node.children)
      }
    }
  }

  function inferLanguage(fileName: string): string {
    const ext = fileName.split('.').pop()?.toLowerCase()
    const map: Record<string, string> = {
      ts: 'typescript', tsx: 'typescript', js: 'javascript', jsx: 'javascript',
      css: 'css', json: 'json', html: 'html', md: 'markdown',
      yml: 'yaml', yaml: 'yaml', xml: 'xml'
    }
    return map[ext || ''] || 'text'
  }

  function reset(): void {
    projectId.value = null
    projectName.value = ''
    fileTree.value = []
    activeFile.value = null
    openTabs.value = []
  }

  return {
    projectId, projectName, fileTree, activeFile, openTabs, expandedFolders,
    fileCount, pageCount, apiCount,
    initFileTree, updateFileContent, openFile, closeTab, saveFile, reset
  }
})
```

### 3.4 useSandboxStore——沙箱状态管理

`useSandboxStore` 管理沙箱的运行状态、预览 URL、日志列表：

```typescript
// src/stores/sandbox.ts
import { defineStore } from 'pinia'
import { ref, computed } from 'vue'

export type SandboxStatus = 'stopped' | 'starting' | 'running' | 'error'

export interface LogEntry {
  id: string
  timestamp: number
  level: 'info' | 'success' | 'warn' | 'error' | 'debug'
  message: string
  source: 'build' | 'runtime' | 'deploy' | 'system'
}

export const useSandboxStore = defineStore('sandbox', () => {
  // ===== State =====
  const status = ref<SandboxStatus>('stopped')
  const url = ref<string | null>(null)
  const port = ref<number | null>(null)
  const logs = ref<LogEntry[]>([])
  const buildTime = ref<number | null>(null)
  const isRefreshing = ref(false)

  // WebSocket 连接
  let wsConnection: WebSocket | null = null

  // ===== Getters =====
  const isRunning = computed(() => status.value === 'running')
  const hasError = computed(() => status.value === 'error')
  const logCount = computed(() => logs.value.length)

  // ===== Actions =====

  /** 设置预览 URL（SSE complete 事件触发） */
  function setPreviewUrl(previewUrl: string): void {
    url.value = previewUrl
    status.value = 'running'
  }

  function setStatus(newStatus: SandboxStatus): void {
    status.value = newStatus
  }

  /** 追加日志条目 */
  function appendLog(source: LogEntry['source'], message: string, level: LogEntry['level'] = 'info'): void {
    logs.value.push({
      id: crypto.randomUUID(),
      timestamp: Date.now(),
      level,
      message,
      source
    })
  }

  /** 连接 WebSocket 获取沙箱实时日志 */
  function connectLogStream(projectId: string): void {
    const wsUrl = `${window.location.protocol === 'https:' ? 'wss:' : 'ws:'}//${window.location.host}/ws/sandbox/${projectId}/logs`
    wsConnection = new WebSocket(wsUrl)

    wsConnection.onmessage = (event) => {
      try {
        const data = JSON.parse(event.data)
        appendLog(data.source || 'runtime', data.message, data.level || 'info')
      } catch {
        appendLog('system', event.data, 'info')
      }
    }

    wsConnection.onerror = () => {
      appendLog('system', '日志 WebSocket 连接异常', 'warn')
    }
  }

  function disconnectLogStream(): void {
    wsConnection?.close()
    wsConnection = null
  }

  /** 刷新预览 */
  function refreshPreview(): void {
    isRefreshing.value = true
    // 通过修改 iframe src 加时间戳强制刷新
    if (url.value) {
      const separator = url.value.includes('?') ? '&' : '?'
      url.value = `${url.value}${separator}_t=${Date.now()}`
    }
    setTimeout(() => { isRefreshing.value = false }, 1000)
  }

  /** 清除日志 */
  function clearLogs(): void {
    logs.value = []
  }

  function reset(): void {
    status.value = 'stopped'
    url.value = null
    port.value = null
    logs.value = []
    buildTime.value = null
    disconnectLogStream()
  }

  return {
    status, url, port, logs, buildTime, isRefreshing,
    isRunning, hasError, logCount,
    setPreviewUrl, setStatus, appendLog,
    connectLogStream, disconnectLogStream,
    refreshPreview, clearLogs, reset
  }
})
```

### 3.5 Store 之间的协调关系

三个 Store 不是孤立的。在 SSE 事件处理过程中，它们之间有明确的协调关系：

```
SSEMessage.onMessage(msg)
    │
    ├──→ chatStore.handleSSEMessage(msg)
    │       │
    │       ├─→ 更新 checklistState[msg.nodeName] = 'done'
    │       ├─→ 更新 checklistState[上一个阶段] = 'done'
    │       ├─→ 更新 checklistState[当前阶段] = 'running'
    │       └─→ messages.push(assistantMessage)
    │
    ├──→ (msg.nodeName === 'execution_planning')
    │       └─→ projectStore.initFileTree(plan.generationOrder)
    │
    ├──→ (msg.nodeName === 'code_generation')
    │       └─→ projectStore.updateFileContent(path, content)
    │
    └──→ (msg.nodeName === 'preview_deploy')
            └─→ sandboxStore.setPreviewUrl(url)
```

**Store 之间不直接互相调用 action**——它们通过"共享的事件处理函数"（`useGenerationStream` composable）来协调。这样做的好处是每个 Store 保持职责单一，协调逻辑集中在 composable 中，方便测试和修改。

## 四、面板一：ChatPanel——AI 对话面板

### 4.1 ChatPanel 不是聊天框

很多开发者把 ChatPanel 理解为"一个类似 ChatGPT 的聊天界面"。这个理解只对了一半。灵码工坊的 ChatPanel 确实有对话气泡和输入框，但它和 ChatGPT 有关键区别：

**ChatGPT 的对话是纯文本的——消息就是文字。灵码工坊的对话是多种内容类型的混合流**——消息可以是 AI 的思考文字（TEXT）、可以是结构化 JSON（需求分析结果）、可以是 Markdown 代码片段（生成的代码预览）、可以是文件变更列表（本次生成了哪几个文件）。这些不同类型的消息需要不同的渲染方式——JSON 需要折叠展示、代码需要语法高亮、文件变更列表需要可点击跳转到编辑器。

**ChatPanel 的第二个关键区别是它不只是一个"对话界面"——它同时承载了「生成进度清单」和「文件变更列表」。** 这两个元素不是单独的面板，而是嵌入在对话流中的——在 AI 的回复气泡里，先看到进度清单逐项点亮，再看到文件变更列表，最后看到 AI 的文字总结。这种"对话 + 进度 + 文件变更"三位一体的布局是灵码工坊对话区区别于普通聊天框的核心特征。

### 4.2 消息类型与渲染策略

基于 `contentType` 字段，ChatPanel 的消息气泡有不同的渲染方式：

| contentType | 渲染组件 | 渲染方式 |
|------------|---------|---------|
| `TEXT` | `TextBubble` | 纯文本，支持换行和基本格式化 |
| `JSON` | `JsonBubble` | 折叠式 JSON 查看器，支持展开/折叠、语法高亮 |
| `MARK_DOWN` | `MarkdownBubble` | `markdown-it` 渲染为 HTML，代码块高亮 |
| `HTML` | `HtmlBubble` | 预留，iframe 内嵌渲染 |

### 4.3 流式消息的渲染

流式渲染是 ChatPanel 最关键的体验细节。后端 SSE 推送的 `text` 是增量内容——不是每次推送全文，而是每次追加一段新文字。前端需要把这些增量内容拼接到同一个 AI 消息气泡中，而不是每次推送都创建一个新的气泡。

**流式拼接的关键是"同一个 message ID"。** 当第一个 SSE message 事件到达时（nodeName 从上一个阶段变为新阶段），前端创建一个新的 assistant 消息，分配一个 ID。后续同一个 nodeName 的 SSE message 事件到达时，前端根据 nodeName 找到最后一条同 nodeName 的 assistant 消息，把 text 追加到它的 content 后面。

```typescript
// 在 chatStore 中实现流式拼接
function handleSSEMessage(msg: SSEMessage): void {
  const nodeName = msg.nodeName

  // 检查上一条消息是否是同一个 nodeName 的 assistant 消息
  const lastMsg = messages.value[messages.value.length - 1]
  if (lastMsg && lastMsg.role === 'assistant' && lastMsg.nodeName === nodeName) {
    // 流式追加：更新最后一条消息的内容
    lastMsg.content += msg.text
  } else {
    // 新阶段：创建新的 assistant 消息
    messages.value.push({
      id: crypto.randomUUID(),
      role: 'assistant',
      content: msg.text,
      contentType: msg.textType,
      timestamp: Date.now(),
      nodeName
    })
  }
}
```

**注意：** 这种流式拼接方式有一个隐含要求——后端的 SSE 推送必须保证同一个阶段的连续推送之间，nodeName 不变。如果后端在需求分析阶段中间插入了其他节点的事件，前端的拼接逻辑会出错。这是一个前后端协约——后端保证事件流的有序性，前端依赖这个约定做流式拼接。

## 五、面板二：GenerationChecklist——六阶段进度清单

## 六、面板三：FileTreePanel——动态文件树

### 6.1 文件树不是一次性渲染的

普通的文件树是静态的——后端返回一个文件列表，前端一次性渲染。灵码工坊的文件树是**动态生长**的——先出现文件夹骨架（执行规划阶段），然后文件节点按照生成顺序逐个冒出来（代码生成阶段）。

这个"生长"效果是产品体验的核心——用户直观地感受到"AI 正在创造文件"，而不是"等 30 秒然后文件树突然满了"。**动态生长不是视觉效果，而是信息透明——让用户知道 AI 在做什么、做到哪了、还有多少没做。**

### 6.2 文件树的三种状态

文件树中的每个文件节点有三种状态（由 `FileNode.status` 表示）：

| 状态 | 含义 | 视觉表现 |
|------|------|---------|
| `new` | 新生成的文件 | 文件名旁有一个绿色的圆点（NEW 标记） |
| `modified` | 被修改过的文件（迭代修改阶段） | 文件名旁有一个橙色的圆点（MOD 标记） |
| `unchanged` | 未被修改的已有文件 | 无标记 |

### 6.3 文件生成的"生长"动画

当 SSE `code_generation` 事件到达时，`projectStore.updateFileContent()` 被调用，文件节点的内容从空字符串变为实际代码。这个变化通过 Vue 的响应式系统自动触发 DOM 更新。为了让用户感知到"新文件出现了"，可以给文件节点加一个 CSS 过渡动画：

```css
.file-tree-node--new {
  animation: file-appear 0.4s ease;
}

@keyframes file-appear {
  0% {
    opacity: 0;
    transform: translateX(-8px);
  }
  100% {
    opacity: 1;
    transform: translateX(0);
  }
}
```

**"生长"动画的关键时机**：文件节点的动画应该在 `status` 从 `undefined` 变为 `'new'` 时触发，而不是在文件节点创建时触发。因为执行规划阶段会一次性创建所有文件节点（status 为 undefined），这些节点不应该有动画——它们只是"预留位置"。只有代码生成阶段逐个填充内容时（status 变为 'new'），文件节点才应该有出现动画。

## 七、面板四：EditorPanel——Monaco 代码编辑器

### 7.1 为什么选 Monaco Editor

Monaco Editor 是 VS Code 使用的同款编辑器引擎。选择它不是因为它"功能最全"，而是因为它**天然支持灵码工坊需要的三种编辑模式**：

1. **代码查看模式**（M2）——只读展示 AI 生成的代码，语法高亮
2. **代码编辑模式**（M3）——用户手动修改代码，智能补全、错误提示
3. **Diff 对比模式**（M4）——迭代修改时展示新旧代码对比（类似 Git diff）

这三种模式在 Monaco Editor 中都是内置能力——不需要自己实现 Diff 算法、不需要自己写语法高亮引擎、不需要自己解析 TypeScript 类型。用第三方编辑器组件可能需要分别处理这三种模式，而 Monaco 一个 API 全部覆盖。

### 7.2 Monaco Editor 的集成方式

不推荐直接用 `monaco-editor` 的裸 API，推荐使用 `@guolao/vue-monaco-editor` 这个 Vue 3 封装库。它处理了 Monaco 的异步加载、Web Worker 配置、主题切换等繁琐的集成细节：

```bash
npm install @guolao/vue-monaco-editor
```

### 7.3 代码实时注入——SSE 事件的编辑器联动

当 SSE `code_generation` 事件到达时，后端推送了文件路径和文件内容。前端的处理流程是：

1. `chatStore.handleSSEMessage()` 创建 assistant 消息（ChatPanel 显示"正在生成 xxx.tsx"）
2. `projectStore.updateFileContent(path, content)` 更新文件树节点
3. 如果该文件正在编辑器中打开（`openTabs` 中有对应 Tab），编辑器的 `editorContent` 自动更新

**这里的关键是 Vue 的响应式系统**——`projectStore.updateFileContent()` 修改了 `fileNode.content`，如果这个 `fileNode` 对应的 Tab 正在编辑器中打开，`activeFile.content` 是同一个响应式引用，编辑器会自动重新渲染。不需要额外的事件通知或手动调用编辑器的 `setValue()`。

但有一个细节需要注意：**如果用户正在手动编辑该文件**（Tab 的 `isDirty = true`），SSE 推送的内容不应该覆盖用户的修改。此时 `projectStore.updateFileContent()` 应该跳过编辑器同步：

```typescript
// 修改 projectStore.updateFileContent()
function updateFileContent(path: string, content: string): void {
  const node = findNodeByPath(fileTree.value, path)
  if (!node) return

  node.content = content
  node.status = 'modified'

  // 如果用户正在编辑该文件（dirty），不覆盖编辑器的内容
  const tab = openTabs.value.find(t => t.path === path)
  if (tab && !tab.isDirty) {
    tab.content = content
  }
}
```

### 7.4 Diff 模式——迭代修改的可视化

当用户进入迭代修改流程（M4），后端通过 `patchFile` 工具修改了某个文件。前端需要展示"改了什么"——不是在对话区用文字描述，而是在编辑器中用 Diff 模式高亮显示改动行。

Diff 模式的使用场景：
- 用户说"把套餐价格改成 ¥29/¥59/¥99"，AI 定位到 PlanCard.tsx 并调用 `patchFile` 修改了 3 行
- 前端收到 SSE 事件，该文件的 `status` 变为 `modified`
- 编辑器自动切换到 Diff 模式：左边是修改前的内容、右边是修改后的内容

```typescript
// 在 EditorPanel 中响应文件修改
function handleFileModified(filePath: string, oldContent: string, newContent: string) {
  const file = activeFile.value
  if (file && file.path === filePath) {
    diffOriginal.value = oldContent
    editorContent.value = newContent
    isDiffMode.value = true
  }
}
```

## 八、面板五：PreviewPanel——沙箱预览面板

### 8.1 预览不是简单的 iframe

如果只是"把一个 URL 放到 iframe 里"，那 PreviewPanel 的实现就太简单了。灵码工坊的 PreviewPanel 需要处理的场景更复杂：

1. **加载状态**：沙箱容器启动需要时间（npm install + npm build + Vite Dev Server 启动），在这段时间里 iframe 不能是空白——需要展示一个加载动画或骨架屏
2. **构建失败**：如果沙箱构建失败，预览区不能展现一个错误页面——需要展示友好的错误信息（"构建失败，AI 正在修复..."），并展示错误详情
3. **跨域限制**：沙箱容器的 URL 和前端不在同一个域（`https://sandbox-xxx.lingmaforge.dev` vs `https://lingmaforge.com`），iframe 的 `sandbox` 属性需要正确配置
4. **设备切换**：用户可以在桌面端和手机端视角之间切换预览——通过改变 iframe 容器的宽度来模拟不同设备
5. **刷新预览**：用户在编辑器中改了代码后，预览区需要刷新

### 8.2 预览面板的状态机

PreviewPanel 有五种状态：

```
idle ──生成开始──→ loading
loading ──构建成功──→ running
loading ──构建失败──→ error
running ──用户刷新──→ running (reload)
running ──生成停止──→ idle
error ──AI 修复重试──→ loading
```

## 九、面板六：LogPanel——实时日志面板

### 9.1 日志面板的数据源

日志面板的数据来源于两个通道：

1. **SSE `build_verification` 事件**——构建阶段的关键日志（安装依赖、构建成功/失败）
2. **WebSocket `/ws/sandbox/{projectId}/logs`**——沙箱容器的实时日志流（npm install 的详细输出、Vite Dev Server 的启动日志）

SSE 只推送关键的构建状态（开始安装、安装完成、构建开始、构建结果），WebSocket 推送详细的过程日志（每个 npm 包的安装进度、TypeScript 编译警告、模块解析日志）。

### 9.2 日志的分类与着色

日志按严重程度分为四个级别，每个级别有不同的颜色：

| 级别 | 颜色 | 用于 |
|------|------|------|
| `info` | 蓝色 | 一般信息（"安装依赖包..."） |
| `success` | 绿色 | 成功信息（"依赖安装完成"、"构建成功"） |
| `warn` | 黄色 | 警告（"package.json 中未声明 xxx 依赖"） |
| `error` | 红色 | 错误（"构建失败: TS2307"） |

日志还按来源（Tab）分为四类：构建、运行、部署、用量。实际展示时，用户可以切换 Tab 只看某一类日志。


### 9.3 日志性能——避免渲染大量 DOM

长时间运行时，日志可能会堆积上千条。如果每条日志都渲染为一个 DOM 节点，页面会变得很卡。解决方案是**虚拟滚动**——只渲染可视区域内的日志条目。

对于灵码工坊的场景，虚拟滚动可以用 `vue-virtual-scroller` 实现，或者更简单地：限制日志的最大条数为 500 条，超过后自动删除最旧的日志。因为用户通常只关注最近的日志，历史日志可以折叠或按需加载：

```typescript
// 在 sandboxStore.appendLog() 中限制最大条数
function appendLog(source: LogEntry['source'], message: string, level: LogEntry['level'] = 'info'): void {
  logs.value.push({
    id: crypto.randomUUID(),
    timestamp: Date.now(),
    level,
    message,
    source
  })

  // 限制最大 500 条
  if (logs.value.length > 500) {
    logs.value.splice(0, logs.value.length - 500)
  }
}
```

## 十一、迭代修改——前端怎么体现"改了一行代码"

### 11.1 迭代修改与首次生成的区别（前端视角）

首次生成是"从无到有"——文件树从空到满，编辑器从空到有代码，预览区从空白到显示应用。迭代修改是"从有到优"——文件树中某些文件的状态从 `unchanged` 变为 `modified`，编辑器切换到 Diff 模式展示改动行，预览区刷新显示修改后的效果。

前端的核心挑战是：**让用户一眼看到"改了什么"——不是重新渲染整个界面，而是在已有界面上高亮变化的部分。**

### 11.2 迭代修改的 SSE 事件处理

迭代修改的 SSE 事件流和首次生成不同——它走的是短流水线（迭代意图理解 → 代码定位 → 修改生成 → 构建验证 → 预览部署）。对应的 nodeName 也不同：

| nodeName | 前端响应 |
|----------|---------|
| `iteration_intent` | ChatPanel 展示"正在理解你的修改意图..." |
| `code_locating` | ChatPanel 展示"正在定位需要修改的代码..." |
| `modification_generation` | FileTreePanel 标记文件为 modified + EditorPanel 切换 Diff 模式 |
| `build_verification` | LogPanel 展示构建日志 |
| `preview_deploy` | PreviewPanel 加载刷新后的 URL |

**最关键的是 `modification_generation` 事件的处理**——这个事件携带的数据结构和 `code_generation` 不同：

```json
{
  "nodeName": "modification_generation",
  "text": "已修改 2 个文件",
  "modifications": [
    {
      "path": "src/components/PlanCard.tsx",
      "type": "patch",
      "patches": [
        { "line": 34, "old": "price: 49", "new": "price: 29" }
      ]
    }
  ]
}
```

前端需要：
1. 在文件树中标记 `PlanCard.tsx` 为 `modified`
2. 如果 `PlanCard.tsx` 正在编辑器中打开，记录旧内容并切换到 Diff 模式
3. 在 ChatPanel 中展示变更文件列表（可点击跳转到编辑器）

### 11.3 Diff 模式的前端实现

Diff 模式不是自动触发的——用户需要主动选择是否查看 Diff。当文件被修改后，编辑器 Tab 上显示一个"MOD"标记。用户点击该标记，编辑器切换到 Diff 模式：左边是修改前的代码（从版本快照中恢复），右边是修改后的代码。

```typescript
// projectStore 中维护版本快照
function takeSnapshot(filePath: string): void {
  const node = findNodeByPath(fileTree.value, filePath)
  if (node?.content) {
    snapshots.value[filePath] = node.content
  }
}

function showDiff(filePath: string): void {
  const original = snapshots.value[filePath]
  const modified = findNodeByPath(fileTree.value, filePath)?.content
  if (original && modified) {
    // 触发 EditorPanel 的 Diff 模式
    diffFile.value = { path: filePath, original, modified }
  }
}
```

## 十二、API 层设计——HTTP 与 SSE 的统一封装

### 12.1 为什么需要统一的 API 层

当前前端原型中没有 API 层——所有数据都是硬编码的。接入真实后端后，API 调用散落在各个组件中会导致：
- 组件耦合了 API 实现细节（URL、请求方法、参数格式）
- 无法统一处理错误（401 跳转登录、网络超时重试）
- 无法统一管理请求状态（loading / error / success）

**API 层的作用是把"怎么发请求"从"发什么请求"中分离出来。** 组件只关心"调用 `generationApi.create({ prompt })` 拿到 `taskId`"，不关心请求是 POST 还是 GET、URL 是什么、headers 里带什么 token。

### 12.2 API 模块的目录结构

```
src/api/
├── request.ts              # Axios 实例 + 拦截器
├── generation.ts           # 生成相关 API
├── project.ts              # 项目 & 文件 API
├── sandbox.ts              # 沙箱 API
└── auth.ts                 # 认证 API（M5 阶段启用）
```

### 12.3 Axios 实例配置

```typescript
// src/api/request.ts
import axios from 'axios'
import { useAuthStore } from '@/stores/auth'

const request = axios.create({
  baseURL: import.meta.env.VITE_API_BASE_URL || '/api',
  timeout: 15000,  // 15 秒超时
  headers: {
    'Content-Type': 'application/json'
  }
})

// 请求拦截器：注入 token（M5 阶段启用）
request.interceptors.request.use(config => {
  const authStore = useAuthStore()
  if (authStore.accessToken) {
    config.headers.Authorization = `Bearer ${authStore.accessToken}`
  }
  return config
})

// 响应拦截器：统一错误处理
request.interceptors.response.use(
  response => {
    const { code, message, data } = response.data
    if (code === 0) {
      return data  // 成功时直接返回 data，简化调用方代码
    }
    // 业务错误
    return Promise.reject(new Error(message || '请求失败'))
  },
  error => {
    // 网络错误或 HTTP 错误
    if (error.response?.status === 401) {
      // 401 处理：跳转登录（M5 阶段实现 token 自动刷新）
      // 开发阶段只打印日志
      console.warn('[API] 401 Unauthorized')
    }
    return Promise.reject(error)
  }
)

export default request
```

## 十三、类型系统——前端的数据契约

### 13.1 为什么需要统一的类型定义

当前原型中没有独立的类型文件——所有类型（`interface`）都内联在 `.vue` 文件的 `<script setup>` 中。这在开发原型时很快，但在多人协作和长期维护中会造成问题：

- **重复定义**：`Msg` 类型在 `SimpleMode.vue` 和 `GenerationMode.vue` 中各定义了一次，字段不完全一致
- **类型漂移**：后端 API 的返回类型变化后，前端各组件中的内联类型不会自动更新——需要在每个组件中逐一修改
- **无法做编译时校验**：修改后端 API 返回字段后，前端不会有任何编译错误——运行时才发现字段名不匹配

## 十四、组件通信模式——面板之间怎么对话

### 14.1 三种通信方式的使用场景

灵码工坊的五个面板之间有大量的数据交互。通信方式的选择直接影响代码的可维护性：

| 通信方式 | 用于 | 示例 |
|---------|------|------|
| **Pinia Store（共享状态）** | 跨面板数据共享 | 文件树节点变化 → 编辑器显示内容；沙箱 URL 变化 → 预览区加载 |
| **Composable（组合式函数）** | 公共行为逻辑 | `useGenerationStream` 处理 SSE 事件并分发到不同 Store |
| **Props + Emits（父子组件）** | 面板内部子组件通信 | 文件树中 `FileTreeNode` 点击 → emit `select-file` → `FileTreePanel` 调用 store action |

**原则：** Store 用于跨面板共享数据，Composable 用于公共行为逻辑，Props/Emits 仅用于组件树内部的局部通信。不要把跨面板的数据通过多层 Props/Emits 传递——那样会导致"Props 地狱"。

### 14.2 面板间的典型通信链路

**链路一：SSE 事件 → ChatPanel + FileTreePanel + EditorPanel**

```
SSEClient.onMessage(msg)
  → useGenerationStream.handleMessage(msg)
    → chatStore.handleSSEMessage(msg)          // 更新对话历史
    → projectStore.updateFileContent(path, c)  // 更新文件树节点内容
    → (editorTab 未 dirty) → 编辑器自动响应式更新
```

**链路二：文件树点击 → 编辑器**

```
FileTreePanel: @select-file(path)
  → projectStore.openFile(path)       // 创建 Tab + 设置 activeFile
  → EditorPanel: watch(activeFile)    // 响应式更新编辑器内容
```

**链路三：用户手动编辑保存 → 文件树 + 预览区**

```
EditorPanel: Ctrl+S
  → projectStore.saveFile(tabId, content)  // 更新文件树节点 + 标记 dirty=false
  → sandboxStore.refreshPreview()          // 刷新预览区
```

**链路四：工具栏操作 → 沙箱 + 预览区**

```
WorkbenchToolbar: 点击 [运行]
  → sandboxApi.start(projectId)
  → sandboxStore.setStatus('starting')
  → sandboxStore.connectLogStream(projectId)
  → (SSE preview_deploy 事件) → sandboxStore.setPreviewUrl(url)
  → PreviewPanel: watch(sandboxStore.url) → iframe 加载
```

## 十五、设计模式与最佳实践

### 15.1 Composable（组合式函数）——Vue 3 的函数式复用

Vue 3 的 Composition API 让逻辑复用从"混入（Mixin）"变成了"组合式函数（Composable）"。灵码工坊的前端应该充分利用这个模式：

```typescript
// src/composables/useGenerationStream.ts
// 封装 SSE 事件处理的核心逻辑
// 被 WorkbenchView 使用，协调三个 Store 的更新

// src/composables/useMonacoEditor.ts
// 封装 Monaco Editor 的初始化、主题切换、Diff 模式切换
// 被 EditorPanel 使用

// src/composables/useSplitPanes.ts
// 封装面板拖拽调整的响应式尺寸管理
// 被 WorkbenchView 使用
```

### 15.2 容器组件与展示组件分离

工作台页面容易变成一个 2000+ 行的巨型组件（`GenerationMode.vue` 当前已经有 1040 行）。应该拆分为：

- **容器组件**（WorkbenchView.vue）：管理布局、SSE 连接、Store 协调
- **展示组件**（ChatPanel, FileTreePanel, EditorPanel, PreviewPanel, LogPanel）：接收 props，emit 事件，不直接读取 Store（通过 composable 桥接）

### 15.3 错误边界——"一个面板崩溃不应该拖垮整个工作台"

如果代码编辑器因为某个文件的语法问题崩溃了，不应该导致整个工作台白屏。Vue 3 提供了 `onErrorCaptured` 钩子来做组件级别的错误边界：

```typescript
// 在每个面板组件的根元素上使用
import { onErrorCaptured, ref } from 'vue'

const panelError = ref<string | null>(null)

onErrorCaptured((error) => {
  panelError.value = error.message
  // 阻止错误向上冒泡到父组件
  return false
})
```

### 15.4 性能优化要点

1. **大文件树用 `v-memo`**：当文件树有 50+ 个节点时，使用 `v-memo` 跳过未变化节点的重新渲染
2. **日志面板用虚拟滚动**：当日志超过 500 条时，使用虚拟滚动避免渲染大量 DOM
3. **Monaco Editor 懒加载**：编辑器不在可视区域时不加载，使用 `v-if` + `IntersectionObserver`
4. **SSE 消息的批量处理**：如果短时间内收到多条 SSE 消息，使用 `requestAnimationFrame` 批量更新 DOM

```typescript
// SSE 消息的批量处理
let pendingUpdates: Array<() => void> = []
let rafId: number | null = null

function scheduleUpdate(update: () => void) {
  pendingUpdates.push(update)
  if (rafId === null) {
    rafId = requestAnimationFrame(() => {
      pendingUpdates.forEach(fn => fn())
      pendingUpdates = []
      rafId = null
    })
  }
}
```

### 15.5 多 Tab 冲突——同一个任务不能被两个窗口同时连接

SSE Sink 是 unicast（一对一），如果用户在两个浏览器 Tab 中打开同一个工作台并连接到同一个 taskId，第二个 Tab 的连接会失败。解决方案：

- 生成任务创建时，在 `localStorage` 中记录 `generation_task_{projectId}: taskId`
- 新 Tab 打开时检查是否存在活跃的 `taskId`，如果有则提示用户或自动接管
- SSE 连接失败时（`EventSource` 的 `error` 事件且 `readyState === CLOSED`），提示"任务已在另一个窗口运行"
