## 一、原型回顾与设计意图解读

### 1.1 原型页面布局

当前的 `lingma-auth.html` 采用了一种精心设计的非对称双栏布局，这是整个产品视觉体系中第一个需要深入理解的设计决策。

**左侧（展示区）**承担的是品牌传递和信任建立的角色。它不是一张简单的配图，而是一个完整的品牌叙事空间：底层是一张 16:9 的背景大图，以 `cover` 方式填充整个容器，`object-position` 锚定在左中位置；中层是一只漂浮的吉祥物形象，以 6 秒为周期的 `ease-in-out` 动画呈现出上下浮动的呼吸感，这个动效的巧妙之处在于它让整个登录页"活"了起来——用户在看到这张页面的第一秒，就已经在潜意识里感受到这个产品是有温度、有性格的；上层是三列特性展示文案，分别是「项目云同步」「AI 实时生成」「一键运行部署」，这三条信息的存在不是随意的，它们在原型里的定位是给尚在犹豫的潜在用户一个快速理解产品核心能力的入口。每一个特性由一个圆形图标容器 + 标题 + 副标题组成，图标容器采用了与品牌色一致的青色描边，底图是半透明白色，这种处理方式在玻璃态卡片上显得干净而不抢戏。

**右侧（交互区）**是登录卡片的所在地。这个卡片的视觉语言非常明确——它不是一张普通的白色面板，而是一张玻璃态（Glassmorphism）卡片。`background` 由三层径向渐变和半透明白色叠加而成，`backdrop-filter: blur(24px)` 让卡片后的背景内容产生柔和的模糊，最外层的伪元素 `::before` 用 `mask-composite: exclude` 技巧绘制了一条青金渐变的光泽描边。这些细节叠加在一起，给用户的感受是"这是一款现代、精致、值得信赖的产品"。

卡片的宽度被固定为 450px，位于右侧距离右边缘 40px 的位置，垂直居中于视口。在 992px 以下的断点，左侧展示区隐藏，卡片水平居中，适配移动端。

### 1.2 两级标签的设计逻辑

原型中有一个容易被忽略但逻辑正确的设计——**两级标签切换**。

第一级是「登录」和「注册」两个 Tab，切换的是整个卡片的上下文。当切换到「注册」时，标题从"继续你的创作"变为"加入灵码工坊"，副标题从"安全可靠的云端环境，守护你的每一次创意"变为"开始你的灵感设计，体验前所未有的开发体验"，提交按钮文字从"登录"变为"注册"，底部链接从"还没有账号？立即注册"变为"已有账号？立即登录"。这些文字变化看似微小，但它们共同构建了一个完整的场景切换——登录是"回家"，注册是"进门"，两块场景共用同一套 UI 骨架，但传递的情感完全不同。

第二级是「验证码登录」和「密码登录」两个 Method Tab，它们控制的是第二个输入框的行为。在验证码模式下，输入框显示"输入 6 位验证码"的占位文字，右侧出现一个"获取验证码"的按钮；在密码模式下，输入框的类型切换为 `password`，占位文字变为"输入登录密码"，右侧按钮隐藏。这一级切换只改变 UI 行为，不改变卡片的上下文（标题、副标题等保持不变）。

两级标签叠加后的合法状态组合是：登录+验证码、登录+密码、注册+验证码、注册+密码。但有一点需要特别注意——**注册模式下的密码登录 Tab 实际上应该走"设置密码"逻辑，而不是验证密码**。原型的 JavaScript 代码中没有区分这个场景，在正式实现中这是一个需要修正的交互问题。

### 1.3 OAuth 入口的设计意图

原型在表单下方设置了分隔线"或使用以下方式继续"，下方排列了三个 OAuth 按钮：GitHub、微信、Google。这三个按钮的设计意图需要结合产品定位来理解。

灵码工坊的目标用户包括独立开发者、设计师和初创团队。GitHub OAuth 瞄准的是开发者群体，这是最快降低注册门槛的方式——开发者不必再记一个新密码，用已有的 GitHub 账号一键登入即可。微信 OAuth 瞄准的是国内用户群体，特别是不太习惯用邮箱注册的非技术用户，微信扫码登录是中文互联网最熟悉的认证方式。Google OAuth 则照顾了海外用户和习惯使用 Google 生态的开发者。

从技术实现的角度，这三个 OAuth 提供者按照优先级分为两档：GitHub 和微信是第一优先级，需要在 M5 阶段就实现；Google 可以稍后补上。原因很直接——前两个覆盖了灵码工坊最重要的两组用户，Google 在没有明确海外需求前可以暂缓。

### 1.4 安全感的塑造细节

原型底部有一个容易被忽略但设计用心的元素——SSL 安全状态指示器。它由一个锁形 SVG 图标、文字"安全连接已建立 SSL 加密保护中"、以及一个带呼吸动画的绿色圆点组成。这个细节在产品设计层面的作用是给用户在输入密码/验证码之前打一针"安心剂"——告诉他这个页面是安全的、加密的、可以放心输入敏感信息的。

在实际开发中，这个模块不应该只是一个静态的装饰。它可以动态读取当前页面的协议类型（`https` 或 `http`），在 `http` 模式下显示为警告状态（黄色圆点 + "当前连接未加密"），在 `https` 模式下显示为正常状态。这样它就从一个纯装饰元素变成了一个真正有用的安全指示器。

## 二、认证体系架构总览

### 2.1 整体认证流程视角

灵码工坊的认证体系采用 **JWT（JSON Web Token）无状态认证架构**。选择这个方案而不是传统的 Session + Cookie 方案，有以下几个核心原因：

**第一，前后端分离的需要。** 前端是 Vue 3 SPA，后端是 Spring Boot REST API，两者通过 HTTP 通信。Session 方案要求浏览器自动在每次请求中携带 Cookie，这在前后端同域部署时可行，但灵码工坊未来可能需要将前端部署在 CDN 上、后端部署在独立的 API 子域名下，跨域 Cookie 管理的复杂度远高于 JWT 方案。JWT 的 token 由前端主动在请求头中携带，完全不依赖浏览器自动行为，跨域场景下的适配成本为零。

**第二，无状态的可扩展性。** 如果未来用户量增长到需要水平扩展后端服务，Session 方案需要引入 Redis 共享 Session 或在负载均衡器上配置会话保持（Sticky Session）。JWT 方案不需要——token 本身就是自包含的，任何一台后端服务器拿到 token 都能独立完成身份校验，无需查询共享存储。这在 M5 阶段可能感觉不到差异，但在 M7 之后当服务器需要扩容时，这个架构决策的价值就体现出来了。

**第三，与微服务架构的兼容性。** 虽然灵码工坊目前是一个单体 Spring Boot 应用，但如果未来需要将 AI 模块、部署模块拆分为独立微服务，JWT token 可以在服务间无缝传递，下游服务无需回源认证中心就能完成身份验证。

### 2.2 Token 的两种形态：Access Token 与 Refresh Token

一个成熟的 JWT 认证方案不应该只有一种 token，而应该采用**双 token 轮换机制**。

**Access Token（访问令牌）**是用户请求后端 API 时在 `Authorization` 头中携带的凭证。它由服务端在用户登录成功后签发，有效期为 **30 分钟**。短期有效的原因很直接——如果 token 被泄露，攻击者的利用窗口只有 30 分钟。这也是为什么 Access Token 的 payload 中通常不包含太多敏感信息，只包含 `userId`、`username`、`role` 这三个最核心的鉴权字段，以及 `iat`（签发时间）和 `exp`（过期时间）两个标准时间声明。

**Refresh Token（刷新令牌）**的作用是在 Access Token 过期后，免除用户重新输入密码的步骤。它的有效期比 Access Token 长得多，设置为 **7 天**。用户在登录时同时拿到两个 token：Access Token 存入 Pinia Store 和 `localStorage`（供页面刷新后恢复），Refresh Token 存入 `localStorage` 并标记为 `httpOnly` 不可访问（实际上 localStorage 做不到真正的 httpOnly，这一步需要后端配合设置 Cookie）。

双 token 轮换的典型流程是：前端每次发起 API 请求前，检查 Access Token 是否即将过期（比如剩余时间小于 5 分钟），如果是，则主动用 Refresh Token 换取新的 Access Token，然后用新 token 发起原请求。这个过程对用户完全透明——他不知道自己手里的 token 已经在后台刷新了好几次。只有当 Refresh Token 也过期时（7 天无操作），才需要用户重新登录。

## 三、技术选型与框架对比

### 3.1 前端认证相关依赖

| 依赖 | 用途 | 必要性 | 说明 |
|------|------|--------|------|
| `axios` | HTTP 客户端 | 必须 | 拦截器中注入 token、处理 401 刷新逻辑 |
| `pinia` | 状态管理 | 必须 | 存储 token、用户信息、登录状态 |
| `pinia-plugin-persistedstate` | Pinia 持久化 | 强烈推荐 | 自动将 store 数据同步到 localStorage，页面刷新不丢失 |
| `vue-router` | 路由守卫 | 必须 | `beforeEach` 守卫控制页面访问权限 |
| `js-cookie` | Cookie 操作 | 可选 | 如果不使用 localStorage 存储 token，改用 Cookie |

### 3.2 为什么选择 Vue 3 的 Pinia 而不是 Vuex

Pinia 是 Vue 3 生态里推荐的状态管理方案，相比 Vuex 有几个在认证场景下特别有价值的优势。

**类型推断更友好。** Pinia 从一开始就是用 TypeScript 设计的，Store 中的 state、getters、actions 都有完整的类型推导。对于认证这种涉及多种数据类型（token 是字符串、用户信息是对象、登录状态是枚举）的场景，类型安全可以避免大量运行时错误。

**没有 Mutations 的概念。** Vuex 的 action → mutation → state 三层架构在 Pinia 中被简化为 action → state 两层。这在认证场景中减少了大量样板代码——比如用户登录就是一个 action 调用 API 然后直接修改 state，不需要额外定义一个 mutation。

**持久化插件生态完善。** `pinia-plugin-persistedstate` 可以以 Store 为粒度配置持久化策略。Auth Store 需要持久化到 localStorage（token 和用户信息需要在页面刷新后保留），但 Chat Store 就不需要（对话历史在 M5 之后走后端持久化）。这种细粒度的控制能力是认证方案中非常重要的。

### 3.3 后端认证框架选择：Spring Security + JWT

Spring Security 是 Java 生态中事实上的认证和授权标准。选择它不是因为它是"最流行"的，而是因为它在灵码工坊的架构下有不可替代的价值。

**过滤器链机制天然适合 JWT。** Spring Security 的 `SecurityFilterChain` 是在请求到达 Controller 之前的一道可编程的拦截层。我们可以在过滤器中解析 `Authorization` 头中的 Bearer Token，验证签名和有效期，然后将用户信息写入 `SecurityContext`。后续的 Controller 方法可以通过 `@AuthenticationPrincipal` 注解直接获取当前登录用户，无需在每个方法中手动解析 token。

**灵活的路径级权限控制。** `SecurityFilterChain` 支持基于 URL 模式的权限配置。认证相关的接口（`/auth/**`）对所有请求放行；项目相关的接口（`/projects/**`）要求已认证；部署相关的接口（`/deploy/**`）不仅要求认证，还要求用户有对应的订阅方案。这种声明式的权限控制在业务增长后非常容易扩展。

**与 OAuth 2.0 的天然集成。** Spring Security 内置了 OAuth 2.0 Client 支持。配置 GitHub 或微信的 OAuth 回调只需要注册一个 `ClientRegistration` Bean 和几行配置，不需要手写 OAuth 协议的握手流程。这在 M5 阶段实现第三方登录时能节省大量时间。

### 3.4 JWT 库选择：jjwt（Java JWT）

在 Java 生态中处理 JWT 有两个主流选择：`jjwt`（io.jsonwebtoken）和 Nimbus JOSE + JWT。对于灵码工坊的场景，`jjwt` 更为合适。

`jjwt` 的 API 设计非常直观——`Jwts.builder()` 构建 token，`Jwts.parser()` 解析 token。它支持 HMAC-SHA 系列和 RSA 系列签名算法，对于灵码工坊来说，使用 `HMAC-SHA256` 算法配合一个 256 位以上的密钥就足够了。这个密钥存储在 `application.yml` 中，通过 Spring 的 `@Value` 注解注入到 JWT 工具类中。生产环境中，密钥绝对不能硬编码在代码仓库中，必须通过环境变量或配置中心注入。

## 四、用户登录全流程设计

### 4.1 场景一：邮箱密码注册

这是所有认证流程中最基础的一条路径。用户在 `/auth` 页面点击「注册」Tab，输入邮箱地址，切换到密码登录模式，输入密码，勾选"记住我"（或保持默认勾选状态），点击「注册」按钮。

前端在这一步需要做三件事。第一，表单验证——邮箱格式是否合法（正则 `/^[^\s@]+@[^\s@]+\.[^\s@]+$/` 做基本校验），密码长度是否达到最低要求（至少 8 位），两次密码是否一致（原型里没有确认密码字段，这是一个需要补充的交互——否则用户一旦输错密码，注册后无法登录，只能走忘记密码流程）。第二，防重复提交——点击注册后按钮进入 loading 状态，`disabled` 为 true，避免用户多次点击导致后端收到重复请求。第三，调用注册 API——`POST /api/auth/register`，携带 `{ email, password, username }`。

后端收到注册请求后，按照以下顺序执行：首先检查邮箱是否已被注册——查询 `user` 表的 `email` 字段，如果存在则返回 `"该邮箱已被注册"` 的错误信息。然后使用 Spring Security 提供的 `BCryptPasswordEncoder` 对密码进行哈希处理，`BCrypt` 的 `encode()` 方法会自动生成一个随机盐值并嵌入到哈希结果中，存储到 `password_hash` 字段。接着生成一个随机的用户名（如果前端没有传 `username`），比如 `"用户" + 随机6位数字`。最后将用户记录插入 `user` 表，为该用户分配默认的 Free 订阅方案，调用 `JwtUtil.generateToken()` 生成 Access Token 和 Refresh Token，将两个 token 和用户基本信息返回给前端。

前端收到成功响应后，将两个 token 和 `user` 对象存入 Pinia Auth Store，同时持久化到 `localStorage`。然后检查 URL 中是否有 `redirect` 参数——如果有（比如用户在首页点了 prompt 输入后触发了登录），跳转到 `redirect` 指定的路径并携带原来的上下文信息；如果没有，跳转到 `/workbench`，进入工作台简洁模式。

### 4.2 场景二：验证码登录

验证码登录是原型中默认的登录方式，它的设计初衷是降低登录门槛——用户不需要记住密码，只需要输入手机号或邮箱，获取一条 6 位验证码，输入后即可登录。对于不常用密码登录的国内用户来说，这种方式比密码登录更友好。

用户输入邮箱后，点击「获取验证码」按钮。前端先校验邮箱格式，通过后向后端发起请求 `POST /api/auth/send-code`。后端生成一个 6 位随机数字验证码，使用 `String.format("%06d", random.nextInt(1000000))` 保证不足 6 位时前面补零。验证码存入 Redis，key 的格式是 `code:{email}`，设置 TTL 为 300 秒（5 分钟）。同时调用邮件服务（或短信服务）向用户发送验证码。为了安全，同一个邮箱在 60 秒内不允许重复发送，这个限制通过在 Redis 中额外存储一个 `code:cooldown:{email}` 的 key 来实现。

前端在用户点击「获取验证码」后，按钮进入 60 秒的倒计时状态，文字显示"59s 后重新获取"并置灰不可点击。这个倒计时由 Vue 的响应式变量驱动，与后端的冷却时间无关——后端的冷却时间是为了安全，前端的倒计时是为了用户体验。两者独立运行，互不干扰。

用户输入验证码后点击「登录」。后端收到请求，从 Redis 中取出 `code:{email}` 的值与用户输入比对。如果 Redis 中不存在该 key（验证码已过期或根本没有发送过），返回"验证码已过期"；如果存在但不匹配，返回"验证码错误"；如果匹配，删除该验证码（一次性使用），然后执行登录逻辑——如果该邮箱对应的用户存在，直接生成 token 返回；如果不存在，自动为该邮箱注册一个新用户（验证码登录的隐含语义是"这个邮箱就是我，我用验证码证明了这一点"），随机生成一个用户名和密码（密码用 BCrypt 加密，但用户不会知道这个密码——后续可以在个人中心设置密码），然后生成 token 返回。

### 4.3 场景三：首页 prompt 触发登录

这是一个特殊的交互路径，也是灵码工坊区别于普通 SaaS 产品的关键体验差异化点。

用户在首页的 Hero 区输入框中输入"做一个待办事项应用"，点击发送按钮。首页代码首先检查用户是否已登录——通过读取 Pinia Auth Store 的 `isAuthenticated` 状态判断。如果已登录，直接跳转到 `/workbench?prompt=做一个待办事项应用`，工作台页面在 `mounted` 时读取 `prompt` 参数，自动填入对话输入框并发送。整个流程对已登录用户来说是无缝的。

如果未登录，跳转到 `/auth?redirect=/workbench&prompt=做一个待办事项应用`。登录页面在加载时读取 `redirect` 和 `prompt` 参数，将它们暂存在组件状态中。用户完成注册或登录后，前端构造完整的跳转 URL：先从 `redirect` 中取出目标路径（`/workbench`），再将 `prompt` 参数拼接上去，形成一个完整的带参跳转。这样用户登录后进入工作台，看到的就是一个自动发送了 prompt 的对话区，而不是一个空的输入框。这个体验的流畅感来自于"上下文不丢失"——用户在首页触发的那个想法，经过登录这一跳后，依然完好地传递到了工作台。

### 4.4 场景四：创意中心克隆触发登录

用户在创意中心浏览模板，对一个通讯录模板产生了兴趣，点击「克隆到工作台」。前端检查登录状态，如果未登录，跳转到 `/auth?redirect=/creative&cloneTemplate=3`。

与首页 prompt 场景不同的是，这个场景携带的不是文本 prompt，而是一个模板 ID。用户登录后，前端检查 `cloneTemplate` 参数存在且有效，调用 `POST /api/projects/clone` 接口，传入模板 ID，后端为该用户创建一个该模板的副本项目。创建成功后跳转到 `/workbench?projectId=新项目ID`，工作台以预览模式加载该项目——文件树已填充模板代码，预览区已渲染结果，对话区置于 AI 已就绪的空状态，用户可以立即开始修改。

### 4.5 场景五：页面间浏览时的认证状态检查

用户在未登录状态下浏览首页、创意中心、价格方案、文档中心这些不需要登录的页面，不会触发任何认证检查。但当用户点击导航栏中的「我的作品」或「个人中心」这些需要登录的页面时，路由守卫介入。

路由守卫（`router.beforeEach`）的逻辑是：首先检查目标路由的 `meta.requiresAuth` 是否为 `true`。如果不需要认证，直接放行。如果需要认证，检查 Pinia Auth Store 中是否存在有效的 Access Token。如果 token 存在且未过期，放行。如果 token 不存在或已过期，尝试用 Refresh Token 刷新——调用 `POST /api/auth/refresh`，如果刷新成功，更新 token 后放行；如果刷新也失败，跳转到 `/auth?redirect=原路径`，用户登录后自动回到他想去的页面。

这里有一个细节需要注意——路由守卫中不做用户信息的异步加载。用户信息（昵称、头像等）如果因为 localStorage 中没有存储（比如首次登录后刷新页面），不应该阻塞路由跳转。路由守卫只检查 token 是否存在且有效，用户信息的加载应该在 `App.vue` 的 `onMounted` 中异步完成，加载失败不影响路由。

## 五、前端实现方案

### 5.1 AuthPage.vue 组件架构

AuthPage 是整个登录系统前端的核心容器。它的组件树设计应该遵循"状态上提、渲染下沉"的原则——所有认证相关的状态（当前是登录还是注册、验证码模式还是密码模式、表单数据、加载状态）全部由 AuthPage 管理，子组件只负责渲染和触发事件。

AuthPage 内部不包含任何具体的表单 UI 代码。它通过条件渲染来决定当前展示哪个子组件。顶层的「登录/注册」Tab 切换由 AuthPage 自身的 `activeTab` 响应式变量控制；内层的「验证码/密码」Method Tab 切换同样如此。

表单的验证逻辑放在 AuthPage 的 `handleSubmit` 方法中，而不是子组件中。这样做的好处是无论子组件如何切换，验证逻辑始终集中在一处，不散落。验证通过后，调用 `authStore.login()` 或 `authStore.register()` action，Store 内部的异步 action 负责调用 API、处理响应、更新 state。

### 5.2 Pinia Auth Store 设计

```typescript
// src/stores/auth.ts
import { defineStore } from 'pinia'
import { ref, computed } from 'vue'
import type { User, LoginRequest, RegisterRequest } from '@/types'
import { authApi } from '@/api/auth'

export const useAuthStore = defineStore('auth', () => {
  // ---- State ----
  const accessToken = ref<string | null>(null)
  const refreshToken = ref<string | null>(null)
  const user = ref<User | null>(null)
  const isAuthenticated = computed(() => !!accessToken.value)

  // ---- Actions ----
  async function login(payload: LoginRequest) {
    const res = await authApi.login(payload)
    accessToken.value = res.data.accessToken
    refreshToken.value = res.data.refreshToken
    user.value = res.data.user
  }

  async function register(payload: RegisterRequest) {
    const res = await authApi.register(payload)
    accessToken.value = res.data.accessToken
    refreshToken.value = res.data.refreshToken
    user.value = res.data.user
  }

  async function sendCode(email: string) {
    await authApi.sendCode(email)
  }

  async function refreshAccessToken() {
    if (!refreshToken.value) throw new Error('No refresh token')
    const res = await authApi.refresh(refreshToken.value)
    accessToken.value = res.data.accessToken
    // Refresh Token 可能也会更新（轮换策略）
    if (res.data.refreshToken) {
      refreshToken.value = res.data.refreshToken
    }
  }

  async function fetchUserProfile() {
    if (!accessToken.value) return
    const res = await authApi.getProfile()
    user.value = res.data
  }

  function logout() {
    accessToken.value = null
    refreshToken.value = null
    user.value = null
    // 调用后端登出接口（用于 token 黑名单）
    authApi.logout().catch(() => {})
  }

  return {
    accessToken,
    refreshToken,
    user,
    isAuthenticated,
    login,
    register,
    sendCode,
    refreshAccessToken,
    fetchUserProfile,
    logout
  }
}, {
  persist: {
    key: 'lingmaforge-auth',
    storage: localStorage,
    // 只持久化 token 和用户信息，不持久化临时状态
    pick: ['accessToken', 'refreshToken', 'user']
  }
})
```

这个 Store 的设计有几个值得注意的决策点。

**为什么 accessToken 和 refreshToken 分开存储？** 因为两个 token 的生命周期不同，使用场景也不同。Access Token 每次 API 请求都需要，频繁读写；Refresh Token 只在 Access Token 即将过期时才使用，读写频率很低。分开存储可以让 Axios 拦截器只关心 Access Token，不必处理 Refresh Token 的逻辑。

**为什么 logout 时调用后端接口但用 catch 吞掉错误？** `logout` 调用的后端接口用于将当前 token 加入黑名单（Redis），防止 token 在被客户端删除后仍然能在服务端被验证通过。这是一个安全增强措施。但如果这个接口调用失败（比如网络断开），不应该阻止用户的登出操作——客户端侧的 token 清除必须完成。因此 `catch(() => {})` 是为了保证"客户端登出"这个底线操作不被后端接口的异常所阻断。

**持久化策略为什么只持久化 token 和 user？** `pinia-plugin-persistedstate` 的 `pick` 选项允许我们精确指定哪些 state 字段需要持久化。Auth Store 中的其他响应式变量（比如表单验证中间状态、加载状态）是瞬时的，不应该持久化到 localStorage 中。这避免了页面刷新后出现一个永远 spinning 的登录按钮或显示过期的表单验证错误信息。

### 5.3 Axios 拦截器设计

```typescript
// src/api/request.ts
import axios from 'axios'
import { useAuthStore } from '@/stores/auth'
import router from '@/router'

const request = axios.create({
  baseURL: import.meta.env.VITE_API_BASE_URL,
  timeout: 15000
})

// 请求拦截器 —— 注入 Access Token
request.interceptors.request.use(config => {
  const authStore = useAuthStore()
  if (authStore.accessToken) {
    config.headers.Authorization = `Bearer ${authStore.accessToken}`
  }
  return config
})

// 响应拦截器 —— 401 自动刷新 & 401 跳转登录
let isRefreshing = false
let pendingRequests: Array<(token: string) => void> = []

request.interceptors.response.use(
  response => response,
  async error => {
    const { config, response } = error
    const authStore = useAuthStore()

    // 不是 401 或者 已经是刷新 token 的请求 → 直接抛出
    if (response?.status !== 401 || config.url?.includes('/auth/refresh')) {
      return Promise.reject(error)
    }

    // 正在刷新 token → 将当前请求加入等待队列
    if (isRefreshing) {
      return new Promise(resolve => {
        pendingRequests.push((newToken: string) => {
          config.headers.Authorization = `Bearer ${newToken}`
          resolve(request(config))
        })
      })
    }

    // 开始刷新
    isRefreshing = true
    try {
      await authStore.refreshAccessToken()
      const newToken = authStore.accessToken!
      // 重放所有等待中的请求
      pendingRequests.forEach(cb => cb(newToken))
      pendingRequests = []
      // 重试当前请求
      config.headers.Authorization = `Bearer ${newToken}`
      return request(config)
    } catch {
      // 刷新失败 → 登出 & 跳转登录
      pendingRequests = []
      authStore.logout()
      router.push({ path: '/auth', query: { redirect: router.currentRoute.value.fullPath } })
      return Promise.reject(error)
    } finally {
      isRefreshing = false
    }
  }
)

export default request
```

这段拦截器代码中有一个很容易被忽略但非常关键的设计——**并发请求的 token 刷新队列**。考虑这样一种场景：用户在页面上同时发起了 3 个 API 请求（比如加载项目列表、加载用户信息、加载部署历史），而此时 Access Token 恰好过期了。后端对这三个请求都返回 401，三个响应拦截器几乎同时触发。

如果没有队列机制，三个拦截器会各自独立地调用 `refreshAccessToken()`，导致后端收到三次刷新请求。这至少是浪费资源，严重的话可能因为并发竞争导致 token 轮换异常。队列机制的解决思路是：第一个收到 401 的拦截器成为"刷新发起者"，设置 `isRefreshing = true` 并开始调用刷新 API；后续收到 401 的拦截器看到 `isRefreshing = true`，就知道已经有人在刷新了，于是将自己的请求包装成一个 Promise 放入 `pendingRequests` 队列中等待。刷新成功后，"刷新发起者"遍历队列，用新的 token 重放所有等待中的请求。整个过程对外部业务代码完全透明——业务代码调用 API 时完全不需要关心 token 是否过期、是否需要刷新。

### 5.4 表单验证策略

验证码表单的验证规则：
- 邮箱：必填，格式校验正则
- 验证码：必填，6 位数字，`/^\d{6}$/`
- 密码（密码登录模式）：必填，至少 8 位，至少包含字母和数字

注册表单的额外规则：
- 确认密码：必填，与密码一致
- 如果用户没有输入用户名，后端自动生成

### 5.5 OAuth 前端对接

OAuth 按钮的点击逻辑不是直接跳转后端 OAuth 接口，而是分两步：

**第一步**，前端构造 OAuth 授权 URL。对于 GitHub OAuth，URL 格式为 `https://github.com/login/oauth/authorize?client_id=xxx&redirect_uri=xxx&scope=user:email`。这个 URL 可以在前端硬编码（不包含敏感信息），但更优雅的做法是让后端提供一个 `GET /api/auth/oauth/providers` 接口，返回所有可用的 OAuth 提供者的授权 URL。这样新增或修改 OAuth 提供者时不需要改前端代码。

**第二步**，用户在第三方平台完成授权后，浏览器被重定向回后端的回调地址（例如 `https://lingmaforge.com/api/auth/oauth/github/callback?code=xxx`）。后端用 `code` 向 GitHub 换取 Access Token，再用 Access Token 获取用户信息（邮箱、头像、用户名）。然后后端查找该邮箱是否已有账号——有则直接登录生成 token，没有则自动注册。最后后端重定向浏览器到前端页面：`/auth?token=xxx&refreshToken=xxx&user=xxx`，或者更好的方式是设置一个一次性的 `login_token` 写入 URL，前端页面加载后用这个 `login_token` 调用一次性兑换接口获取真正的 Access Token 和 Refresh Token。后者的好处是 token 不出现在浏览器地址栏中（用完即失效）。

### 6.1 Spring Security 配置

```java
// config/SecurityConfig.java
@Configuration
@EnableWebSecurity
public class SecurityConfig {

    @Bean
    public SecurityFilterChain filterChain(HttpSecurity http) throws Exception {
        http
            // 禁用 CSRF —— SPA + JWT 场景下 CSRF 不适用
            .csrf(csrf -> csrf.disable())
            // 配置跨域
            .cors(cors -> cors.configurationSource(corsConfigurationSource()))
            // 关闭 Session —— JWT 无状态
            .sessionManagement(session -> 
                session.sessionCreationPolicy(SessionCreationPolicy.STATELESS))
            // 路径权限配置
            .authorizeHttpRequests(auth -> auth
                // 放行认证相关接口
                .requestMatchers("/api/auth/**").permitAll()
                // 放行公开资源
                .requestMatchers("/api/public/**").permitAll()
                // 放行 Swagger
                .requestMatchers("/swagger-ui/**", "/v3/api-docs/**").permitAll()
                // 其余所有接口需要认证
                .anyRequest().authenticated()
            )
            // 在 UsernamePasswordAuthenticationFilter 之前插入 JWT 过滤器
            .addFilterBefore(
                jwtAuthenticationFilter(), 
                UsernamePasswordAuthenticationFilter.class
            );

        return http.build();
    }

    @Bean
    public CorsConfigurationSource corsConfigurationSource() {
        CorsConfiguration config = new CorsConfiguration();
        config.setAllowedOriginPatterns(List.of("*")); // 开发阶段，生产环境应限制
        config.setAllowedMethods(List.of("GET", "POST", "PUT", "DELETE", "OPTIONS"));
        config.setAllowedHeaders(List.of("*"));
        config.setAllowCredentials(true);
        
        UrlBasedCorsConfigurationSource source = new UrlBasedCorsConfigurationSource();
        source.registerCorsConfiguration("/**", config);
        return source;
    }

    @Bean
    public PasswordEncoder passwordEncoder() {
        return new BCryptPasswordEncoder();
    }
}
```

这段配置中有几个重要的设计决策值得展开说明。

**为什么关闭 CSRF 保护？** CSRF（跨站请求伪造）攻击的核心前提是浏览器会自动携带 Cookie 到目标域名。在传统的 Session 认证模式下，攻击者可以诱导用户点击一个恶意链接，用户的浏览器会自动带上该网站的 Session Cookie 发起请求。但 JWT 模式下，token 由前端代码主动在 `Authorization` 头中设置，浏览器不会自动附带。这意味着攻击者即使诱导用户发起跨站请求，请求中也不会包含 token，后端会以 401 拒绝。因此 CSRF 保护在 JWT 场景下是多余的。

**为什么设置 Session 为 STATELESS？** 前文已经讨论过，JWT 是无状态认证，后端不维护用户会话。Spring Security 的 `SessionCreationPolicy.STATELESS` 告诉框架"永远不要创建 HTTP Session"，这样可以避免 Spring Security 在请求处理完成后创建 `JSESSIONID` Cookie。

**JWT 过滤器的插入位置为什么是 `UsernamePasswordAuthenticationFilter` 之前？** `UsernamePasswordAuthenticationFilter` 是 Spring Security 处理表单登录的默认过滤器。JWT 过滤器需要在它之前执行，因为 JWT 过滤器会先尝试从请求头中提取 token 并完成认证。如果认证成功，`SecurityContextHolder` 中已经有了认证信息，`UsernamePasswordAuthenticationFilter` 就会跳过。

### 6.2 JWT 过滤器实现

```java
// filter/JwtAuthenticationFilter.java
@Component
public class JwtAuthenticationFilter extends OncePerRequestFilter {

    @Autowired
    private JwtUtil jwtUtil;

    @Override
    protected void doFilterInternal(
            HttpServletRequest request,
            HttpServletResponse response,
            FilterChain filterChain) throws ServletException, IOException {
        
        String authHeader = request.getHeader("Authorization");
        
        // 没有 Bearer Token → 直接放行，后续由 SecurityConfig 决定能否访问
        if (authHeader == null || !authHeader.startsWith("Bearer ")) {
            filterChain.doFilter(request, response);
            return;
        }

        String token = authHeader.substring(7);
        
        try {
            // 解析 token 获取用户 ID
            Long userId = jwtUtil.getUserIdFromToken(token);
            
            // 检查 token 是否在黑名单中（登出时加入）
            if (jwtUtil.isTokenBlacklisted(token)) {
                filterChain.doFilter(request, response);
                return;
            }

            // 构建 Authentication 对象并写入 SecurityContext
            UsernamePasswordAuthenticationToken authentication =
                new UsernamePasswordAuthenticationToken(userId, null, List.of());
            authentication.setDetails(new WebAuthenticationDetailsSource().buildDetails(request));
            
            SecurityContextHolder.getContext().setAuthentication(authentication);
            
        } catch (JwtException e) {
            // Token 无效或过期 → 清除上下文，后续过滤器会返回 401
            SecurityContextHolder.clearContext();
        }

        filterChain.doFilter(request, response);
    }
}
```

### 6.3 JWT 工具类

```java
// common/util/JwtUtil.java
@Component
public class JwtUtil {

    @Value("${jwt.secret}")
    private String secret;

    @Value("${jwt.access-token-expiration}")
    private long accessTokenExpiration; // 1800000 ms = 30 min

    @Value("${jwt.refresh-token-expiration}")
    private long refreshTokenExpiration; // 604800000 ms = 7 days

    @Autowired
    private RedisTemplate<String, String> redisTemplate;

    /**
     * 生成 Access Token
     */
    public String generateAccessToken(Long userId, String username) {
        return Jwts.builder()
                .setSubject(String.valueOf(userId))
                .claim("username", username)
                .setId(UUID.randomUUID().toString()) // jti 用于黑名单
                .setIssuedAt(new Date())
                .setExpiration(new Date(System.currentTimeMillis() + accessTokenExpiration))
                .signWith(Keys.hmacShaKeyFor(secret.getBytes(StandardCharsets.UTF_8)))
                .compact();
    }

    /**
     * 生成 Refresh Token（有效期更长）
     */
    public String generateRefreshToken(Long userId) {
        return Jwts.builder()
                .setSubject(String.valueOf(userId))
                .setId(UUID.randomUUID().toString())
                .setIssuedAt(new Date())
                .setExpiration(new Date(System.currentTimeMillis() + refreshTokenExpiration))
                .signWith(Keys.hmacShaKeyFor(secret.getBytes(StandardCharsets.UTF_8)))
                .compact();
    }

    /**
     * 从 Token 中提取用户 ID
     */
    public Long getUserIdFromToken(String token) {
        Claims claims = parseToken(token);
        return Long.valueOf(claims.getSubject());
    }

    /**
     * 验证 Token 是否有效
     */
    public boolean validateToken(String token) {
        try {
            parseToken(token);
            return !isTokenBlacklisted(token);
        } catch (JwtException e) {
            return false;
        }
    }

    /**
     * 将 Token 加入黑名单（登出时调用）
     */
    public void blacklistToken(String token) {
        Claims claims = parseToken(token);
        String jti = claims.getId();
        Date expiration = claims.getExpiration();
        long ttl = expiration.getTime() - System.currentTimeMillis();
        if (ttl > 0) {
            redisTemplate.opsForValue()
                .set("BLACKLIST:token:" + jti, "1", ttl, TimeUnit.MILLISECONDS);
        }
    }

    /**
     * 检查 Token 是否在黑名单中
     */
    public boolean isTokenBlacklisted(String token) {
        try {
            String jti = parseToken(token).getId();
            return Boolean.TRUE.equals(redisTemplate.hasKey("BLACKLIST:token:" + jti));
        } catch (JwtException e) {
            return false;
        }
    }

    private Claims parseToken(String token) {
        return Jwts.parserBuilder()
                .setSigningKey(Keys.hmacShaKeyFor(secret.getBytes(StandardCharsets.UTF_8)))
                .build()
                .parseClaimsJws(token)
                .getBody();
    }
}
```

### 6.4 认证服务实现

```java
// module/auth/service/impl/AuthServiceImpl.java
@Service
public class AuthServiceImpl implements AuthService {

    @Autowired
    private UserMapper userMapper;
    
    @Autowired
    private PasswordEncoder passwordEncoder;
    
    @Autowired
    private JwtUtil jwtUtil;
    
    @Autowired
    private RedisTemplate<String, String> redisTemplate;

    @Override
    public AuthResponse login(LoginRequest request) {
        // 根据邮箱查询用户
        User user = userMapper.selectByEmail(request.getEmail());
        if (user == null) {
            throw new BusinessException("用户不存在");
        }
        
        // 根据登录方式校验
        if ("password".equals(request.getMethod())) {
            // 密码登录
            if (!passwordEncoder.matches(request.getPassword(), user.getPasswordHash())) {
                throw new BusinessException("密码错误");
            }
        } else if ("code".equals(request.getMethod())) {
            // 验证码登录
            String cachedCode = redisTemplate.opsForValue()
                .get("code:" + request.getEmail());
            if (cachedCode == null) {
                throw new BusinessException("验证码已过期，请重新获取");
            }
            if (!cachedCode.equals(request.getCode())) {
                throw new BusinessException("验证码错误");
            }
            // 验证码一次性使用，用完删除
            redisTemplate.delete("code:" + request.getEmail());
        }

        // 生成 token
        String accessToken = jwtUtil.generateAccessToken(user.getId(), user.getUsername());
        String refreshToken = jwtUtil.generateRefreshToken(user.getId());
        
        return AuthResponse.builder()
                .accessToken(accessToken)
                .refreshToken(refreshToken)
                .user(UserDto.fromEntity(user))
                .build();
    }

    @Override
    public AuthResponse register(RegisterRequest request) {
        // 邮箱去重
        if (userMapper.selectByEmail(request.getEmail()) != null) {
            throw new BusinessException("该邮箱已被注册");
        }

        // 构建用户实体
        User user = new User();
        user.setEmail(request.getEmail());
        user.setPasswordHash(passwordEncoder.encode(request.getPassword()));
        user.setUsername(request.getUsername() != null 
            ? request.getUsername() 
            : "用户" + String.format("%06d", new Random().nextInt(1000000)));
        user.setStatus(1);

        userMapper.insert(user);

        // 分配默认 Free 订阅方案（M5+）
        // subscriptionService.assignDefaultPlan(user.getId());

        // 注册成功直接登录
        String accessToken = jwtUtil.generateAccessToken(user.getId(), user.getUsername());
        String refreshToken = jwtUtil.generateRefreshToken(user.getId());

        return AuthResponse.builder()
                .accessToken(accessToken)
                .refreshToken(refreshToken)
                .user(UserDto.fromEntity(user))
                .build();
    }

    @Override
    public void sendVerificationCode(String email) {
        // 60 秒冷却检查
        String cooldownKey = "code:cooldown:" + email;
        if (Boolean.TRUE.equals(redisTemplate.hasKey(cooldownKey))) {
            throw new BusinessException("验证码发送过于频繁，请 60 秒后再试");
        }

        // 生成 6 位验证码
        String code = String.format("%06d", new Random().nextInt(1000000));
        
        // 存入 Redis，5 分钟有效
        redisTemplate.opsForValue()
            .set("code:" + email, code, 300, TimeUnit.SECONDS);
        
        // 设置 60 秒冷却
        redisTemplate.opsForValue()
            .set(cooldownKey, "1", 60, TimeUnit.SECONDS);

        // 发送验证码（邮件 / 短信）
        // mailService.sendVerificationCode(email, code);
        log.info("[DEV] 验证码已发送至 {}: {}", email, code);
    }

    @Override
    public AuthResponse refreshToken(String refreshToken) {
        if (!jwtUtil.validateToken(refreshToken)) {
            throw new BusinessException("Refresh Token 无效或已过期");
        }
        
        Long userId = jwtUtil.getUserIdFromToken(refreshToken);
        User user = userMapper.selectById(userId);
        if (user == null) {
            throw new BusinessException("用户不存在");
        }

        // 生成新的 Access Token
        String newAccessToken = jwtUtil.generateAccessToken(user.getId(), user.getUsername());
        // 可选：也轮换 Refresh Token（增强安全性）
        String newRefreshToken = jwtUtil.generateRefreshToken(user.getId());

        // 将旧的 Refresh Token 加入黑名单（防止复用）
        jwtUtil.blacklistToken(refreshToken);

        return AuthResponse.builder()
                .accessToken(newAccessToken)
                .refreshToken(newRefreshToken)
                .build();
    }
}
```

### 6.5 Token 黑名单机制

Token 黑名单是一个容易被忽略但安全性很高的机制。JWT 的一个"缺陷"是它在过期之前始终有效——即使服务器想让某个 token 失效，也没有办法"收回"一个已经签发的 JWT。黑名单机制就是对这个问题的一种补偿。

当用户点击「退出登录」时，后端不仅要在客户端清除 token，还要将当前 Access Token 的 `jti`（JWT ID）加入 Redis 黑名单，TTL 设置为该 token 的剩余有效时间。之后所有携带该 token 的请求，JWT 过滤器会先检查 Redis 黑名单，发现该 token 已被列入就直接拒绝。

Refresh Token 在黑名单中的 TTL 更长（因为是 7 天有效期的 token），但逻辑相同。当用户调用刷新接口获取新的 Access Token 时，旧的 Refresh Token 也被加入黑名单——这被称为 **Refresh Token 轮换（Rotation）**，可以防止 Refresh Token 被泄露后被反复使用。

## 八、Token 管理策略

### 8.1 Token 存储位置选择

这是一个在认证方案设计中经常会引发争论的问题：token 应该存储在哪里？

| 方案 | 安全性 | XSS 风险 | CSRF 风险 | 跨域 | 便捷性 |
|------|--------|---------|----------|------|--------|
| localStorage | 中 | 高（JS 可读） | 无 | 好 | 高 |
| Cookie (httpOnly) | 高 | 无（JS 不可读） | 有（需配合 SameSite） | 需要配置 | 中 |
| 内存 (Pinia State) | 高 | 无 | 无 | 好 | 低（刷新丢失） |
| sessionStorage | 中 | 高（JS 可读） | 无 | 好 | 低（标签页关闭丢失） |

对于灵码工坊，我推荐 **localStorage + 内存（Pinia）双层存储**，而不是 Cookie。

理由有四点。第一，灵码工坊的 token 不涉及资金交易或极高敏感度的个人隐私数据，localStorage 的 XSS 风险可以通过 CSP（Content Security Policy）和严格的输入输出过滤来有效控制。第二，Cookie 的跨域管理在前后端分离部署时是一个持续的痛点——前端的 `fetch` 需要设置 `credentials: 'include'`，后端的 CORS 需要配置 `allowCredentials(true)` 且不能使用 `*` 通配符，Cookie 的 `domain`、`path`、`SameSite`、`Secure` 属性在不同部署环境下的表现差异很大。对于灵码工坊这种可能部署在多个环境（开发、测试、生产、CloudStudio）的项目，Cookie 方案的调试成本远高于 localStorage。第三，Refresh Token 存储在 localStorage 中配合 Pinia 持久化插件，页面刷新后可以无缝恢复到 Pinia Store，用户体验不受影响。第四，如果未来确实需要增强安全性，可以很容易地从 localStorage 迁移到 httpOnly Cookie + CSRF Token 的方案——因为 token 的读写逻辑集中在 Pinia Store 和 Axios 拦截器中，修改两个文件即可，不涉及业务代码的重构。

### 8.2 Token 刷新时机

Token 的刷新时机不应该只在收到 401 时才触发——这样用户体验会产生一个微小的延迟：用户发起请求 → 后端返回 401 → 拦截器刷新 token → 重试请求。这个延迟虽然只有几百毫秒，但如果发生在页面加载时（多个并发请求同时触发），可能导致页面出现闪烁或短暂的白屏。

更好的策略是**主动刷新 + 被动兜底**。前端在每次发起请求前，检查 Access Token 的过期时间（通过解析 JWT payload 中的 `exp` 字段）。如果距离过期时间小于 5 分钟，且当前没有正在进行的刷新操作，则主动发起刷新请求，拿到新 token 后再发起原请求。如果主动刷新失败（网络问题或 refresh token 也过期了），则降级到被动兜底——等原请求返回 401 时走拦截器的刷新队列逻辑。

在实际代码中，这个逻辑可以放在 Axios 的请求拦截器中，与 token 注入逻辑放在一起。需要注意的是，JWT 解析操作（`JSON.parse(atob(token.split('.')[1]))`）非常轻量，不会对性能产生可感知的影响。

## 九、路由守卫设计

### 9.1 路由配置

```typescript
// src/router/index.ts
import { createRouter, createWebHistory } from 'vue-router'
import { useAuthStore } from '@/stores/auth'

const routes = [
  {
    path: '/',
    name: 'home',
    component: () => import('@/views/HomePage.vue'),
    meta: { requiresAuth: false }
  },
  {
    path: '/auth',
    name: 'auth',
    component: () => import('@/views/AuthPage.vue'),
    meta: { requiresAuth: false, guestOnly: true }  // 已登录用户不能访问
  },
  {
    path: '/workbench',
    name: 'workbench',
    component: () => import('@/views/workbench/WorkbenchPage.vue'),
    meta: { requiresAuth: true }
  },
  {
    path: '/creative',
    name: 'creative',
    component: () => import('@/views/CreativePage.vue'),
    meta: { requiresAuth: false }
  },
  // ... 其他路由
]

const router = createRouter({
  history: createWebHistory(),
  routes
})

// 全局前置守卫
router.beforeEach((to, from, next) => {
  const authStore = useAuthStore()

  // 需要认证的页面
  if (to.meta.requiresAuth) {
    if (!authStore.isAuthenticated) {
      // 未登录 → 重定向到登录页，携带当前路径作为 redirect 参数
      next({ path: '/auth', query: { redirect: to.fullPath } })
    } else {
      next()
    }
  }
  // 仅游客可访问的页面（登录页）
  else if (to.meta.guestOnly) {
    if (authStore.isAuthenticated) {
      // 已登录 → 重定向到工作台
      next('/workbench')
    } else {
      next()
    }
  }
  // 公开页面
  else {
    next()
  }
})

export default router
```

路由守卫的逻辑分为三层，每一层的判定都清晰且没有歧义。

**第一层**——目标路由需要认证（`requiresAuth: true`）。检查 Auth Store 中的 `isAuthenticated` 状态。注意这里使用的是 computed 属性而不是直接检查 `accessToken` 是否存在——因为 token 存在但已经过期的边界情况需要由 Store 内部的逻辑来处理，路由守卫不应该做 token 解析的重复逻辑。未登录时，跳转到 `/auth?redirect=原路径`，注意这里的参数名是 `redirect` 而不是 `from` 或 `back`，因为在登录页面需要读取这个参数来构造登录后的跳转目标。

**第二层**——目标路由是游客独享的（`guestOnly: true`）。目前只有 `/auth` 页面属于这一类。已登录用户不应该看到登录页面，所以直接重定向到 `/workbench`。这里有一个微妙的场景——如果用户在 `/auth?redirect=/creative` 这个 URL 上下已经登录，路由守卫会直接跳到 `/workbench` 而不是 `/creative`。这是符合逻辑的——如果用户已经登录了，就没有必要经过登录页的跳转。

**第三层**——公开页面。直接放行，不做任何检查。

### 9.2 登录后的跳转目标还原

跳转还原的优先级是：`redirect` 参数指定的路径 > 默认页面（工作台）。

具体来说，当用户在未登录状态访问 `/works` 并被重定向到 `/auth?redirect=%2Fworks`，登录成功后，前端代码会解析 `redirect` 参数，提取出 `/works`，然后调用 `router.push('/works')`。这里有一个需要处理的情况——如果 `redirect` 参数中包含了其他 query 参数（比如 `/works?sort=recent`），需要进行 URL 编码解码的完整处理。

如果 `redirect` 参数不存在（用户直接访问 `/auth` 进行登录），则跳转到 `/workbench`——工作台是灵码工坊的默认落地页。

## 十、安全策略与防御方案

### 10.1 密码安全

密码存储使用 BCrypt 哈希算法。BCrypt 在 Spring Security 中通过 `BCryptPasswordEncoder` 使用，默认的 `strength` 参数（即 log rounds）为 10，这意味着哈希计算需要 2^10 = 1024 轮迭代。这个参数在安全性和性能之间是一个合理的平衡点——单次哈希在普通服务器上耗时约 100-200ms，对登录体验的影响可忽略不计，但对暴力破解来说，1024 轮迭代意味着攻击者试错的时间成本被放大了 1000 倍。

BCrypt 的另一个优势是它自动处理了盐值（Salt）。每次调用 `encode()` 时，BCrypt 会生成一个 16 字节的随机盐值，将其嵌入到 60 字符的哈希结果中。这意味着两个用户即使设置了完全相同的密码，他们在数据库中的 `password_hash` 字段值也是不同的。这消除了彩虹表攻击的可能性。

### 10.2 验证码安全

验证码相关的安全策略是认证系统中最容易被低估的部分。一个不安全的验证码实现可能导致批量注册、短信轰炸、甚至是账号盗用。灵码工坊的验证码安全策略分为三个层面。

**第一层：速率限制。** 同一个邮箱在 60 秒内只能发送一次验证码。这个限制通过 Redis 中的 `code:cooldown:{email}` key 来实现，TTL 为 60 秒。在冷却期内重复请求，后端直接返回错误响应，不执行任何发送逻辑。这个策略的主要目的是防止恶意用户用脚本对一个邮箱进行验证码轰炸。

**第二层：IP 级别限制。** 除了邮箱维度的限制，还应该加入 IP 维度的限制。同一个 IP 在 1 小时内最多发送 10 次验证码，超过这个上限后该 IP 被临时封禁 1 小时。这个策略的目的是防止攻击者用不同的邮箱进行批量注册或批量请求。实现方案是在 Redis 中维护一个 `code:ip:{ip}` 的计数器，每次发送验证码时 `INCR`，TTL 设置为 1 小时。当计数器超过 10 时，拒绝请求。

**第三层：验证次数限制。** 同一个验证码最多只能被尝试验证 5 次。超过 5 次后，该验证码被作废（Redis 中删除），用户需要重新获取。这个策略的目的是防止攻击者对验证码进行暴力枚举——6 位数字验证码的穷举空间是 100 万，不加限制的情况下脚本可以在几秒钟内试完。限制 5 次后，攻击者的命中概率从 100% 降到了 0.0005%，基本杜绝了暴力枚举的可能性。这个策略不单独使用 Redis key，而是在验证码的 Redis value 中附带一个尝试计数器，或者使用一个独立的 `code:attempts:{email}` key。

### 10.3 Token 安全

Token 的安全策略围绕"缩短泄露后的危害窗口"这个核心目标展开。

Access Token 的有效期设置为 30 分钟，足够短以至于即使 token 被泄露，攻击者的操作窗口也只有 30 分钟。30 分钟对于正常用户来说完全够用——用户不会感知到 token 的刷新过程。

Refresh Token 的有效期为 7 天，配合轮换机制。每次用 Refresh Token 换取新的 Access Token 时，旧的 Refresh Token 被加入黑名单，新的 Refresh Token 被签发。这意味着如果一个 Refresh Token 被泄露，合法的用户和攻击者之间会形成一种"竞争"关系——谁先用这个 Refresh Token 刷新，另一个人的 Refresh Token 就失效了。合法用户在下一次刷新时发现 Refresh Token 失效（因为攻击者已经抢先刷新了），被迫重新登录，这就给用户发出了一个"你的账号可能被盗用"的无声信号。

JWT 的签名密钥（Secret）绝对不能硬编码在前端代码或公开仓库中。开发环境的密钥可以存储在 `application-dev.yml` 中并通过 Git 忽略文件排除；生产环境的密钥必须通过环境变量（`JWT_SECRET`）注入，或在 Kubernetes 中使用 Secret 资源管理。

### 10.4 前端安全

前端的认证安全主要面临 XSS（跨站脚本攻击）的威胁。如果攻击者通过 XSS 漏洞在页面中注入了恶意脚本，该脚本可以读取 `localStorage` 中的所有数据，包括 token 和用户信息。针对这个威胁的防御策略是 CSP（Content Security Policy）。

CSP 是一组 HTTP 响应头，告诉浏览器哪些来源的脚本可以执行。一个严格的 CSP 配置可以阻止攻击者的内联脚本执行，从而切断 XSS 攻击的执行路径。在前端项目的 `index.html` 中配置：

```html
<meta http-equiv="Content-Security-Policy" 
      content="default-src 'self'; 
               script-src 'self'; 
               style-src 'self' 'unsafe-inline'; 
               img-src 'self' https: data:;
               connect-src 'self' https://api.lingmaforge.com;">
```

`script-src 'self'` 表示只允许同源的脚本执行，任何内联脚本（`<script>` 标签直接写在 HTML 中或 `onclick="..."` 属性）都会被浏览器阻止。Monaco Editor 和 Vue 的动态组件都通过构建工具编译为独立的 JS 文件，不依赖内联脚本，所以这个限制不会影响正常功能。`unsafe-inline` 只放行给了 `style-src`，因为 Vue 的 `<style scoped>` 和部分 CSS-in-JS 方案依赖于内联样式。

## 十二、OAuth 第三方登录方案

### 12.1 为什么需要 OAuth

OAuth 登录的核心价值在于**把"注册"这个产品体验中最大的流失点变成一个按钮**。传统邮箱注册需要用户打开邮箱、找到验证码、回到页面输入、设置密码、确认密码——这个流程每一步都在流失用户。一个 GitHub OAuth 按钮可以把这个流程缩短到"点击 → 授权 → 完成"，转化率提升是数量级级别的。

对于灵码工坊这个面向开发者和设计师的产品，GitHub OAuth 还有一个额外的价值——它天然携带了用户在 GitHub 上的身份信息（头像、用户名），这些信息可以直接同步到灵码工坊的个人资料中，省去了用户手动设置头像和昵称的步骤。这个细节让用户觉得"这个产品懂我"。

### 12.2 GitHub OAuth 实现

GitHub OAuth 是基于标准的 Authorization Code Grant 流程。核心交互有三次 HTTP 往返：

**第一次往返 —— 用户点击「GitHub 登录」按钮。** 前端不直接跳转到 GitHub 授权页，而是先跳转到后端的 `/api/auth/oauth/github` 接口。后端构造 GitHub 的授权 URL：

```
https://github.com/login/oauth/authorize
  ?client_id={GITHUB_CLIENT_ID}
  &redirect_uri=https://api.lingmaforge.com/api/auth/oauth/github/callback
  &scope=user:email
  &state={RANDOM_STATE}
```

`state` 参数是一个随机生成的字符串，存储在 Redis 中（TTL 10 分钟），用于防止 CSRF 攻击。后端将用户重定向到这个 URL，浏览器跳转到 GitHub 的授权页面。

**第二次往返 —— 用户在 GitHub 上授权后，GitHub 将浏览器重定向回回调地址。** 回调 URL 上携带了 `code` 和 `state` 两个参数。后端首先校验 `state` 参数是否与 Redis 中存储的一致（不一致则说明是 CSRF 攻击，拒绝后续流程）。校验通过后，后端用 `code` 向 GitHub 的 Token 端点发起 POST 请求：

```
POST https://github.com/login/oauth/access_token
Body: { client_id, client_secret, code }
```

GitHub 返回 Access Token。后端用这个 Access Token 调用 GitHub 的 User API：

```
GET https://api.github.com/user
Authorization: Bearer {github_access_token}
```

获取到用户的 GitHub ID、用户名、头像 URL。然后调用 Email API 获取用户的主邮箱。至此，后端拿到了创建或绑定本地账号所需的所有信息。

**第三次往返 —— 后端将浏览器重定向回前端。** 这次重定向的 URL 携带一个一次性的 `login_token`：`https://lingmaforge.com/auth?login_token={ONE_TIME_TOKEN}`。前端 AuthPage 在 `onMounted` 时检测到 `login_token` 参数存在，调用 `POST /api/auth/exchange-token` 接口，用 `login_token` 换取正式的 Access Token 和 Refresh Token。`login_token` 在 Redis 中的 TTL 设置为 60 秒，且只能使用一次，使用后立即删除。这种"一次性兑换 token"的模式确保了即使浏览器地址栏中的 `login_token` 被截图或分享出去，攻击者也无法利用它（因为它已经失效了）。

### 12.3 微信 OAuth 的特殊处理

微信 OAuth 的实现与 GitHub 稍有不同，主要体现在以下几个方面。

微信使用的是 OpenID 机制而不是通用的 OAuth 2.0。微信授权后返回的是一个 `code`，但换取的不是通用的用户信息，而是该用户在"当前应用"下的唯一标识 `openid`。同一个微信用户在不同的微信应用中拥有不同的 `openid`，这意味着灵码工坊不能通过 `openid` 来跨应用识别用户。

微信的授权流程需要配置"回调域名"——只有在微信开放平台后台登记过的域名才能接收微信的回调。对于灵码工坊的开发环境（`localhost`），通常使用内网穿透工具（如 ngrok）暴露一个临时公网域名进行调试。

微信获取用户信息的 API 不是 `/user`，而是 `/sns/userinfo`。获取 `openid` 之后，用 `openid` 和 `access_token` 调用 `GET https://api.weixin.qq.com/sns/userinfo` 获取用户的昵称、头像、UnionID（如果绑定了开放平台）。

## 十三、异常场景与容错处理

### 13.1 网络断连时的登录

用户在弱网环境下点击登录，请求可能长时间处于 pending 状态。前端需要设置合理的超时时间并给出正确的反馈。

Axios 的 `timeout` 配置设置为 15 秒——这是一个合理的时间，大多数正常的 HTTP 请求在 2-3 秒内就能完成，15 秒足够覆盖网络波动的窗口。超时后 Axios 抛出 `ECONNABORTED` 错误，前端捕获后展示"网络连接超时，请检查网络后重试"的 Toast 提示。按钮从 loading 状态恢复到可点击状态，用户可以再次尝试。

对于验证码发送，超时时间可以设置得更短——10 秒。因为验证码的发送只是一个"触发"，真正的验证码发送是由后端的邮件服务异步完成的。前端不需要等待邮件真正发送出去，只需要确认请求已经被后端接收。

### 13.2 并发登录冲突

同一个账号在两个不同的浏览器或设备上同时登录是一个需要处理的场景。JWT 无状态认证默认不会检测并发登录——每个设备都持有独立的 token，互不干扰。

如果产品需要"后登录者踢掉先登录者"的策略（即一个账号同时只能在一个设备上登录），可以通过以下方式实现：每次用户登录时，生成一个新的 UUID 作为 `session_id`，存储到 Redis 的 `session:{userId}` key 中，同时在 JWT 的 payload 中携带这个 `session_id`。JWT 过滤器每次解析 token 时，对比 token 中的 `session_id` 和 Redis 中存储的是否一致。如果用户在新设备上登录，Redis 中的 `session_id` 被覆盖为新的值，旧设备的 token 中的 `session_id` 与 Redis 不一致，请求被拒绝，旧设备被"踢下线"。

对于灵码工坊来说，这个特性在 M5 阶段不是必须的。建议先实现基础的多设备共存方案（允许同时在多个设备登录），后续根据用户反馈决定是否需要"踢下线"策略。

### 13.3 密码重置流程

忘记密码是一个高频的异常场景。灵码工坊的密码重置流程分为三步：

**第一步：用户点击「忘记密码」链接。** 浏览器跳转到一个独立的密码重置页面 `/auth/reset-password`，或者在当前登录卡片中通过状态切换展示密码重置表单（类似于登录/注册 Tab 的切换）。用户输入注册邮箱，点击发送验证码。

**第二步：验证身份。** 后端发送验证码到用户邮箱，用户输入验证码验证身份。验证通过后，不是直接生成 token 登录，而是进入第三步。这里特别注意——不能在验证码验证通过后直接登录，因为密码重置的场景可能是"我在别人电脑上登录，发现密码忘了"，如果直接登录就跳过了密码设置步骤。

**第三步：设置新密码。** 用户输入新密码（两遍确认），提交到 `POST /api/auth/reset-password` 接口。接口参数包括邮箱、验证码、新密码。后端校验验证码后，更新该用户的 `password_hash` 字段，删除验证码，返回成功响应。前端提示"密码已重置，请使用新密码登录"，并跳转到登录页面。

### 13.4 Redis 不可用时的降级

Redis 在灵码工坊的认证系统中承担了验证码存储、token 黑名单、冷却计时、IP 限流等多个职责。如果 Redis 宕机或不可达，这些功能都不可用。

在这个场景下需要做的不是让整个认证系统瘫痪，而是**优雅降级**。降级策略是：验证码功能暂时不可用（返回 503 提示"验证码服务暂时不可用，请使用密码登录"），密码登录不受影响（密码登录不依赖于 Redis），token 黑名单功能暂时关闭（意味着登出后的 token 在过期前仍然有效——这是一个可以接受的安全妥协），IP 限流功能暂时关闭。

这个降级策略的关键在于，即使 Redis 不可用，认证核心（密码登录 + JWT 签发验证）仍然可以正常工作。前端在收到验证码发送失败的响应后，自动将 Method Tab 切换到"密码登录"，并提示用户当前验证码服务不可用。

```
   │ ←─── {accessToken: "new", refreshToken: "new"} ───│                                │
   │                                │                                │
   │ 更新 Store + localStorage       │                                │
   │ 用新 accessToken 重试原请求      │                                │
```
