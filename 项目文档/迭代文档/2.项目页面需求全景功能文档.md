## 三、模块功能详述

---

### 模块一：用户与认证系统

**业务定位**：用户与认证系统是平台的**入口级基础设施**，承担用户身份注册、登录鉴权、个人画像维护、安全防护与通知触达等职责。该模块不直接产生业务价值，但为所有功能模块提供统一的用户标识与会话管理能力，是平台运转的前提条件。

**用户角色**：游客（未登录用户）、已注册用户、管理员。

---

#### 1.1 账号注册

- **功能名称**：多渠道用户注册
- **入口路径**：`/auth` 页面 → 注册 Tab

**核心能力描述**：
用户可通过邮箱地址或手机号码创建平台账号。注册流程采用分步式交互：第一步，用户输入邮箱地址或手机号码，系统校验格式合法性（邮箱需匹配 RFC 5322 规范，手机号需匹配中国大陆 11 位或国际区号格式）后，向该渠道发送 6 位数字验证码，同时启动 60 秒倒计时防止重复发送。第二步，用户填写收到的验证码、设置登录密码（需满足不少于 8 位、含大小写字母与数字的复杂度要求），并可选择性填写昵称。提交后，后端完成验证码校验、密码盐值加密（BCrypt）及用户记录写入，同时在用户表中初始化默认套餐等级（免费版）、生成唯一用户 ID，并返回 JWT Token 对（Access Token 有效期 2 小时，Refresh Token 有效期 7 天），前端收到后跳转至工作台页面，完成"注册即登录"的无缝体验。异常场景处理包括：验证码过期或错误时给出明确提示并允许重新获取；邮箱/手机号已被注册时提示用户直接登录或找回密码；网络超时 30 秒未响应时前端展示"请求超时，请重试"并提供手动重试按钮。

**业务价值**：作为用户增长的首要入口，降低注册摩擦直接提升新用户转化率。邮箱注册覆盖专业开发者群体，手机号注册覆盖国内移动端用户，双通道并行保障覆盖面。

**依赖与关联**：
- 依赖短信网关（阿里云 SMS / 腾讯云 SMS）或邮件服务（JavaMailSender）发送验证码
- 依赖 Redis 存储验证码（TTL 5 分钟）实现无状态验证
- 注册成功后自动触发 1.3 个人中心初始化与 5.3 订阅管理初始化（免费套餐）
- 为 1.2 OAuth 登录提供账号绑定基础

**非功能需求**：
- 验证码发送成功率 ≥ 99%，发送延迟 P99 < 3s
- 密码存储必须使用 BCrypt 或 Argon2 算法，不可明文或使用弱哈希
- 同一 IP 每小时注册次数上限 5 次，防止批量注册
- 注册接口响应时间 P99 < 500ms

---

#### 1.2 账号登录

- **功能名称**：多模式登录系统
- **入口路径**：`/auth` 页面 → 登录 Tab

**核心能力描述**：
平台提供三种登录方式以满足不同场景下的用户需求。**密码登录**：用户输入已注册的邮箱或手机号，配合密码完成验证，后端校验密码哈希后签发 JWT Token 对。连续 5 次密码错误时，账户临时锁定 15 分钟，并向绑定邮箱/手机发送安全提醒。**验证码快捷登录**：用户输入已注册手机号或邮箱，获取 6 位验证码后一键登录，无需输入密码，降低重复访问的登录摩擦。验证码有效期 5 分钟，每个号码每日最多获取 10 次。**OAuth 第三方登录**：支持 GitHub、微信开放平台、Google 三种 OAuth 2.0 授权码流程。首次 OAuth 登录时，系统自动创建绑定账号（以 OAuth Provider 返回的邮箱为唯一标识），若该邮箱已注册则直接绑定已有账号；非首次登录直接完成身份认证。OAuth 登录成功后同样返回 JWT Token 对，前端表现与密码登录完全一致。

**忘记密码**：用户在登录 Tab 底部点击"忘记密码"链接后进入密码重置流程。该流程通过独立页面或全屏弹窗承载，分为三步：第一步，用户输入已注册的邮箱地址或手机号码，系统校验该账号存在且未被封禁后，向该渠道发送 6 位验证码，前端启动 60 秒倒计时。第二步，用户填写收到的验证码，系统校验通过后进入密码重置页。第三步，用户设置新密码（需满足不少于 8 位、含大小写字母与数字的复杂度要求，且不能与最近 3 次使用过的密码相同），两次输入一致后提交。后端完成密码更新后，立即将该账号所有已登录设备的 Refresh Token 全部失效（全局强制登出），并向绑定邮箱和手机发送"密码已重置"的安全通知。前端在提交成功后引导用户使用新密码重新登录。异常场景包括：验证码错误或过期提示重新获取；该邮箱/手机号未注册时提示"该账号不存在，请先注册"并引导至注册 Tab；24 小时内同一账号最多发起 3 次密码重置请求，超出后锁定密码重置功能至次日。

**业务价值**：密码登录保障传统用户习惯，验证码登录降低移动端和重复访问的摩擦，OAuth 登录面向开发者（GitHub）和国内外用户（微信/Google）降低注册决策成本，忘记密码流程确保用户在任何情况下都不会因凭证丢失而无法访问账号。四种方式组合覆盖了用户从首次接触到长期使用中可能遇到的所有认证场景。

**依赖与关联**：
- 依赖 1.1 注册完成的用户账号数据
- 依赖 JWT Token 签发与验证基础设施
- 依赖 Redis 存储密码重置验证码（TTL 5 分钟）
- 前端 Auth Store 统一管理登录状态，路由守卫拦截未登录访问
- 登录成功后刷新 1.3 个人中心数据与 4.1 项目列表
- 密码重置成功后触发 1.4 设备管理的全局登出逻辑与 1.5 安全通知推送
- 账号安全事件（异常登录）触发 1.5 安全通知推送

**非功能需求**：
- 密码登录接口响应时间 P99 < 500ms
- OAuth 完整授权流程（含第三方回调）总耗时 < 5s
- 忘记密码全流程完成时间 < 90s
- JWT Token 签名使用 RS256 非对称算法，公私钥分离管理
- Refresh Token 存储于 HttpOnly Cookie 中，不暴露给 JavaScript
- 密码重置 Token 一次性使用，使用后立即失效
- 同一账号同时在线的设备数可通过 1.4 设备管理查询与控制

#### 1.3 个人中心

- **功能名称**：用户画像管理与账户概览
- **入口路径**：`/profile` → 侧边菜单导航

**核心能力描述**：
个人中心是用户管理自身身份信息与查看平台使用状态的核心页面，采用左侧导航菜单 + 右侧内容区的经典布局，菜单项包括：个人资料、账户概览、创作数据、账号安全、通知设置五个模块，点击或滚动时自动高亮当前区域。**个人资料编辑**：用户可修改昵称、头像（支持上传裁剪，限制 2MB / 512×512px）、个人简介（最长 200 字符）、职业角色（从预设下拉列表选择，如前端开发/后端开发/全栈/设计师/产品经理/学生/其他）及所在地区。保存时后端校验敏感词并更新用户表，头像上传后生成 CDN 缩略图 URL。**账户概览**：展示当前订阅套餐名称与续费倒计时、默认支付方式（如已绑定 VISA/微信/支付宝）、月度 AI 生成次数与 Token 消耗进度条、部署时长余额与团队席位使用数，点击各指标可跳转至 5.3 订阅管理查看详情。页面初始数据通过 `GET /user/profile` 一次性获取，编辑操作通过 `PUT /user/profile` 提交增量变更。

**业务价值**：个人中心是用户留存和增购转化的关键触点——账户概览中的用量进度条持续提醒用户"还有多少额度"，形成自然升级动力；创作数据图表让用户感知到平台为其带来的产出价值，增强黏性。

**依赖与关联**：
- 依赖 1.1/1.2 提供的用户身份标识
- 依赖 5.3 订阅系统提供的套餐与用量数据
- 依赖 4.1 项目列表提供的作品统计数据
- 头像上传依赖文件存储服务（OSS / 本地存储）
- 修改的昵称和头像需同步更新至 3.3 创意详情页的创作者信息与 4.1 项目卡片

**非功能需求**：
- 头像上传接口响应时间 P99 < 3s（含 CDN 分发）
- 个人资料编辑支持表单自动保存草稿（localStorage 备份），防止误关闭丢失
- 敏感操作（修改绑定手机/邮箱）需二次身份验证

#### 1.4 账号安全

- **功能名称**：多层级安全防护中心
- **入口路径**：`/profile` → 账号安全

**核心能力描述**：
账号安全模块提供五项安全能力，用户可在统一界面中集中管理。**密码管理**：用户输入旧密码与新密码完成密码修改，新密码需满足复杂度要求且不能与旧密码相同；提供"忘记密码"入口，通过已绑定邮箱或手机验证码完成密码重置，重置成功后强制登出所有已有设备。**双重验证（2FA）**：支持 TOTP 标准（兼容 Google Authenticator / Microsoft Authenticator）和短信验证码两种二次验证方式。用户启用 TOTP 时，后端生成 Base32 密钥并返回二维码（通过 ZXing 生成），用户扫码后在 30 秒窗口内输入动态码完成绑定；每次登录后若检测到新设备或异地 IP，要求输入 6 位 TOTP 动态码方可完成认证。**登录设备管理**：展示当前账号已登录的所有设备列表，包括设备名称、操作系统、浏览器、IP 地址、登录时间与最近活动时间。用户可以一键登出任意设备（除了当前设备），登出操作将使该设备上的 Refresh Token 立即失效。**已连接第三方账号**：展示当前已绑定的 OAuth 平台（GitHub / 微信 / Google），支持绑定新的第三方账号或解绑已有绑定（解绑前需确认当前账号已绑定至少一种其他登录方式，以防止锁死）。**账号注销**：用户可在账号安全页最底部点击"注销账号"按钮发起注销流程。为防止误操作与恶意注销，系统设置了严格的前置校验：用户必须确认当前无未结清的付费订阅（如有，需先取消订阅并等待当前计费周期结束）、无进行中的 AI 生成任务、所有项目已手动确认删除或放弃。校验通过后，系统要求用户输入当前密码或 6 位 2FA 动态码进行二次身份验证，并弹出二次确认对话框（明确告知注销后数据将进入 30 天冷静期，超期后不可恢复）。用户确认后，账号状态标记为"已申请注销"，所有活跃 Token 立即失效、所有沙箱实例停止、所有已发布创意自动下架。30 天冷静期内用户可随时通过重新登录撤销注销申请（账号恢复至正常状态）；冷静期结束后系统自动执行数据清除——删除个人信息（昵称、头像、邮箱、手机号脱敏化处理）、删除所有项目文件、匿名化创意中的创作者标识（创意作品本身作为社区资产予以保留但标记为"创作者已注销"）。注销完成后系统向用户邮箱发送最终确认通知。

**业务价值**：账号安全是用户信任的基石。2FA 有效防止撞库与凭证泄露导致的账号被盗，设备管理让用户对自己账号的访问范围完全可控，第三方绑定管理提供"多把钥匙开一扇门"的灵活登录体验，账号注销机制遵循"隐私设计"原则——赋予用户完全的数据自主权，同时通过 30 天冷静期和阶梯化数据清除策略在用户权益与平台合规之间取得平衡。

**依赖与关联**：
- 依赖 JWT Token 管理与 Session 存储（或 Redis）
- 依赖短信/邮件服务用于安全通知
- 2FA 绑定/验证状态影响 1.2 登录流程（登录后二次校验）
- 设备登出操作触发 1.5 安全类通知推送
- 密码重置成功后需通知所有已登录客户端刷新 Token 状态
- 账号注销前需校验 5.3 订阅状态（无未结清账单），注销后触发 3.5 已发布创意的自动下架、2.6 沙箱实例停止、4.1 项目列表的批量清理
- 注销撤销机制依赖于账号状态机与 30 天冷静期的定时任务

**非功能需求**：

### 模块二：AI 工作台

**业务定位**：AI 工作台是灵码工坊的**核心价值交付模块**。它将"用户输入自然语言需求 → AI 自动生成完整前端应用"的完整流程封装为可视化的 6 阶段流水线，并围绕代码生成构建了编辑、预览、迭代的一站式在线 IDE 环境。该模块直接体现平台的核心竞争力。

**用户角色**：登录用户（生成需消耗配额）、管理员（可为用户发放额外配额）。

---

#### 2.1 简洁模式（入口）

- **功能名称**：自然语言驱动的工作台入口
- **入口路径**：`/workbench`（默认展示简洁模式）

**核心能力描述**：
简洁模式是用户接触 AI 能力的起点，设计理念为"一句话描述需求，一键开启生成"。页面布局采用居中单栏设计，核心区域为一个大型自然语言输入框（支持多行文本，最长 2000 字符），placeholder 提供引导性示例（如"帮我生成一个带用户登录和数据看板的 SaaS 管理后台"）。输入框上方展示灵码工坊吉祥物形象，下方提供四类快捷模板入口：**官网模板**（企业品牌官网/产品落地页）、**后台模板**（SaaS 管理后台/数据看板）、**电商模板**（商品展示/购物车/结算流程）、**看板模板**（数据分析看板/实时监控面板）。点击任意模板卡片自动将预设提示词填入输入框，用户可在此基础上修改或直接提交。输入框右侧提供**模型选择器**，支持 Pro（最强推理，适合复杂企业级应用）、Standard（均衡性能，适合常规项目）、Fast（极速响应，适合简单原型）三档模型切换，选择结果影响后端路由的 LLM 配置。底部工具栏提供需求优化（点击后由 AI 将简短描述扩展为结构化需求）、项目导入（上传已有项目配置继续编辑）等辅助功能。"生成"按钮点击后，前端校验输入内容非空与长度限制，然后调用 `POST /generation/create` 获取任务 ID，并自动切换至生成模式。

**业务价值**：简洁模式通过模板引导和模型选择将 AI 生成的使用门槛降至最低——用户不需要理解任何技术术语即可开始创作，显著降低首次使用流失率。模板系统同时为平台运营提供了"主推方向"的控制抓手。

**依赖与关联**：
- 依赖模型选择状态，将其作为参数传递给 2.2 生成模式的流水线
- 依赖 Pinia Workbench Store 管理当前模式状态与提交动作
- 提交触发生成任务的创建 API，返回的 taskId 用于 2.2 的 SSE 连接
- 模板数据需通过 `GET /templates` 获取（后端由运营通过 7.3 管理端配置）

**非功能需求**：
- 输入框支持 Markdown 基本语法高亮（如代码块、列表）
- 模板加载时间 < 200ms
- 模型切换时需展示各模型的 Token 消耗倍率与预计耗时，辅助用户决策
- 页面首次加载至可交互时间（FCP）< 1.5s

#### 2.2 生成模式（6 阶段流水线）

- **功能名称**：可视化 AI 代码生成流水线
- **入口路径**：`/workbench`（从简洁模式提交后自动进入）

**核心能力描述**：
生成模式是平台最核心的交互界面，采用 IDE 风格的四栏布局——左侧项目文件树、中部 AI 对话面板、右侧代码/预览面板、底部诊断面板。当用户从简洁模式提交需求后，系统立即启动 6 阶段智能流水线，每个阶段通过 SSE（Server-Sent Events）实时向客户端推送进度、日志与产物：**第一阶段"需求分析"**——Agent 解析用户自然语言输入，提取应用名称、页面列表、API 接口定义、功能特性与样式偏好，输出结构化的 `RequirementSpec`，前端同步渲染为"需求确认清单"。**第二阶段"执行规划"**——Agent 根据需求规格生成文件生成计划 `PlanResult`，包括待创建文件列表（含类型、用途、依赖关系）、生成顺序（按依赖拓扑排序）、构建命令，前端渲染为"文件生成计划表"。**第三阶段"代码生成"**——Agent 按计划顺序逐文件生成代码，每生成一个文件即通过 SSE 推送文件路径与内容，前端实时更新文件树节点状态（pending → generating → generated）并在预览面板中展示代码。**第四阶段"样式优化"**——Agent 扫描生成的 CSS 文件，优化样式一致性、响应式适配与设计令牌使用，生成差异补丁。**第五阶段"构建验证"**——后端在沙箱环境中执行 `npm install && npm run build`，构建日志通过 SSE 逐行推送到前端日志面板，任何编译错误会触发智能修复（按配置的最大重试次数）或回退至代码生成阶段。**第六阶段"预览部署"**——构建成功后启动 Dev Server 或部署静态产物，返回预览 URL，前端 iframe 加载该 URL 展示运行中的应用。流水线中的任何阶段若发生不可恢复的错误，系统会向用户展示明确的错误信息与建议操作。

**业务价值**：6 阶段流水线将"AI 黑盒生成"转变为"透明可控的生产流程"，用户能实时观察 AI 的思考与执行过程，建立了对生成结果的信任。阶段化设计也为后续的断点续传与手动干预提供了架构基础。

**依赖与关联**：
- 上游依赖 2.1 简洁模式提交的需求文本与模型选择
- 下游输出为 2.3 代码编辑器中的文件内容、2.4 文件树的节点状态、2.6 预览沙箱的运行 URL
- 依赖 2.7 迭代模式的上下文数据（项目文件树 + 需求规格）
- 依赖 2.8 构建日志面板展示阶段 5 的实时输出
- 依赖 Pinia Workbench Store 管理流水线状态机，通过纯函数 reducer 处理每条 SSE 消息
- 依赖 LangGraph4j 后端 StateGraph 驱动 6 节点条件路由与重试逻辑

**非功能需求**：
- SSE 连接需实现自动重连（最多 5 次，间隔递增）
- 简单应用（≤ 5 个文件）从提交到预览就绪总耗时 < 60s
- 流水线进度需以 Checklist 形式展示已完成/进行中/待处理的阶段列表
- 每个阶段失败后的错误信息需包含发生时间、错误类型与建议操作

#### 2.3 实时代码编辑器

- **功能名称**：基于 Monaco Editor 的多文件代码编辑器
- **入口路径**：`/workbench` 生成模式 → 右侧代码面板 Tab

**核心能力描述**：
平台集成 Monaco Editor（VS Code 内核）提供专业级代码编辑体验。编辑器支持 HTML、CSS、JavaScript、TypeScript、JSX、TSX、Vue SFC、JSON、Markdown 等多种编程语言的语法高亮、自动补全与代码折叠。用户可在编辑器中进行代码修改（直接键盘输入或粘贴替换），每次修改触发 500ms 防抖的自动保存（调用 `PUT /project/{id}/file`），保存成功后文件树中对应文件的状态标记更新为"modified"。编辑器顶部展示当前打开的文件路径与语言类型，支持多 Tab 快速切换已打开文件。编辑器底部状态栏展示光标行列位置、编码格式（UTF-8）、缩进设置（2 空格）及文件大小。对于 AI 生成后用户手动修改的文件，编辑器提供"与 AI 原始版本对比"的快捷操作入口，跳转至 2.5 的差异对比视图。

**业务价值**：专业级代码编辑器是面向开发者用户的核心体验要素。它让 AI 生成不再是"一次性消费"，用户可以随时对生成结果进行精细控制，将 AI 产出转化为符合自身标准的成品代码。

**依赖与关联**：
- 上游依赖 2.2 生成流水线推送的文件内容
- 下游修改通过 2.7 迭代模式提交为新的 AI 修改请求
- 依赖 2.4 文件树选择操作触发编辑器文件切换
- 自动保存依赖 `projectApi.saveFile()` 接口
- 文件修改状态同步至 2.5 差异对比视图的基线数据

**非功能需求**：
- Monaco Editor 按需加载，不应阻塞页面首屏渲染
- 单次语法高亮与自动补全响应延迟 < 50ms
- 自动保存失败时需在编辑器顶部展示"保存失败，点击重试"提示条
- 需为 `.vue` 文件提供 Vue SFC 语法高亮（template/script/style 分段）

#### 2.4 文件资源管理器

- **功能名称**：项目文件树面板
- **入口路径**：`/workbench` 生成模式 → 左侧文件树面板

**核心能力描述**：
文件资源管理器以树形结构展示当前项目的完整文件目录，节点类型包括目录节点（可展开/折叠，展开状态持久化至 localStorage）和文件节点（展示文件名、语言图标与修改状态标记）。文件状态分为三种：`new`（蓝色圆点，表示 AI 新生成且未被手动修改）、`modified`（橙色圆点，表示已被用户编辑过）、`unchanged`（无标记，表示从模板继承或克隆得来）。点击文件节点触发 2.3 代码编辑器加载对应文件内容；右键文件节点弹出上下文菜单，提供以下操作：重命名文件、删除文件、复制文件路径、下载文件。文件树的顶部工具栏提供"新建文件"与"新建目录"按钮，点击后在当前选中位置创建文件并立即进入文件名编辑状态。文件树的展示数据通过 `GET /project/{id}/tree` 获取，在后端由 `ProjectFileService` 按路径层级构建嵌套结构返回。

**业务价值**：文件树为用户提供了项目的"地图"——在 AI 生成的数十个文件中快速定位目标、理解项目结构、确认文件覆盖完整性。它是 IDE 体验中不可或缺的导航组件。

**非功能需求**：
- 1000 个文件节点的树渲染时间 < 500ms
- 目录展开/折叠状态需在页面刷新后保持
- 新增文件后自动在编辑器中打开该文件并聚焦

#### 2.5 代码差异对比

- **功能名称**：AI 生成代码的版本差异可视化
- **入口路径**：`/workbench` 生成模式 → 迭代修改后自动弹出或手动触发

**核心能力描述**：
当用户通过 2.7 对话式迭代触发了 AI 对代码的修改后，系统自动计算并展示修改前后的代码差异。差异对比采用 Monaco Editor 内置的 DiffEditor 组件，左右并排展示"修改前（AI 原始版本）"与"修改后（迭代结果）"两个编辑器视图，差异行以绿色（新增）、红色（删除）、蓝色（修改）高亮标记。在差异视图的一侧提供变更文件列表，按修改程度（新增行数 + 删除行数）降序排列，点击文件条目可快速跳转至该文件的差异详情。用户可以选择"接受全部变更"（将迭代结果写入项目文件）或"拒绝全部变更"（回退至迭代前状态），也可以逐文件接受/拒绝。对于小幅修改（如仅改动了 1-2 行），支持内联差异模式——在同一个编辑器中以标注形式展示修改，减少空间占用。

**业务价值**：差异对比解决了"AI 修改了什么"的透明度问题。用户在迭代过程中能精确追踪每次修改的范围与影响，避免了"黑盒修改"导致的不确定感。逐文件接受/拒绝的粒度控制让用户可以对 AI 产出保持最终决策权。

**依赖与关联**：
- 上游依赖 2.7 迭代修改产生的代码变更
- 差异基线数据来自 2.2 生成模式或上一次迭代后保存的文件快照
- 接受/拒绝操作提交至 `PUT /project/{id}/file` 接口，覆盖或保留文件内容
- 变更文件列表数据由 Pinia Workbench Store 通过 reducer 自动计算

**非功能需求**：
- 单文件差异计算与渲染时间 < 200ms
- 同时打开 10 个以上差异文件时需支持虚拟滚动减少 DOM 节点
- 差异视图需支持键盘快捷键（`Ctrl+Shift+.` 切换下一差异，`Ctrl+Shift+,` 切换上一差异）

#### 2.6 实时预览

- **功能名称**：沙箱隔离的应用实时预览
- **入口路径**：`/workbench` 生成模式 → 右侧预览面板 Tab

**核心能力描述**：
预览面板通过 iframe 加载沙箱环境中运行的用户应用，提供"所见即所得"的预览体验。面板顶部工具栏提供设备尺寸切换器（Desktop 1440px / Tablet 768px / Mobile 375px），切换时 iframe 宽度对应变化但保持高度自适应；还提供"刷新预览"按钮（在代码修改后手动触发重新加载）、"新窗口打开"按钮（在新标签页中打开预览 URL）和"复制链接"按钮。代码修改后，若沙箱支持 HMR（Hot Module Replacement），预览自动更新；否则用户手动刷新。预览面板右上角展示沙箱状态指示灯：绿色表示运行中，黄色表示启动中，红色表示已停止或出错。

**业务价值**：实时预览是"对话→生成→预览"闭环的最终交付环节。用户无需任何部署操作即可看到生成的完整应用运行效果，这是平台相对于传统 AI 代码生成工具的核心差异化能力。

**依赖与关联**：
- 上游依赖 2.2 流水线中阶段 6（预览部署）返回的预览 URL
- 上游依赖沙箱服务的启动与运行状态（SandboxService）
- 代码更新后需触发沙箱的 HMR 或手动刷新
- 依赖 2.8 构建日志面板在预览启动失败时提供错误排查信息

**非功能需求**：
- iframe 必须设置 `sandbox` 属性（`allow-scripts allow-same-origin`），禁止顶层导航
- 预览 iframe 与主站之间需使用独立子域名隔离（如 `*.lingma-preview.dev`），防止 XSS 攻击劫持主站 Token
- 沙箱启动时间（从代码就绪到预览可交互）< 30s

#### 2.7 对话式迭代

- **功能名称**：基于自然语言对话的增量代码修改
- **入口路径**：`/workbench` 生成模式 → 中部 AI 对话面板 → 输入框

**核心能力描述**：
在项目首次生成完毕后，用户可在对话输入框中以自然语言继续提出修改需求——例如"把导航栏改成深色主题"、"增加一个用户列表页面"、"修复登录按钮的响应式问题"。迭代流程由专门的 IterationAgent 驱动，其工作分为三阶段：**意图理解**——Agent 解析用户指令，判断修改范围（指定文件还是全局搜索）、修改类型（样式调整、功能新增、Bug 修复）；**代码定位**——Agent 通过 `searchCode` 工具在项目文件中搜索关键词，定位需要修改的具体代码位置；**修改生成**——Agent 通过 `patchFile` 工具以最小化增量的方式修改代码（支持行级替换和文件级新增），修改完成后触发 2.5 代码差异对比视图弹出，等待用户确认。对话面板以聊天气泡形式展示用户指令与 AI 响应（包含修改意图确认、搜索到的文件列表、修改摘要），支持多轮连续对话，每次迭代都以上一轮的结果为基线。

**业务价值**：对话式迭代将传统的"写需求文档→排期→开发→验收"流程压缩为"说人话→出结果"的即时反馈循环。用户不需要打开代码编辑器逐行修改，只需描述期望的变化即可完成迭代，大幅降低非开发者的使用门槛。

**依赖与关联**：
- 上游依赖 2.2 生成模式产出的完整项目文件与需求规格上下文
- 依赖 IterationAgent 的工具集（searchCode、patchFile、writeFile）
- 下游触发 2.5 差异对比视图的数据计算
- 依赖 2.8 构建日志面板在迭代引入编译错误时提供定位信息
- 对话历史需持久化至 `lf_chat_message` 表，确保页面刷新后不丢失

**非功能需求**：
- 单次迭代响应时间（从用户回车到差异对比弹出）< 30s
- 对话历史支持滚动加载（分页获取，每次加载 20 条）
- 需防止迭代导致项目文件数量爆炸——单次迭代最多新增 5 个文件

#### 2.8 构建日志面板

- **功能名称**：实时构建输出与控制台日志展示
- **入口路径**：`/workbench` 生成模式 → 底部诊断面板 → 日志 Tab

**核心能力描述**：
底部诊断面板的日志 Tab 以终端风格展示项目构建过程的实时输出，内容通过 SSE 流式推送到前端。日志面板支持以下特性：**实时流式追加**——构建过程中每一行 stdout 和 stderr 都即时追加显示，stderr 以红色文字高亮。**自动滚动**——日志输出时自动滚动至最底部，用户手动向上滚动后暂停自动滚动（右上角显示"自动滚动已暂停"并提供"回到底部"按钮）。**搜索过滤**——面板内提供搜索框，支持关键词过滤（仅显示包含关键词的行）和高亮匹配。**日志级别过滤**——支持按级别（INFO、WARN、ERROR）快速筛选日志行。**错误定位**——ERROR 级别的日志行提供"一键复制到剪贴板"按钮和"AI 诊断"按钮（点击后将错误信息发送给 AI 获取修复建议）。构建完成后，面板顶部显示构建结果摘要（总耗时、警告数、错误数），并提供"查看完整日志"按钮（下载为 `.log` 文件）。

**业务价值**：构建日志面板面向开发者用户，让 AI 生成过程中的编译问题透明可见。结合错误定位与 AI 诊断功能，用户无需离开工作台即可完成"发现问题→理解原因→修复问题"的完整调试闭环。

**依赖与关联**：
- 上游依赖 2.2 流水线中阶段 5（构建验证）和阶段 6（预览部署）的日志输出
- 依赖 SSE 推送管道，与 2.2 进度推送共用同一 EventSource 连接
- "AI 诊断"功能复用 2.7 迭代 Agent 的代码分析与修复能力

**非功能需求**：
- 日志流传输延迟 < 500ms（从后端产生到前端展示的时间差）
- 日志面板支持展示至少 10000 行日志而不出现性能卡顿
- 虚拟滚动实现，仅渲染可视区域内的日志行

### 模块三：创意中心

**业务定位**：创意中心是灵码工坊的**内容市场与社区生态层**。它汇聚了平台官方精选与用户发布的优质应用模板，供所有用户（包括未登录游客）浏览、筛选、订阅与克隆，是平台从"AI 工具"走向"创作社区"的战略支点。

**用户角色**：游客（浏览权限）、登录用户（订阅/克隆/发布权限）、创作者（发布/管理自己的创意权限）、管理员（审核/下架权限）。

#### 3.1 创意浏览与筛选

- **功能名称**：多维度的创意发现与筛选系统
- **入口路径**：`/creative`

**核心能力描述**：
创意浏览页面采用左侧筛选面板 + 右侧卡片网格的经典布局。左侧筛选面板提供四个维度的过滤条件：**分类筛选**——官网、电商、看板、工具、个人站，单选或多选（通过 Checkbox 实现），选中后右侧立即刷新结果；**标签云**——以不同字体大小的标签展示热门技术标签（如 AI、SaaS、响应式、企业级、TypeScript、Vue、React），字体大小反映该标签下的创意数量，点击标签可快速筛选；**价格筛选**——免费/付费（Pro）两个选项，付费创意需订阅平台付费套餐方可克隆和使用；**排序选择**——综合推荐、最新发布、最热（按订阅数）、高评分，点击后立即应用。右侧结果区以卡片或列表两种视图模式展示创意（通过视图切换按钮切换），卡片模式下每个创意展示缩略图、标题、创作者信息（头像+昵称+认证标记）、订阅数、评分、价格标签和状态标记（运行中/已发布）。列表模式下展示简化的信息行，适合快速浏览大量条目。筛选条件以 URL 查询参数形式存储（如 `?category=saas&tag=AI&price=free&sort=hot`），确保页面刷新后可还原筛选状态。页面底部提供分页控件，默认每页展示 12 个创意，支持按页码跳转。

**业务价值**：多维筛选让用户在海量创意中快速定位符合自身需求的作品——一位想做电商的用户可以在 3 次点击内找到"电商分类 + 免费 + 最热"的高质量模板，然后一键克隆为自己的项目开始定制。筛选效率直接决定了创意到项目的转化漏斗效率。

**依赖与关联**：
- 依赖 3.3 创意发布产生的创意数据
- 依赖 3.4 社区生态中的订阅数、评分与浏览数据
- 筛选条件的分类、标签元数据由 7.3 管理端配置维护
- 前端对接 `CreativeController.list()` 接口，支持多条件组合查询与分页

**非功能需求**：
- 筛选项变更后结果刷新时间 < 200ms
- 创意卡片缩略图需支持懒加载，降低首屏加载带宽
- 1000+ 创意列表需支持分页（服务端分页）与缓存策略

#### 3.2 精选轮播（Hero）

- **功能名称**：运营精选创意轮播展示
- **入口路径**：`/creative` 顶部 Hero 区域

**核心能力描述**：
创意中心顶部展示 4 张精选创意轮播卡片，采用自动轮播 + 手动导航的混合交互模式。每张轮播卡片包含创意的高清缩略图（作为背景，应用暗色渐变遮罩以保证前景文字可读性）、创意标题、简短描述、关键指标（订阅量、评分、收藏数、好评率以百分比展示）。轮播默认每 5 秒自动切换至下一张，切换时应用淡入淡出过渡动画（持续时间 500ms）。卡片底部提供圆点指示器标识当前进度，用户可点击圆点手动跳转至任意一张轮播，鼠标悬停时暂停自动播放。每张轮播卡片为可点击区域，点击后跳转至对应创意的详情页或直接触发克隆操作。

**业务价值**：Hero 轮播是创意中心的"橱窗"——平台运营团队通过精选内容引导用户注意力，推广高质量或付费创意，直接影响创意市场的转化率与营收。轮播的自动播放和视觉冲击力也为页面增加了动态活力。

**依赖与关联**：
- 精选内容由 7.3 管理端的"首页推荐位管理"配置（选择哪 4 个创意进入轮播）
- 轮播中的指标数据来自 3.4 社区生态的实时统计
- 点击跳转触发 3.3 创意详情的打开或克隆流程

**非功能需求**：
- 轮播图片预加载（前后各一张），确保切换时不出现白屏
- 缩略图需提供 2x 高清版本以适配 Retina 屏幕
- 轮播组件需在移动端支持触控滑动切换

#### 3.3 创意详情与操作

- **功能名称**：创意详情查看、订阅与克隆
- **入口路径**：`/creative` 点击任意创意卡片

**核心能力描述**：
每个创意对应一个详情视图（通过弹窗 Modal 或独立路由页面实现），展示创意的完整信息：大尺寸预览缩略图（支持多张轮播）、详细文字描述、标签列表（可点击跳转筛选同标签创意）、技术栈说明、文件结构预览（只读的简化文件树）、实时演示链接（如果创作者提供了沙箱 URL）。详情页右下区域展示创作者信息卡片（头像、昵称、简介、认证状态、作品总数、总订阅数），并提供"关注"按钮（点击后关注该创作者，后续发布的新创意会出现在关注者的创意中心首页推荐中）。详情页提供两个核心操作按钮：**订阅**——将创意加入自己的订阅列表，方便在工作台中快速找到并引用；**克隆为我的项目**——后端完整复制该创意对应的项目模板到当前用户的工作空间中，生成一个全新的项目，用户可以在此项目上进行二次开发和自定义。克隆操作需要校验用户的套餐权限（免费用户仅可克隆免费创意，付费用户可克隆所有创意）。

**业务价值**：订阅与克隆是创意中心商业闭环的关键环节。订阅构建了"用户→创作者"的持续关注关系，克隆完成了"灵感→我的项目"的价值转化。这两个操作将创意中心从单纯的浏览页面升级为驱动用户项目创建的核心渠道。

**依赖与关联**：
- 上游依赖 3.1 创意列表数据的点击跳转
- 订阅操作依赖 3.4 创作者关系链的建立与维护
- 克隆操作依赖 4.1 项目创建 API（以创意关联的模板项目为源进行 Fork）
- 权限校验依赖 5.1 用户套餐信息

**非功能需求**：
- 创意详情页首屏加载时间 < 2s
- 克隆操作需在后端异步执行，前端展示进度提示（如"正在克隆项目…"）
- 克隆后的项目需继承原始创意的依赖版本信息与架构约定

#### 3.3.1 创意评分与评论

- **功能名称**：创意作品的用户评价体系
- **入口路径**：`/creative` → 创意详情 → 评分与评论区

**核心能力描述**：
每个创意的详情页底部展示该创意的用户评价区域，包含评分与评论两个子模块。**评分系统**：采用 5 星评分制，用户可点击 1-5 星对创意进行整体评价（需登录后方可评分）。已评分用户再次点击可修改自己的评分，每次修改覆盖历史评分值。创意卡片和详情页头部展示该创意的平均评分（精确到小数点后 1 位）与评价总人数。为防止恶意刷分，同一用户对同一创意仅能评分一次（修改不产生新记录），且仅当用户实际克隆并使用过该创意后才可评分（基于常规设计补充）。评分数据通过 `POST /creative/{id}/rate` 提交，后端按创意 ID 维护评分聚合缓存（Redis Sorted Set），定时同步至 MySQL。**评论系统**：用户在评论区可发布文字评论（最长 500 字符，支持 Markdown 基本语法），评论以时间倒序排列，每条评论展示评论者头像、昵称、发布时间、评论内容，以及"有用"（👍）按钮和计数。评论支持两级嵌套回复——用户可回复某条评论，回复内容最长 300 字符。评论发布后端校验敏感词（通过第三方内容审核 API 或本地敏感词库），命中敏感词的评论自动拦截或进入审核队列（由 7.3 管理端处理）。评论者本人可编辑或删除自己的评论，创意作者可删除自己创意下的任何评论。评论区默认展示前 20 条热门评论（按"有用"计数排序），支持切换为"最新"排序和分页加载。

**业务价值**：评分与评论体系构建了创意的"社会证明"——高质量创意通过好评积累脱颖而出，新用户通过他人评价降低选择决策成本。同时对创作者形成正向激励：高评分带来更多曝光和订阅。评分与克隆挂钩的设计防止了"水军刷分"破坏社区信任。

**依赖与关联**：
- 评分数据聚合为 3.1 创意列表的"高评分"排序提供依据、3.2 轮播卡片的好评率指标、3.4 趋势图中的评分分布数据
- 评论数据存储于 `lf_creative_comment` 表，评分数据存储于 `lf_creative_rating` 表
- 评论敏感词过滤依赖内容审核服务或 7.3 管理端的人工审核队列
- 评分权限校验依赖 4.1 用户的项目克隆记录
- 评论新增/删除触发 1.5 作品通知（通知创意作者）和 1.7 消息通知中心的推送

**非功能需求**：
- 评分操作响应时间 < 200ms，平均评分更新延迟 < 1s
- 评论列表首屏加载时间 < 500ms（含前 20 条评论的渲染）
- 敏感词检测响应时间 < 100ms（本地词库）或 < 1s（远程 API）
- 单条评论支持至少 100 条二级回复的嵌套展示

#### 3.3.2 创意举报

- **功能名称**：用户对违规创意的举报入口
- **入口路径**：`/creative` → 创意详情页 → "举报"链接（位于页面底部或右上角溢出菜单）

**核心能力描述**：
任何已登录用户在浏览创意详情时，若发现该创意存在违规行为（如包含恶意代码、侵犯他人知识产权、含有不当内容、涉嫌欺诈、虚假宣传或其他违反社区规范的行为），可通过"举报"入口发起投诉。点击"举报"后弹出一个表单弹窗，用户需选择举报类型（预设选项包括：侵权内容、恶意代码/挖矿脚本、不当/敏感内容、虚假描述与实际不符、垃圾广告、其他），并在补充说明文本框中填写详细描述（最长 500 字符，必填）。用户可选择性上传截图证据（最多 3 张，每张限制 2MB）。提交后，举报记录进入 7.3 管理端的投诉处理队列，创意状态不受影响（在被管理员判定违规前保持正常展示）。管理员处理举报后（确认违规并下架、或判定不成立驳回），系统通过 1.7 消息通知中心通知举报人处理结果（仅告知"已处理"，不透露具体处理细节以保护被举报方隐私）。为防止恶意举报，同一用户对同一创意 7 天内最多举报 1 次，且对于已被管理员判定为"不成立"的举报类型，用户不可就同一类型再次提交。

**业务价值**：举报功能是社区内容治理的群众防线——它将内容审核的压力从纯管理端人工排查分散到全体用户的共同监督中，让违规内容更快被发现和处理，是 UGC 平台安全运营的标配机制。与 7.3 管理端审批流上下联动，形成"用户发现→平台处理→结果反馈"的完整治理闭环。

**依赖与关联**：
- 举报提交进入 7.3 管理端的投诉处理队列，管理员处理后结果通知举报人
- 举报数据存储于 `lf_creative_report` 表，需记录举报人、被举报创意、举报类型、描述、处理状态与处理结果
- 举报结果通过 1.7 消息通知中心推送至举报人
- 恶意举报防滥用依赖举报频次限制与管理员对"不成立"举报的标记机制

**非功能需求**：
- 举报提交响应时间 < 500ms
- 举报处理结果通知延迟 < 1 小时（管理端处理完成后自动触发）
- 单个用户每日最多提交 10 次举报，防止举报功能被滥用

#### 3.4 社区生态

- **功能名称**：创作者关系链与平台趋势展示
- **入口路径**：`/creative` 右侧创作者边栏 + 底部趋势区

**核心能力描述**：
创意中心在侧边栏和底部提供社区生态的四个关键组件。**热门创作者列表**：展示当前平台上订阅数最高的 5 位创作者，每位展示头像、昵称、简介（一行省略）、认证标记（蓝色 V 标识，表示平台认证创作者）、订阅数与"关注"按钮（已关注则显示"已关注"）。点击创作者头像进入其个人主页（展示该创作者发布的所有创意）。**本周上新**：独立区域展示本周内新发布的创意列表，区分免费和付费标签，按发布时间倒序排列，展示 6-8 个创意缩略卡片。**创意趋势图**：以折线图形式展示平台在过去 7 天内的三项关键趋势——每日新增创意数、每日新增订阅数、每日活跃用户数，帮助用户感知平台生态的健康度与活跃度。**创作者认证体系**：平台为高质量创作者提供认证机制，认证标识在创作者头像旁以蓝色 V 形徽章展示，在创意卡片、创作者列表和创作者主页中均可见。认证采用"用户申请 + 管理端审核"模式——创作者在个人中心（1.3 或 3.4 创作者主页）点击"申请认证"按钮，填写申请表单（真实姓名或企业名称、联系方式、个人简介、代表作链接、身份证明材料上传），提交后进入 7.2 管理端的认证审核队列。管理员审核通过后该创作者获得认证标识，同时在创意搜索排序中获得额外权重加成。若认证创作者后续出现违规行为，管理员可在 7.2 中撤销其认证身份。认证标识分两个等级：个人认证（蓝色 V，面向独立开发者和设计师）和机构认证（金色 V，面向企业或设计工作室），认证标准由管理端策略配置决定。

**业务价值**：社区生态组件为创意中心填充了"人"的维度。热门创作者激励创作者持续发布高质量作品，趋势图增强了平台公信力，本周上新保障了新发布的创意能获得足够的初期曝光。创作者认证体系在创作者经济中起到"信任中介"的作用——认证标识降低了用户筛选高质量创作者的成本，同时也为平台运营提供了对优质创作者的识别与扶持能力。

**依赖与关联**：
- 关注关系依赖 `lf_user_follow` 表，需提供"关注/取消关注"API
- 趋势数据从后台定时统计任务（每小时聚合）生成，前端通过 `GET /creative/trends?days=7` 获取
- 本周上新与 3.1 创意浏览共用同一创意列表 API，通过时间范围参数过滤
- 创作者认证申请提交至 7.2 管理端的认证审核队列，审核结果通过 1.7 消息通知中心推送
- 认证标识数据存储于 `lf_user` 表的 `verified_type` 和 `verified_at` 字段

**非功能需求**：
- 趋势图数据缓存 1 小时，减少实时聚合开销
- 关注/取消关注操作需即时更新 UI 按钮状态（乐观更新 + 失败回滚）
- 创作者列表数据刷新间隔 5 分钟

#### 3.5 创意发布

- **功能名称**：用户将项目发布为社区创意
- **入口路径**：`/works` → 项目卡片操作菜单 → "发布为创意"

**核心能力描述**：
拥有完整项目的用户可将自己的作品发布到创意中心供其他用户浏览、订阅和克隆。发布流程通过一个分步弹窗完成：**第一步"创意信息"**——用户填写创意名称（必填，最长 80 字）、描述（必填，最长 500 字，支持 Markdown 格式）、选择分类（从预设下拉列表选择）和标签（从已有标签库中选择或输入新标签，最多 5 个）。**第二步"定价与预览"**——用户设置定价方式（免费/付费一次购买/按月订阅），上传或从项目中截取预览缩略图（支持多张，第一张为主图），提供沙箱演示链接（可选）。**第三步"确认发布"**——系统汇总前两步信息并展示预览效果，用户确认后提交发布请求。提交后，创意进入"待审核"状态（在 7.3 管理端审批流中排队），审核通过后创意在创意中心可见。已发布的创意支持"编辑"（修改创意信息，修改后无需重新审核，但涉及定价或描述重大变更时需重新提交审核）和"下架"（从创意中心移除，但不影响原项目）。

**驳回管理与重新提交**：当创意审核被驳回后，该创意的状态更新为"已驳回"，同时在创意中心不可见。用户在 4.1 项目列表或专门的"我的创意"管理页面中可查看自己提交的所有创意及其当前审核状态（待审核 / 已发布 / 已驳回 / 已下架）。对于"已驳回"的创意，卡片上以橙色标签标识驳回状态，点击进入创意详情页后，顶部展示驳回原因横幅（包含管理员填写的驳回原因与建议），横幅下方提供"修改并重新提交"按钮。点击后进入编辑模式——用户根据驳回原因修改创意的相关信息（如修改不符合规范的描述、补充缺失的缩略图、调整不合理的定价等），所有字段可编辑且保留上次提交的值。修改完毕后点击"重新提交"，创意的状态变为"待审核"，重新进入 7.3 管理端的审批队列（按原提交时间保留排队优先级，非重新排队），审核通过后正常发布。同一创意最多允许重新提交 3 次，超过 3 次驳回后创意状态变为"已拒绝"（不可再次提交），用户需联系客服或重新以新创意发布。被驳回的创意不影响其关联的原项目——用户仍可在工作台中对原项目进行任何编辑操作。

**业务价值**：创意发布是 UGC 生态的供给端入口，将用户从"消费者"转变为"创作者"。定价机制赋予优质创作者变现能力，审核流程保障了创意质量与平台安全。驳回后重新提交的完整流程避免了"一次驳回就放弃"的创作者流失——通过明确的驳回原因引导和保留上次提交内容的设计，用户在 2-3 次修改内即可达到发布标准，最大化 UGC 供给端的转化率。

**依赖与关联**：
- 上游依赖 4.1 项目完成生成或开发的可用项目
- 发布提交触发 7.3 管理端的审批流程
- 审核结果（通过/驳回）通过 1.7 消息通知中心推送至用户
- 发布成功后创意数据进入 3.1 创意浏览的数据源
- 已驳回创意的管理入口位于 4.1 项目列表的扩展标签页"我的创意"中
- 创意状态流转（待审核→已发布/已驳回→重新提交→待审核→已发布/已拒绝）需在创意状态机中完整定义

**非功能需求**：
- 发布提交需校验项目必须完整（至少包含 1 个可渲染页面的代码）
- 缩略图上传需自动压缩至 800×600px，每张限制 500KB
- 审核队列需支持按提交时间排序，管理员单次审核操作 < 3s
- 重新提交时前端需保留用户上次填写的所有表单数据，减少重复输入
- 驳回通知推送延迟 < 1 分钟（从管理员操作驳回到用户收到通知）

### 模块四：作品管理

**业务定位**：作品管理是用户**个人项目资产的统一管理中心**。它提供了项目从创建、编辑、发布到归档的全生命周期管理能力，是连接 AI 工作台（产出）与创意中心（分享）的中间枢纽。

**用户角色**：登录用户（管理自己的项目）、团队管理员（管理团队项目）。

#### 4.1 项目列表与搜索

- **功能名称**：个人项目全集展示与高效检索
- **入口路径**：`/works`

**核心能力描述**：
作品管理页面以卡片网格形式展示当前用户的所有项目（默认按最近更新时间降序排列）。每张项目卡片包含：项目缩略图（由项目首页自动截取生成）、项目名称、项目描述（两行截断）、项目状态标签——已部署（绿色）、开发中（蓝色）、生成中（黄色带加载动画）、草稿（灰色）——以及项目统计（页面数、API 接口数）。页面顶部提供多功能工具栏：**搜索框**——支持按项目名称或关键词进行模糊匹配搜索，输入 300ms 防抖后实时过滤结果；**分类筛选标签栏**——全部、最近（7 天内更新）、收藏（用户标记为收藏的项目）、回收站（已软删除的项目），点击切换数据源；**排序选择器**——按更新时间、创建时间、文件数量排序。工具栏右侧提供"新建项目"按钮用于快速创建空白项目。卡片支持右键菜单操作（编辑、重命名、标记收藏/取消收藏、发布为创意、删除）。页面底部提供分页控件。

**业务价值**：项目列表是用户访问平台后的"控制面板"——在这里用户可以快速找到正在进行的工作、管理资产状态、并通过搜索和筛选在面对大量项目时保持效率。回收站机制防止用户误删除造成的损失。

**依赖与关联**：
- 数据源为 `ProjectController.listProjects()` API，需支持分页、排序、搜索参数
- 项目状态由 2.2 生成模式的进度和 2.6 预览部署结果驱动更新
- 发布为创意操作连接 3.5 创意发布流程
- 回收站数据依赖软删除字段（`is_deleted`、`deleted_at`）
- 收藏状态由 `lf_project_favorite` 关联表维护，前端通过 `POST/DELETE /project/{id}/favorite` 切换收藏状态，列表查询时通过 `is_favorite` 参数过滤

**非功能需求**：
- 项目列表首屏加载时间 < 1s（前 20 条）
- 搜索防抖 300ms，搜索结果返回 < 200ms
- 项目缩略图按需生成（首次访问项目列表时异步截取，缓存至 CDN）
- 收藏/取消收藏操作响应时间 < 300ms，需采用乐观更新策略确保 UI 即时反馈

#### 4.2 项目 CRUD 操作

- **功能名称**：项目生命周期全管理
- **入口路径**：`/works` 项目卡片操作菜单 + `/workbench` 工作台

**核心能力描述**：
项目 CRUD 覆盖从创建到删除的完整生命周期。**新建项目**：支持三种创建方式——（1）通过 `/workbench` 工作台输入需求由 AI 自动生成；（2）从模板创建——在 `/works` 点击"新建"，从模板列表中选择一个预设模板（如空白 Vue 项目、空白 React 项目），后端基于模板初始化项目结构；（3）克隆创意——从 3.3 创意中心克隆他人的创意为自己的项目。**编辑项目**：在工作台中修改项目代码（通过 2.3 代码编辑器），或在工作台外修改项目元信息（名称、描述、框架）。**删除项目**：删除操作将项目移入回收站（软删除），回收站中的项目在 30 天后自动彻底清理。在回收站期间用户可以选择"还原"（恢复至正常状态）或"彻底删除"（立即从数据库和文件系统中清除）。**项目导出**：用户可导出项目为 ZIP 压缩包，包含完整的源代码文件与 `package.json` 依赖配置，下载后可在本地环境直接运行。

**业务价值**：完整的 CRUD 操作赋予用户完全的项目控制权——自由创建、放心修改、安全删除，形成"创作→修改→存档→清理"的完整资产管理闭环。恢复机制防止误操作导致的资产损失。

**依赖与关联**：
- 新建项目依赖模板系统（后端预设模板或 3.3 克隆源）
- 删除/还原依赖软删除字段与定时清理任务
- 项目导出依赖文件系统读取与 ZIP 打包服务
- 克隆创意依赖创意关联的模板项目数据

**非功能需求**：
- 新建项目（模板方式）从点击到跳转工作台 < 2s
- 项目导出（含 50+ 文件）生成时间 < 5s，文件大小不限
- 回收站定时清理任务每日凌晨执行，清理超过 30 天的已删除项目

#### 4.3 项目统计概览

- **功能名称**：个人项目资产统计速览
- **入口路径**：`/works` 页面顶部统计卡片区

**核心能力描述**：
在项目列表上方展示四张统计卡片，提供用户项目资产的宏观视图：**全部项目**（所有未删除项目的总数）、**已部署**（当前有活跃沙箱和预览 URL 的项目数）、**开发中**（生成完成但尚未部署或正在修改的项目数）、**草稿**（创建但未完成首次生成的项目数）。数字以大字突出显示，下方附有百分比环比变化（如"较上周 +12%"）。点击任意统计卡片可将项目列表切换至仅显示对应状态的项目（如点击"已部署"则列表仅展示已部署项目），卡片高亮标识当前筛选状态。数据通过 `GET /project/stats` 实时获取。

#### 4.4 最近动态时间线

- **功能名称**：项目操作活动日志
- **入口路径**：`/works` 页面右侧动态面板

**核心能力描述**：
最近动态以时间线形式展示当前用户所有项目的近期操作记录，包括但不限于：生成完成（"项目 X 的 AI 代码生成已完成"）、文件更新（"你修改了项目 X 的 App.tsx"）、部署事件（"项目 X 已部署至预览环境"）、创意发布（"项目 X 已发布为创意"）、克隆记录（"你从创意 Y 克隆了项目 X"）。每条动态包含操作图标（按类型区分颜色和图标）、操作描述文字、时间戳（当日显示时分、昨日显示"昨天"、更早显示日期），以及可点击的跳转链接（点击项目名称跳转至工作台编辑，点击创意名称跳转至创意详情）。时间线默认展示最近 20 条动态，底部提供"加载更多"按钮。动态数据来自后端 `lf_activity_log` 表的实时查询，通过 `GET /project/{id}/activities` 或 `GET /user/activities` 获取。

**业务价值**：活动时间线为用户提供了项目变更的审计追踪能力——当多个项目并行推进时，用户可以快速回溯"我昨天对哪个项目做了什么"，减少认知负担并提升工作效率。

**依赖与关联**：
- 活动日志的写入由 2.2 生成完成、2.3 文件修改保存、3.5 创意发布等事件自动触发
- 依赖 `lf_activity_log` 表的 INSERT 操作无感知写入（异步或同步）
- 跳转链接需连接至 2.2 工作台或 3.3 创意详情

**非功能需求**：
- 活动日志写入应为异步操作，不影响主业务响应时间
- 单用户活动日志保留期 180 天，超期数据自动归档至冷存储

#### 4.5 项目版本历史

- **功能名称**：Git 式项目版本快照与回滚
- **入口路径**：`/workbench` → 工具栏 "版本历史" 按钮

**核心能力描述**：
版本历史为每个项目维护一个按时间排序的快照链，类似于 Git 的提交记录。每当用户触发以下事件时系统自动创建快照：（1）AI 代码生成流水线完成，（2）用户接受一次迭代修改的差异，（3）用户手动点击"保存版本"。每个快照包含：快照编号（如 V1、V2）、创建时间、创建触发方式（自动/手动）、快照描述（自动生成或用户输入）、包含的文件数量与代码总行数。在版本历史面板中，用户可查看快照列表，点击任意快照查看该版本的完整文件树（只读模式，不可编辑），并可选择"回滚至此版本"——操作后当前项目代码完全替换为该快照的状态，同时自动创建一个新的快照（Vn）记录回滚前的状态，确保回滚操作可撤销。用户也可在回滚前选择"对比当前版本"——在两个版本之间打开 2.5 差异对比视图，精确查看变化。

**业务价值**：版本历史解决了"AI 修改不如预期后如何安全回退"的核心痛点。它让用户可以大胆尝试各种迭代指令，知道任何时候都可以通过一次点击回到之前的状态，消除了使用 AI 迭代时的焦虑感。

**依赖与关联**：
- 快照创建由 2.2 生成完成和 2.7 差异接受操作自动触发
- 快照数据存储依赖文件系统或数据库的全量/增量存储方案
- 回滚操作需触发 2.6 沙箱重启以反映回滚后的代码
- 版本对比复用 2.5 差异对比的 DiffEditor 组件

**非功能需求**：
- 单个快照存储空间 < 项目文件总大小的 2 倍（存储全量副本）
- 单项目最多保留 50 个快照，超出后自动删除最旧的快照
- 快照读写操作响应时间 < 2s（50 文件级项目）

### 模块五：订阅与计费

**业务定位**：订阅与计费模块是平台的**商业化变现引擎**。它通过三级套餐体系实现用户分层付费，通过用量监控实现成本透明化，通过支付与账单管理实现财务闭环。

**用户角色**：游客（浏览套餐）、登录用户（订阅/支付/管理）、管理员（财务对账/发票审批）。

#### 5.1 套餐选择

- **功能名称**：三级定价套餐展示与选择
- **入口路径**：`/pricing`

**核心能力描述**：
定价页面以三列卡片形式展示平台的三种订阅套餐：**基础版（¥19/月 或 ¥190/年）**——包含每月 5 次 AI 生成、2 个活跃项目、1 次部署、基础模板库访问；**专业版（¥49/月 或 ¥490/年）**——包含每月 30 次 AI 生成、10 个活跃项目、5 次部署、Pro 模型权限、创意克隆权限；**尊享版（¥99/月 或 ¥990/年）**——包含无限 AI 生成、无限活跃项目、无限部署、全模型权限、团队协作（5 席）、API 密钥、优先支持。每张卡片突出展示月付价格（年付标注折扣后月均价和总节省金额），按年/按月通过 Toggle 切换。卡片内列出该套餐的所有包含权益（勾选标记）和限制项（叉号标记）。每张卡片底部的 CTA 按钮根据用户当前状态动态变化：未登录用户显示"免费开始"（引导注册），已登录免费用户显示"立即升级"，已是该套餐用户显示"当前方案"（灰色不可点击），更高套餐用户则不显示按钮。点击升级按钮跳转至支付页面。

**业务价值**：三级套餐的梯度设计覆盖了从个人探索者（基础版）到专业开发者（专业版）再到小型团队（尊享版）的完整用户谱系，通过用量限额和功能差异驱动用户从低套餐向高套餐的渐进式升级。

**依赖与关联**：
- 套餐定义数据由后端配置（配置文件或 `lf_plan` 表）驱动，前端通过 `GET /plan/list` 获取
- CTA 按钮状态依赖用户登录状态（1.2）和当前套餐信息（5.3）
- 升级操作触发 5.5 支付流程
- 套餐权益变更需同步更新至 5.2 功能对比表

**非功能需求**：
- 套餐配置变更需即时生效，不应要求前端发布
- 价格展示支持人民币（¥）和美元（$）两种货币单位（通过用户地区自动选择或手动切换）

#### 5.2 功能对比表

- **功能名称**：多维度套餐功能对比
- **入口路径**：`/pricing` 套餐卡片下方

**核心能力描述**：
功能对比表以表格形式提供三种套餐的详细功能差异对比，按功能维度分为四个分组：**AI 能力**（每月生成次数、可用模型范围、单次生成最多文件数、迭代修改次数）、**项目部署**（活跃项目数、部署次数、沙箱运行时长、自定义域名）、**团队协作**（团队席位、角色权限、批量操作、审计日志）、**服务支持**（工单优先级、在线客服、专属客户经理、SLA 承诺）。每行左侧为功能名称与说明，右侧三列分别对应三级套餐的包含情况（勾号/叉号/带数字标签如"10"）。表格支持行分组折叠/展开，默认展开所有分组。表头采用粘性定位（sticky），滚动时表头保持可见，方便用户在长表格中对比。

**业务价值**：功能对比表帮助决策者在信息充分的情况下选择套餐——当用户看到专业版比基础版在"可用模型范围"一项上的显著差异时，升级转化率会因信息透明而提升。

**依赖与关联**：
- 数据源与 5.1 共用套餐配置数据
- 表格内容随套餐配置变更而自动更新

**非功能需求**：
- 对比表中的功能项需支持 Tooltip 悬停展示详细说明
- 在移动端表格需改为折叠卡片模式（每个套餐独立展开显示权益清单）

#### 5.3 订阅管理

- **功能名称**：个人订阅状态管理与升级
- **入口路径**：`/subscription`

**核心能力描述**：
订阅管理页面展示当前用户的订阅详情并提供管理操作。**当前方案详情卡片**：展示当前套餐名称、月付/年付价格、下一次续费日期、已使用天数（以进度条展示计费周期完成百分比）、以及该套餐的核心权益列表。**用量监控区域**（详见 5.4）展示四项用量指标。**操作区域**：提供"升级套餐"按钮（点击跳转至 5.1 定价页选择更高套餐，系统自动计算按比例退款或折抵）、"管理订阅"按钮（打开 Stripe/支付宝的客户门户，可修改支付方式或账单地址）、"取消订阅"按钮（停止自动续费，当前周期结束后降级为免费版）。页面底部展示订阅历史——用户曾订阅过的套餐及切换记录。

**业务价值**：订阅管理是用户与平台付费关系的"中央控制面板"——透明的套餐信息、清晰的用量进度、便捷的升级/取消操作，降低了用户对付费的焦虑感并提升了续费率。

**依赖与关联**：
- 依赖 5.1 的套餐数据渲染当前方案信息
- 依赖 5.4 的实时用量数据填充进度条
- 升级/取消操作依赖 5.5 支付系统的订单管理与退款能力
- 订阅状态变更触发 1.5 账单类通知

**非功能需求**：
- 订阅状态需与支付侧实时同步（延迟 < 5s）
- 取消订阅前展示挽留弹窗（列出用户当前已使用但取消后将丧失的权益）
- 套餐升级按比例计费，计算精度精确到天

#### 5.4 用量监控

- **功能名称**：实时资源消耗追踪
- **入口路径**：`/subscription` 用量监控区域

**核心能力描述**：
用量监控区域以四个仪表式卡片展示用户当月核心资源消耗：**AI 生成次数**（已用/总量，进度条 + 数字展示）、**Token 用量**（已消耗 Token 数/总量限制）、**部署时长**（沙箱运行累计小时数/月限制）、**团队席位**（已占用/总席位数）。每张卡片包含三个元素：环形或条状进度条（按比例着色——低于 60% 为绿色，60-90% 为橙色，超过 90% 为红色）、当前数值与上限数值的文本展示，以及预估剩余可用的天数（如"按当前消耗速度，预计还可使用 12 天"）。当某项指标接近上限时（如超过 90%），卡片右上角显示橙色警告图标，并在 1.5 通知中推送"用量即将耗尽"的提醒。用量数据通过 Redis 原子计数实时更新，后端通过 `GET /subscription/usage` 返回当前周期内的所有指标值。

**业务价值**：用量监控让资源消耗从"看不见的黑盒"变为"随时可查的仪表盘"——用户在接近用量上限时能提前感知并自主决策是减少使用还是升级套餐，而非在超额时突然遇到功能阻断的挫败体验。

**依赖与关联**：
- 计数数据源来自 2.2 生成调用（AI 次数 + Token）、2.6 沙箱运行（部署时长）、7.2 团队管理（席位）
- Redis 原子计数需定时同步至 MySQL 持久化存储
- 用量接近上限需触发 1.5 账单类通知推送
- 超出上限时 2.2 生成请求将被拦截并提示升级套餐

**非功能需求**：
- 用量计数精确度 ≥ 99%，实时更新延迟 < 5s
- 每月 1 日 00:00（UTC+8）自动重置月度计数器
- 用量超限拦截响应时间 < 100ms（直接基于 Redis 返回值判断）

#### 5.5 账单与支付

- **功能名称**：支付流水、账单记录与发票管理
- **入口路径**：`/subscription` → 账单与支付 Tab

**核心能力描述**：
**账单记录表**以表格形式展示用户的历史账单列表，每行包含：账单编号、账单周期（如 2026-06-01 至 2026-06-30）、套餐名称、支付金额、支付方式（VISA / 微信支付 / 支付宝）、支付状态（已支付/待支付/已退款）、支付时间。表格支持按时间范围、支付状态筛选，支持点击账单编号查看详情。**支付方式管理**：展示当前已绑定的支付方式（支持 VISA/MasterCard 信用卡、微信支付、支付宝），提供"添加支付方式"按钮（打开对应的安全绑卡页面，不存储完整卡号仅存储 Token）。**发票下载**：已支付的账单提供"开具发票"按钮，点击后填写发票抬头（公司名称 + 纳税号）和接收邮箱，后端生成 PDF 电子发票并发送至邮箱，同时提供页面内直接下载。系统支持普通发票和增值税专用发票两种类型，开票信息自动保存以避免重复填写。

**业务价值**：完整的账单与支付系统是商业化产品的合规底线。清晰的账单记录保障了财务透明度，便捷的发票服务降低了企业用户的报销摩擦，多种支付方式覆盖了国内外用户的支付习惯。

**依赖与关联**：
- 支付流程依赖第三方支付网关（微信 Native 支付 / 支付宝当面付 / Stripe）的回调通知
- 支付成功后触发 5.3 订阅状态的自动更新
- 账单记录与 5.4 用量抵扣数据需保持一致性（同一周期的生成次数计费与用量展示对应）
- 发票生成依赖 PDF 生成库（iText 或基于 HTML 的 Puppeteer 渲染）

**非功能需求**：
- 支付回调处理需实现幂等性（同一笔交易多次回调不重复处理）
- 发票生成时间 < 5s，格式为 PDF/A 兼容
- 账单记录至少保留 3 年以满足税务审计要求
- PCI-DSS 合规：不存储完整信用卡号，仅保留支付网关返回的 Token 和卡号后 4 位

#### 5.6 订阅设置

- **功能名称**：订阅偏好与自动化配置
- **入口路径**：`/subscription` → 设置区域

**核心能力描述**：
订阅设置以 Toggle Switch 组提供四项自动化配置：**自动续费**——开启后在每个计费周期结束时自动从默认支付方式扣款续费，关闭后到期降级为免费版。**续费提醒**——开启后在续费日前 3 天和 1 天分别发送提醒通知。**用量提醒**——开启后在任意用量指标超过 80% 和 95% 时发送提醒通知。**取消订阅**——提供取消订阅的入口（触发挽留弹窗与确认流程，详见 5.3）。所有开关的操作实时生效，通过 `PUT /subscription/settings` 提交。设置区域下方展示下一账单的预估金额和扣款日期。

**业务价值**：订阅设置赋予用户对计费关系的精细控制权——自动续费提升平台的续费率，用量提醒防止用户在不知情的情况下超额，取消操作的"软挽留"设计在尊重用户选择权的同时最大化留存可能性。

**依赖与关联**：
- 自动续费触发 5.5 支付流程的自动扣款
- 提醒通知通过 1.5 通知设置的账单类通知渠道推送
- 取消订阅生效后触发 5.3 套餐降级与 5.4 用量限额的重新设定

**非功能需求**：
- 设置项切换后 1 秒内完成保存，失败时回滚 Switch 状态并 Toast 提示
- 自动续费扣款失败时需重试 3 次（间隔 24 小时），全部失败后降级并通知用户

#### 6.1 分组目录导航

#### 6.2 文档内容渲染

- **功能名称**：Markdown 文档的富文本展示
- **入口路径**：`/doc` → 中间内容区

**核心能力描述**：
文档正文区域以精美的排版展示 Markdown 格式的文档内容，支持以下渲染特性：**标准 Markdown 语法**——标题（H1-H6 带底部边框分隔）、段落、加粗、斜体、删除线、有序/无序列表、引用块（左侧蓝色边框）、水平分割线、链接（外部链接新窗口打开，内部锚点链接平滑滚动）、图片（支持点击放大预览）、表格（斑马纹行 + 表头固定）。**代码块增强**——代码块显示语言标签（右上角）、提供"复制代码"按钮（点击后复制整个代码块内容并显示"已复制"提示，2 秒后恢复）、多语言代码示例支持 Tab 切换（如 npm/yarn/pnpm 安装命令）。**信息提示框**——支持 Info（蓝色）、Warning（黄色）、Danger（红色）三种级别的提示框，用于强调重要信息或警示。文档底部提供"这篇文章对你有帮助吗？"的反馈按钮（👍 / 👎），反馈数据用于优化文档质量排名。

**业务价值**：高质量的文档渲染体验直接影响用户的学习效率和平台留存——清晰的排版、便捷的代码复制和反馈机制让文档中心成为用户信赖的自助支持渠道，降低工单负荷。

**依赖与关联**：
- 文档内容通过 `GET /doc/article/{id}` 获取 Markdown 原文
- Markdown 渲染依赖前端解析器（如 marked + highlight.js 或 Shiki）
- 代码块复制依赖 Clipboard API
- 反馈数据通过 `POST /doc/feedback` 提交，汇总至 7.1 管理端大盘

**非功能需求**：
- Markdown 渲染时间 < 100ms（中等文档长度）
- 代码高亮需支持 HTML/CSS/JavaScript/TypeScript/Vue/React/JSON/Bash 八种语言
- 文档反馈按钮操作响应 < 200ms

#### 6.3 TOC 导航与滚动同步

- **功能名称**：右侧目录导航与滚动位置同步
- **入口路径**：`/doc` → 右侧 TOC 面板

**核心能力描述**：
文档页面右侧展示一个固定的目录导航面板（Table of Contents），自动提取文档中的所有 H2 和 H3 标题生成层级目录。当前正在阅读的标题在 TOC 中以高亮样式（左侧蓝色竖条 + 文字加粗）标识，当用户在文档内容区滚动时，TOC 高亮项通过 IntersectionObserver 实现实时同步。点击 TOC 中的任意标题，页面平滑滚动至对应内容位置。TOC 标题过长时自动截断并以省略号显示，hover 时展示完整标题 Tooltip。面板顶部显示"目录"标题，底部提供一个"回到顶部"的快捷按钮。

**业务价值**：TOC 导航让用户能在长篇文档中快速建立全局认知并精准跳转——面对一份包含 20 个章节的 API 文档，用户可在 2 秒内找到目标章节，显著提升查阅效率。

**依赖与关联**：
- 依赖 6.2 渲染后的文档 DOM 结构（H2/H3 标签的 ID 锚点）
- 滚动同步依赖浏览器 IntersectionObserver API
- 点击跳转依赖 CSS `scroll-behavior: smooth` 或 JavaScript 滚动 API

**非功能需求**：
- TOC 高亮同步延迟 < 100ms
- 当视口内同时存在多个标题时，高亮最接近视口顶部的一个
- TOC 面板在文档页最大高度下支持内部滚动（`max-height: calc(100vh - header-height)`）

#### 6.4 FAQ 问答

- **功能名称**：手风琴式常见问题解答
- **入口路径**：`/doc` → 页面底部 FAQ 区域

**核心能力描述**：
FAQ 区域以手风琴（Accordion）交互展示平台的常见问题与解答。每个 FAQ 条目由一个问题和答案组成，默认折叠状态仅展示问题文本（左侧带问号图标），点击问题文本或右侧的展开箭头后，答案内容以平滑的高度动画展开显示。支持同时展开多个条目（非互斥模式），用户点击已展开的条目则折叠。FAQ 内容分类展示（如"账号相关"、"生成问题"、"部署相关"、"计费疑问"），每个分类作为独立的手风琴组。页面顶部提供 FAQ 搜索框，输入关键词实时过滤匹配的问题（模糊匹配问题和答案文本），匹配结果高亮显示关键词。FAQ 内容通过 `GET /doc/faq` 获取，支持按分类参数过滤。

**业务价值**：FAQ 是降低客服工单量的第一道防线——用户遇到问题时首先可以在 FAQ 中搜索答案，大多数"如何修改生成后的代码？""为什么构建失败？"之类的重复性问题可以在此自助解决。

**依赖与关联**：
- 依赖后端 FAQ 内容管理（数据库存储或配置文件）
- FAQ 分类与 6.1 文档分组的"常见问题"目录对应
- FAQ 搜索的高效性影响用户对文档站的整体满意度

**非功能需求**：
- 手风琴展开/折叠动画持续时间 300ms（ease-in-out 缓动）
- FAQ 搜索防抖 200ms，结果过滤响应 < 100ms

### 模块七：管理端系统（独立前端项目）

**业务定位**：管理端是平台**运营治理的中枢系统**，面向内部管理员提供用户管理、内容审核、资源监控、财务对账等管理能力。管理端定位为独立的 Vue 3 前端项目，与用户端通过统一的 RESTful API 通信，使用独立的管理员认证体系。

**用户角色**：超级管理员、运营管理员、财务管理员（角色权限分离）。

#### 7.0 管理员认证与权限体系

- **功能名称**：管理端身份认证与角色权限控制
- **入口路径**：管理端独立域名或 `/admin` 路由 → 管理端登录页

**核心能力描述**：
管理端拥有完全独立的认证体系，与用户端的登录系统物理隔离。**管理员账号创建**：管理员账号不由外部注册产生，仅由超级管理员在管理端"系统设置→管理员管理"中手动创建。超级管理员填写新管理员的邮箱、初始密码（首次登录强制修改）、分配角色（运营管理员或财务管理员），系统生成管理员账号并发送邮件通知。**管理端登录**：管理员访问管理端 URL 时，若未登录则重定向至管理端登录页（与用户端 `/auth` 页面完全独立的设计）。管理员输入邮箱和密码完成认证（复用与用户端相同的 JWT 机制但使用独立密钥对），支持 2FA 二次验证（强制要求，不可关闭）。登录成功后签发的 Admin Token 仅对管理端 API 有效，不能用于用户端 API。**角色与权限控制**：系统预置三种管理员角色——超级管理员（拥有管理端全部功能的查看与操作权限，包括创建/删除其他管理员、修改系统级配置）、运营管理员（可访问 7.1 大盘监控、7.2 用户管理、7.3 创意审批，但不可访问 7.5 财务管理中的退款审批和发票开票操作）、财务管理员（可访问 7.1 大盘的资金数据板块、7.5 全量财务管理功能，但不可访问 7.2 用户封禁和 7.3 创意审批）。权限控制在前端路由级别（通过路由元信息 `meta.roles` 过滤可见菜单项）和后端 API 级别（通过自定义注解 `@RequireRole` 在 Controller 方法上校验）双重实施。**操作审计日志**：管理端的每一次敏感操作（用户封禁/解封、创意审批通过/驳回、退款审批、管理员创建/删除、系统配置修改）均自动记录完整的审计日志，包含操作人 ID、操作时间、操作类型、操作对象 ID、操作前后状态变更、IP 地址和 User-Agent，存储于 `lf_admin_audit_log` 表，支持在管理端内按时间范围和操作人检索，日志保留期不少于 1 年。

**业务价值**：独立的管理员认证体系和细粒度的角色权限控制是平台安全运营的底线——将管理员操作与普通用户活动从认证层面完全隔离，防止用户端凭证泄露波及管理端；角色分离确保"运营不碰钱、财务不封人、超管一把关"的权责分离原则，降低内部操作风险。

**依赖与关联**：
- 依赖独立的 JWT 密钥对（与用户端密钥分离）和 Redis 存储管理端 Session
- 角色权限定义影响 7.1-7.5 所有管理端页面的可见菜单项与可操作按钮
- 操作审计日志为 7.5 财务对账和合规审计提供追溯依据
- 管理员创建/删除操作的通知需通过邮件或站内信发送至全体管理员

**非功能需求**：
- 管理端登录页首次加载时间 < 1s
- 管理端 Token 有效期 1 小时，Refresh Token 有效期 4 小时
- 管理员连续 5 次密码错误后账号锁定 30 分钟，并向全体管理员发送安全告警
- 审计日志写入应为同步阻塞操作（不可丢失），但不影响主业务的响应时间
- 管理端所有接口必须通过 HTTPS 传输，禁止 HTTP 明文

#### 7.1 大盘指标监控

- **功能名称**：平台核心运营数据总览
- **入口路径**：管理端 `/dashboard`

**核心能力描述**：
管理端首页以可视化仪表盘形式展示平台的全局运营指标，分为三个数据板块。**运营总览**——展示全站累计注册用户数、日活跃用户数（DAU）、月活跃用户数（MAU）、活跃项目总数、创意上架总数，每个指标以数字卡片形式呈现并附带环比变化趋势（如"较昨日 +5%"）和微型趋势折线图。**资金与计费总览**——展示累计充值金额、本月月度经常性收入（MRR）、年度经常性收入（ARR）增长曲线、付费转化率漏斗图（注册用户 → 活跃用户 → 试用用户 → 付费用户 → 续费用户），漏斗图以百分比标注每层转化率。**大模型消耗总量**——按模型通道（DeepSeek、Qwen、Moonshot、Claude 等）分别展示当日/当周的 Token 消耗总量与对应的 API 调用费用，以堆叠柱状图或饼图展示各通道占比，同时显示各通道的平均响应延迟（P50/P99）。

**业务价值**：大盘监控让管理层能够"一分钟看清平台全貌"——运营指标反映增长趋势，资金指标反映商业健康度，模型消耗指标反映成本结构与优化空间，三者结合为运营决策提供数据支撑。

**依赖与关联**：
- 数据源来自各业务表（用户表、项目表、创意表、生成记录表、支付表）的定时聚合
- 趋势数据依赖定时统计任务（每小时/每日）预计算
- 大模型消耗需从 AI 调用日志中提取每次调用的模型名称、Token 数、响应时间

**非功能需求**：
- 仪表盘数据刷新间隔为 5 分钟（可配置），首次加载 < 3s
- 需支持按时间范围（今日/本周/本月/自定义）筛选所有指标
- 图表需支持导出为 PNG 图片以供汇报使用

#### 7.2 用户运营管理

- **功能名称**：全站用户列表与单用户管理
- **入口路径**：管理端 `/users`

**核心能力描述**：
用户管理模块提供全站用户的列表查询与单用户操作能力。**用户列表**以表格形式展示用户数据，列包括：用户 ID、昵称、邮箱、手机号（脱敏显示中间 4 位）、注册时间、当前套餐等级、注册来源（邮箱/手机/GitHub/微信/Google）、状态（正常/已封禁）。表格支持按任意列排序，顶部提供多条件筛选：按 ID/邮箱/手机号搜索、按套餐等级过滤、按状态过滤。点击用户行进入**用户详情页**，左侧展示用户基本信息与操作历史，右侧展示两个关键看板：**接口调用看板**——以折线图展示该用户近 7 天调用各 API 端点的频次分布曲线，下方的表格展示该用户的 Token 消耗总量、已用/未用生成额度、平均每次生成耗时。**用户操作面板**——管理员可以在此执行以下管理操作：手动调整该用户的套餐级别与有效期、发放额外的生成配额或优惠券（输入数量和原因备注）、一键封禁/解封用户账号（封禁后该用户无法登录且所有活跃 Token 立即失效，解封后恢复正常）。**创作者认证审核**——当用户在 3.4 提交认证申请后，管理员在"认证审核"子模块中查看所有待审核的认证申请（列表展示申请人、申请时间、认证类型、提交材料），点击进入详情页查看用户提交的身份证明材料、代表作链接、个人简介等。管理员可做出"通过"（分配个人认证或机构认证标识，认证标识立即在 3.4 社区生态中展示）或"驳回"（填写原因，用户可重新申请）的决定。认证通过后，该用户在创意搜索排序中获得额外权重加成。管理员也可在已认证用户列表中对违规创作者执行"撤销认证"操作。所有管理操作均需记录操作日志（操作人、操作时间、操作内容、操作原因）。

**业务价值**：用户管理是平台运营治理的基础能力——管理员可以快速定位异常用户（如用量异常飙升的可能刷 API 行为）、辅助高价值用户（人工发放配额激励留存）、处理违规行为（封禁恶意用户），保障平台生态的健康运转。创作者认证审核为平台内容质量提供了人工把关机制——确保认证标识的可信度，防止虚假认证破坏社区信任。

**依赖与关联**：
- 依赖 1.1/1.2 的用户注册与登录数据
- 依赖 5.3/5.4 的套餐与用量数据
- 依赖 3.4 的创作者认证申请提交
- 封禁操作需联动 1.2 JWT Token 的失效处理
- 认证审核结果通过 1.7 消息通知中心通知申请人
- API 调用数据依赖 7.4 的调用日志记录

**非功能需求**：
- 用户列表数据量大时（10 万+）需支持服务端分页与高效索引
- 封禁/解封操作 1 秒内生效于全平台会话
- 所有管理操作需记录审计日志，保存不少于 1 年

#### 7.3 项目与创意审批

- **功能名称**：创意市场内容审核与项目资源审计
- **入口路径**：管理端 `/content`

**核心能力描述**：
**创意审批流**——当用户在 3.5 提交创意发布后，该创意进入审批队列，状态为"待审核"。管理员在审批列表中查看所有待审核的创意，按提交时间排序（可切换为紧急优先或按用户等级排序）。点击任意待审核创意进入**审批详情页**——左侧展示创意的完整信息（名称、描述、分类、标签、定价、缩略图），中间展示该创意关联的项目源码文件树与代码预览（只读模式，支持语法高亮），右侧为关联项目的沙箱预览 iframe（如可用）。管理员在页面底部做出审批决定：**通过**——创意状态更新为"已发布"，在创意中心立即可见；**驳回**——管理员填写驳回原因（从预设模板选择 + 手动补充），创意状态更新为"已驳回"，用户通过 1.7 消息通知中心收到驳回提示与原因。**已上架创意管理**——展示所有已发布的创意列表，支持搜索与筛选，管理员可查看详情、查看投诉记录（来源于 3.3.2 用户举报）、或执行"下架"操作（创意立即从创意中心隐藏，用户收到通知）。**投诉处理**——管理员在投诉处理队列中逐条审核用户提交的举报（按举报时间排序），查看举报详情（举报类型、描述、截图证据、被举报创意信息），做出"确认违规并下架"（下架创意并通知创作者）或"驳回举报"（通知举报人举报不成立）的决定。**模板管理**——管理员在此模块中配置和管理工作台简洁模式（2.1）所使用的快捷模板。模板列表展示所有已配置模板的名称、分类（官网/后台/电商/看板）、关联的提示词文本、是否启用。管理员可新增模板（填写模板名称、选择分类、编写预设提示词、上传模板缩略图）、编辑已有模板、启用/禁用模板（禁用后不在 2.1 简洁模式中展示）、删除模板。模板数据存储于 `lf_template` 表，简洁模式通过 `GET /templates?enabled=true` 获取当前启用的模板列表。**项目资源审计**——展示全站用户生成的项目列表，支持按文件数、生成时间、沙箱运行状态排序，用于识别异常项目（如文件数量异常多、构建时间异常长的可疑项目）。**异常应用拦截**——系统自动监控沙箱实例的 CPU 使用率与网络流量，对 CPU 占用超过 80% 持续 5 分钟或流量异常飙升的实例进行自动告警，管理员可一键终止可疑沙箱实例。

**业务价值**：内容审核是 UGC 平台的安全底线——防止恶意代码、挖矿脚本、侵权内容通过创意市场传播。结合自动异常检测与人工审批的双重机制，在审核效率和安全性之间取得平衡。模板管理赋予运营团队"引导用户创作方向"的能力——通过配置和调整模板内容，运营可以将用户的生成需求引导至平台主推的应用类型上，同时降低新用户的输入困惑。

**依赖与关联**：
- 审批数据源来自 3.5 创意发布提交
- 投诉处理数据源来自 3.3.2 用户举报提交
- 审批结果触达用户依赖 1.7 消息通知中心
- 沙箱异常检测依赖定时监控服务（每分钟轮询沙箱宿主资源使用率）
- 下架操作联动 3.1 创意列表的立即移除
- 模板数据供 2.1 简洁模式的 GET /templates 接口消费
- 模板缩略图上传依赖文件存储服务

**非功能需求**：
- 审批列表加载时间 < 500ms
- 源码预览需在沙箱环境中渲染，防止恶意代码在管理端执行
- 异常检测告警延迟 < 2 分钟（从检测到告警推送）

#### 7.4 Token 与模型调度

- **功能名称**：大模型 API 资源管理与智能路由
- **入口路径**：管理端 `/models`

**核心能力描述**：
**API Key 资金看板**——以表格卡片形式展示各 AI 模型厂商的实时信息：厂商名称（DeepSeek / Qwen / Moonshot / Anthropic）、当前账户余额、当月累计消耗金额与 Token 数、API 调用成功率、平均响应延迟（P50/P95/P99）。余额低于预设阈值（如低于 ¥100 或总余额的 10%）的卡片以红色边框高亮，并在页面顶部显示全局告警条。点击任意厂商卡片进入该厂商的详细分析（按时间维度的消耗趋势、按用户维度的消耗 Top 10、按模型维度的消耗分布）。**模型路由配置**——管理员可在此配置平台各模型的流量路由策略：设置主模型与备用模型的映射关系（如 DeepSeek-Pro 主、DeepSeek-Flash 备，当主模型可用率低于 95% 时自动切换），设置不同用户套餐对应的模型权限（如免费用户仅限 Fast 模型，付费用户可选 Pro 模型），设置全局单用户 API 调用频率限制（Rate Limit，如每分钟最多 10 次生成请求）。修改后的路由策略实时生效（通过配置中心或 Redis 发布订阅通知所有后端实例）。

**业务价值**：Token 与模型管理赋予平台对 AI 成本的控制力——通过实时监控各通道的消耗与余额，平台可以及时充值或切换备用模型，防止因单个模型服务中断导致用户无法使用。智能路由与分级模型权限确保高价值用户获得更好的模型体验。

**依赖与关联**：
- 消耗数据来自每次 AI API 调用的日志记录（需记录：模型名、Token 数、响应时间、请求用户、请求时间）
- 路由策略影响 2.2 生成流水线中 AgentFactory 的模型选择
- 余额告警触发管理端站内告警 + 邮件/短信通知超级管理员
- Rate Limit 需通过网关层或拦截器实现，对 2.1 提交生成与 2.7 提交迭代生效

**非功能需求**：
- 路由策略变更 3 秒内全集群生效
- 消耗数据统计延迟 < 30s（从 API 调用到面板展示）
- Rate Limit 响应时间 < 5ms（Redis Lua 脚本原子操作）

#### 7.5 资金与财务管理

- **功能名称**：支付对账、发票审批与退款管理

**核心能力描述**：

**业务价值**：财务管理模块是平台商业化的合规保障——清晰的对账单确保财务数据的准确性和可审计性，规范的发票审批流程满足企业用户的税务需求，完善的退款机制保护消费者权益同时防止恶意退款。

- 发票申请依赖 5.5 用户提交的开票请求

**非功能需求**：

- 财务相关所有接口需二次身份认证（管理员密码或 2FA 验证）
- 退款审批需要三人审核机制：运营发起→财务复核→超级管理员终批（基于常规设计补充）

### 5.3 系统级全局能力

以下能力不归属于单一业务模块，而是以全局基础设施或横切关注点的方式贯穿整个平台。

#### 5.3.1 暗色模式

平台提供全局暗色/亮色主题切换能力。切换入口位于全局顶部导航栏的用户头像下拉菜单中（选项"深色模式"旁带 Toggle Switch），也支持跟随操作系统级偏好设置自动切换（通过 CSS `prefers-color-scheme` 媒体查询检测）。用户手动切换后的选择持久化存储于 localStorage 和用户偏好表中（登录状态下同步至后端，确保跨设备一致）。主题切换通过 CSS 自定义属性（设计令牌）实现——在 `:root` 和 `[data-theme="dark"]` 选择器下分别定义两组完整的颜色令牌（`--ink`、`--soft`、`--paper`、`--line`、`--muted` 等），所有页面和组件统一引用令牌变量而非硬编码颜色值。暗色模式下页面主背景色由浅色系切换为深蓝灰系（`#0d1117` 基底），文字颜色相应反转为浅色系，卡片阴影减弱，代码编辑器（Monaco Editor）同步切换至暗色主题。过渡动画持续时间 300ms，避免瞬时切换造成视觉不适。

**依赖与关联**：依赖全局 CSS 设计令牌体系（`global.css`）、localStorage 读取与用户偏好 API（`PUT /user/preferences`）
**非功能需求**：主题切换过渡时间 < 300ms，所有 9 个页面和所有组件需完整适配暗色模式

#### 5.3.2 多语言国际化（i18n）

平台支持中文（简体）、英文、日文三种语言的界面切换。语言切换入口位于全局底部页脚的"语言"下拉选择器（移动端位于侧滑导航菜单底部）。切换后，所有界面文本（导航菜单、按钮、表单标签、提示文案、错误信息）即时替换为对应语言，但用户生成的内容（如项目名称、创意描述、代码注释）不做翻译仅保持原文。语言文件采用 `vue-i18n` 管理，按模块拆分为独立 JSON 语言包（如 `auth.zh-CN.json`、`auth.en.json`、`auth.ja.json`），按需懒加载以减少首屏体积。文档中心（模块六）的文档内容独立维护三种语言版本（由管理端在发布文档时分别填写），根据当前语言设置自动展示对应版本；若无对应语言版本则回退至中文。用户语言选择持久化至 localStorage 和用户偏好表（登录状态同步至后端）。

**依赖与关联**：依赖 `vue-i18n` 前端国际化框架、管理端文档多语言编辑能力
**非功能需求**：语言切换响应时间 < 200ms，三方语言覆盖率 ≥ 95%，语言包按需加载体积 < 50KB/语言

#### 5.3.3 首页/营销落地页导航引导

首页（路由 `/`）作为平台的品牌入口页面，面向未登录访客传达产品价值主张并引导进入核心功能。首页包含以下导航触点，形成完整的"吸引→了解→行动"转化漏斗：Hero 区"开始创作"主按钮以及顶部导航栏"免费开始"按钮——未登录用户点击后跳转至 `/auth` 注册页，注册成功后重定向至 `/workbench` 简洁模式；已登录用户点击后直接跳转至 `/workbench`。模板预览卡片的"试试这个"按钮——点击后将该模板对应的预设提示词填入简洁模式输入框，引导用户快速体验生成。底部 CTA 横幅的"查看定价"链接——跳转至 `/pricing`，向考虑付费的用户展示套餐详情。动态信息流中的"去看看"链接——跳转至 `/creative`，引导用户浏览社区作品。首页与各模块之间的跳转关系遵循"先注册后体验"原则——任何需要登录的功能入口在未登录状态下均重定向至 `/auth`，并在 URL 中附带 `redirect` 参数以在登录成功后跳回目标页面。

**依赖与关联**：依赖 M1 认证状态判断跳转逻辑、M2 简洁模式的模板预设参数传递、M3 创意中心的精选内容展示
**非功能需求**：首页首次内容渲染（FCP）< 1.5s，Hero 区动效不阻塞页面交互

#### 5.3.4 客服与工单系统（对接外部）

平台不自建完整的客服工单系统，而是通过以下渠道提供用户支持，在用户端以统一入口整合：在线客服（对接第三方客服 SaaS 如 Intercom/Zendesk/美洽，以右下角浮动聊天气泡呈现，已登录用户自动携带用户 ID 和当前套餐信息）、帮助中心（链接至 6.1 文档中心，引导用户自助解决问题）、问题反馈（在全局页脚提供"问题反馈"链接，点击后打开简洁的表单弹窗——标题、描述、上传截图（可选）、联系邮箱（自动填充已登录用户邮箱），提交后生成工单并邮件通知支持团队）。退款申请入口（在 5.3 订阅管理页的"取消订阅"挽留弹窗和 5.5 账单详情页中提供"客服介入"链接，跳转至在线客服并预设退款咨询标签）。平台运营团队在管理端 7.3 之外不需要管理工单系统——所有用户提交的问题反馈以邮件形式直接送达支持团队邮箱。

**依赖与关联**：依赖第三方客服 SaaS 的 JavaScript SDK 嵌入、邮件服务转发问题反馈
**非功能需求**：在线客服气泡不应阻塞移动端页面主要内容，需在移动端支持收起为最小化状态

### 5.4 异常处理原则

| 场景类型         | 处理策略                                                     |
| ---------------- | ------------------------------------------------------------ |
| **网络超时**     | 前端在 30s 无响应后展示"请求超时，请检查网络并重试"提示，提供手动重试按钮；每项操作最多自动重试 2 次 |
| **API 返回错误** | 前端根据 `Result.code` 判断：400 类错误展示后端 `message` 字段内容；500 类错误展示"服务繁忙，请稍后重试"并记录错误日志；401 未授权自动触发 Token 刷新或跳转登录 |
| **AI 生成失败**  | 流水线在任意阶段失败时，前端 Checklist 标记失败阶段，展示错误信息与建议操作；后端支持按配置的最大重试次数自动重试（默认 2 次） |
| **沙箱启动失败** | 前端预览面板展示"预览启动失败"提示，附带构建日志中的错误摘要；用户可查看完整构建日志并通过迭代对话请求 AI 修复 |
| **支付异常**     | 支付回调采用幂等处理防止重复扣款；支付成功后未及时更新订阅状态时，用户可手动点击"同步支付状态"按钮触发重新查询 |
| **用量超限**     | 用户在用量达到 100% 后被拦截提交新生成请求，前端弹出引导升级弹窗（展示当前用量 + 推荐套餐）；管理员可临时发放额外配额 |
| **并发冲突**     | 同一项目同时被多次迭代修改时，后端采用乐观锁机制，后提交的修改返回 409 冲突错误并提示用户刷新后再试 |

### 5.5 数据保留策略

| 数据类型       | 保留周期                 | 过期处理                                       |
| -------------- | ------------------------ | ---------------------------------------------- |
| 用户账号       | 永久（除非用户主动注销） | 注销后 30 天软删除宽限期，之后彻底清除个人信息 |
| 项目文件       | 永久（除非用户删除）     | 软删除进入回收站，30 天后自动彻底删除          |
| 生成记录       | 180 天                   | 超期后仅保留统计汇总数据，原始记录归档至冷存储 |
| 活动日志       | 180 天                   | 超期后归档至日志存储系统                       |
| 账单记录       | 3 年                     | 满足税务合规要求，超期后可删除                 |
| Token 调用日志 | 90 天                    | 超期后仅保留按用户/按模型的日聚合数据          |
| 沙箱实例       | 24 小时无活动后自动销毁  | 超时停止并释放资源                             |
